# 详细检查清单

## 1. description检查清单

**检查项**：
- [ ] 是否包含触发关键词（"当用户说"、"在XXX场景时触发"）
- [ ] 是否避免功能说明表述（"这是一个"、"该Skill用于"）
- [ ] 长度是否合理（<100字符）
- [ ] 是否包含用户会说的自然语言关键词
- [ ] description 是否模拟了用户自然语言表达

**评分标准**：
- 90分：完美符合（触发条件明确、关键词完整）
- 70分：基本符合（有触发关键词但不够精准）
- 50分：不符合（写成功能说明）

---

## 2. gotchas检查清单

**检查项**：
- [ ] gotchas.md是否存在
- [ ] 内容是否非空（>500字节）
- [ ] 是否有"现象-问题-解决"结构
- [ ] 是否有真实坑点（不是"暂无"、"TODO")
- [ ] 坑点数量是否合理（>3个）

**评分标准**：
- 90分：优秀（>5个真实坑点，结构完整）
- 70分：良好（有坑点但内容较少）
- 40分：不合格（空壳或占位符）

---

## 3. 文件组织检查清单（三分类标准 + 轻量Skill豁免）

**检查项**：
- [ ] 轻量Skill豁免判定：SKILL.md ≤ 80行 且 无附属文件 → 自动判定为合理，跳过以下检查
- [ ] （非轻量）根目录仅含核心文件：SKILL.md + gotchas.md + examples.md（可选 README.md）
- [ ] （非轻量）参考材料（字段映射、命令参考、Schema、详细清单）是否在 reference/ 目录下
- [ ] （非轻量）配置文件是否在 config/ 目录下（如 config.json）
- [ ] （非轻量）脚本是否在 scripts/ 目录下
- [ ] SKILL.md主文件是否<200行（非轻量Skill适用）
- [ ] 是否使用[链接](reference/xxx.md)引用
- [ ] 是否遵循渐进式披露原则
- [ ] 引用深度是否为1层（附属文件之间无互引，仅从SKILL.md引用）
- [ ] reference/ 目录是否非空
- [ ] 根目录是否有纯参考类 .md 文件（如 commands-reference.md、schema.md）→ 应移入 reference/

**评分标准**：
- 92分：轻量Skill（单文件≤80行，职责单一，无需拆分）
- 90分：优秀（三分类清晰，主文件<150行，附属文件完整）
- 70-75分：良好（有三分类意识，主文件<250行，有部分拆分）
- 50分：不合格（该拆不拆：单文件>150行无附属文件，或参考材料散落根目录）

---

## 4. 过度约束检查清单

**检查项**：
- [ ] 是否避免"第一步第二步"表述
- [ ] 是否有决策点（"判断"、"选择"、"根据"）
- [ ] 是否约束目标而不是路径
- [ ] 是否给Agent足够自由度
- [ ] 是否描述决策框架

**评分标准**：
- 90分：优秀（决策框架清晰，无过度约束）
- 70分：良好（有决策点但有部分路径约束）
- 50分：不合格（死板流程，无决策点）

---

## 5. 已知知识检查清单

**检查项**：
- [ ] 是否避免"什么是"、"介绍"等教程式表述
- [ ] 是否补充团队特有知识
- [ ] 是否避免重复Agent已知内容
- [ ] 是否聚焦Agent不知道的部分
- [ ] 是否有价值增量

**评分标准**：
- 90分：优秀（只补充Agent不知道的）
- 70分：良好（大部分是特有知识）
- 40分：不合格（大量教程式内容）

---

## 6. 内存配置检查清单

**检查项**：
- [ ] config.json是否存在
- [ ] 配置是否合理（不是空文件）
- [ ] 是否包含团队偏好
- [ ] Skill是否需要持久化配置
- [ ] 配置项是否有说明
- [ ] 是否有日志文件记录运行历史（log.json 等，追加而非覆盖）
- [ ] config 和 log 的功能是否区分清晰（静态配置 vs 运行历史）

**评分标准**：
- 90分：优秀（config完整且合理）
- 70分：良好（有配置但较简单）
- 50分：不合格（无config.json）
- N/A：Skill不需要配置（不扣分）

---

## 7. 脚本检查清单

**检查项**：
- [ ] Skill是否需要脚本（稳定能力）
- [ ] scripts目录是否存在
- [ ] 脚本是否有注释
- [ ] 脚本是否有错误处理
- [ ] 脚本调用方式是否正确
- [ ] 脚本是否有错误处理（try/except 或等价机制）
- [ ] 脚本是否存在"魔法数字"且缺少注释说明原因
- [ ] 脚本执行方式是否明确（"执行脚本"或"作为参考阅读"）

**评分标准**：
- 90分：优秀（脚本完整、注释清晰）
- 70分：良好（有脚本但较简单）
- 50分：不合格（需要脚本但缺失）
- N/A：Skill不需要脚本（不扣分）

**检查项**：
- [ ] 是否有下节预告或学习路径（如果是教程类Skill）

- 50分：不合格（缺少关键元素）

---

## 11. 渐进式披露结构检查清单

**检查项**：
- [ ] description 是否仅写触发条件（元数据），不包含正文内容
- [ ] SKILL.md 是否 < 200 行，仅包含核心逻辑和导航
- [ ] 附属文件是否从 SKILL.md 直接引用（一层深度）
- [ ] 参考材料是否在 reference/ 目录下实现按需加载
- [ ] 是否避免了链式引用（A.md → B.md → C.md）
- [ ] 是否有明确的按需加载策略

**评分标准**：
- 90分：完美符合三分类结构，reference/ 组织清晰，引用深度仅一层
- 70分：基本符合但有部分偏差
- 40分：单文件堆砌或深层嵌套引用

---

## 12. 日志检查清单

**检查项**：
- [ ] log.json 或等价日志文件是否存在
- [ ] 日志是否为追加写入（而非覆盖）
- [ ] 日志是否包含结构化字段（timestamp、action、status）
- [ ] 日志是否可以追溯历史操作
- [ ] 日志是否与 config.json 功能区分清晰

**评分标准**：
- 90分：日志完整且结构清晰
- 70分：有日志但格式简单
- 50分：无日志或只覆盖不追加
- N/A：Skill不需要历史记忆（不扣分）

---

## 13. 迭代维护检查清单

**检查项**：
- [ ] Skill 是否设置更新频率或检查周期
- [ ] gotchas 是否在上次更新后有新增内容
- [ ] description 是否仍然匹配当前使用场景
- [ ] 是否有测试用例验证 Skill 输出
- [ ] 是否建立了"踩坑 → 补 gotchas"的反馈闭环

**评分标准**：
- 90分：有明确的迭代机制且持续更新
- 70分：偶尔更新但无固定机制
- 30分：写完即抛弃，从不更新

---

## 14. 配置文件位置检查清单

**检查项**：
- [ ] config.json 是否放在合理位置（Skill 目录内）
- [ ] 配置路径是否在 SKILL.md 中有明确说明
- [ ] 不同 Skill 的配置是否不会相互覆盖
- [ ] 配置文件的读写权限是否正确
- [ ] 是否支持首次运行自动创建配置（onboarding 流程）

**评分标准**：
- 90分：配置位置合理且有 onboarding 流程
- 70分：有配置文件但位置不够规范
- N/A：Skill不需要配置（不扣分）

---

## 总评分公式

```
总分 = (检查项1得分 + 检查项2得分 + ... + 检查项14得分) / 14
等级 = A(90-100) / B(75-89) / C(60-74) / D(<60)
```

**权重调整**：
- gotchas：权重2.0（信号最强、最有价值的内容）
- description：权重1.5（触发机制最关键）
- 渐进式披露：权重1.5（核心结构设计）
- 其他项：权重1.0
