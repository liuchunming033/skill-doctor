# Gotchas —— 教程创作踩坑记录

## 坑点 1：章节文件名含空格导致链接失效

**现象**：章节文件名如 `01-为什么要学 Skill？.md`，在 README 中直接写 `[链接](./chapters/01-为什么要学 Skill？.md)`，GitHub 上点击 404。

**问题**：Markdown 链接中的空格不会自动转义，浏览器/GitHub 无法正确解析含空格的路径。

**解决**：空格用 `%20` 转义：`[链接](./chapters/01-为什么要学%20Skill？.md)`。

---

## 坑点 2：图片路径在子目录中失效

**现象**：图片放在 `images/`，章节在 `chapters/` 子目录，章节中 `![](images/xxx.png)` 渲染不出。

**问题**：Markdown 图片相对路径是相对于当前文件所在目录解析的，`chapters/` 下的文件引用 `images/` 实际指向 `chapters/images/`，而目录不存在。

**解决**：章节中图片路径用 `../images/xxx.png`。README 中仍用 `images/xxx.png`。

---

## 坑点 3：PDF 目录跳转页码不准

**现象**：用 `pdftotext` 搜索章节标题定位页码，章节改名或调整内容后页码偏移，目录书签跳到错误页。

**问题**：`pdftotext` 提取的页码基于 PDF 物理页码而非文档页码，章节内容增减导致目标位置物理页码变化。

**解决**：每次改章节内容后重新构建验证目录跳转；`find-chapter-pages.py` 用模糊匹配而非精确匹配标题。

---

## 坑点 4：代码块长行溢出 PDF 页面

**现象**：代码块中的长命令或 URL 超出 PDF 页面宽度，右边内容被截断。

**问题**：CSS 默认 `white-space: pre` 禁止换行，长行在固定页面宽度下必然溢出。

**解决**：CSS 设 `white-space: pre-wrap` 而不是 `pre`，允许代码块在单词边界自动换行。

---

## 坑点 5：一上来就规划 27 章

**现象**：先设计完整章节目录，然后一章一章填，写到第 5 章就动力耗尽、半途而废。

**问题**：完美主义者倾向导致过度设计，缺少反馈循环。没有读者反馈，写作变成单向苦力。

**解决**：先写核心 3 章，发布 MVP 版本，看反馈。有反馈才有动力继续写。27 章是一轮一轮扩展出来的，不是一次性规划出来的。

---

## 坑点 6：PDF 页面内容被截断

**现象**：Puppeteer 生成的 PDF 中，表格或代码块跨页时被截断，上半部分在 A 页，下半部分消失。

**问题**：Puppeteer 的默认分页行为不考虑内容完整性，恰好在页边界处割断元素。

**解决**：CSS 中添加 `page-break-inside: avoid` 用于表格和代码块，必要时用 `page-break-before: always` 强制分页。

---

## 坑点 7：中文排版在 PDF 中断词不当

**现象**：英文 PDF 工具处理中文时，在错误位置断行（如标点符号后断行、英文单词中间断开）。

**问题**：Puppeteer 的 CSS 排版引擎对中文字符的断行规则处理不完善，中英文混排时尤其明显。

**解决**：CSS 设 `word-break: break-all` 用于代码块，正文用 `word-break: normal`，配合 `hyphens: none` 禁止英文断字。

---

## 坑点 8：README 中英文版本内容不同步

**现象**：改完中文 README 忘了更新英文版，GitHub 首页展示过时信息，中英文内容不一致。

**问题**：中英文 README 是两套独立维护的文件，没有自动同步机制，容易遗漏。

**解决**：每次修改 README 后，同步更新另一语言版本。最好在同一次 commit 中完成，避免不同步窗口期。
