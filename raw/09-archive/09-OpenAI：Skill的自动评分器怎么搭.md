## 核心观点

你更新了一个 Skill，它跑完了，结果看起来没问题——但你怎么知道它真的变好了，没把别的地方搞坏？

上一期基于 OpenAI 的文章把评估的最小闭环立起来了（定义成功→识别失控点→建小测试集），但测试集有了，怎么打分？这一期讲的就是**评分器这一层怎么构建**，以及怎么输出评分结果才能跨版本比较。

---

## 第一步：记录运行轨迹

要给 Skill 打分，先得有东西可以打。OpenAI 的做法是用 `codex --trace-json` 参数——加上这个参数后，Skill 每执行一步就会输出一条结构化记录：跑了什么命令、创建了什么文件，全部按时间顺序记下来。

这不是日志摘要，是一条完整的可以被程序解析的**运行轨迹**。有了运行轨迹，评分器才有东西可以解析。没有运行轨迹，你只能看最终产物，看不到过程。

---

## 第二种评分器：确定性检查（Deterministic Check）

纯代码规则，不用大模型。你写死判断逻辑，程序直接对着运行轨迹核对。它只回答一个问题：**基础动作做了没有？**

- 有没有跑 `npm install`？
- 有没有创建 `package.json`？
- 命令顺序是不是符合预期？

### 两个关键特性
1. **确定性**：同样的行为每次判断结果一样，没有模糊地带
2. **可调试**：一旦失败，打开记录文件就能看到 Agent 从哪一步开始偏的——以前只知道任务翻车了，现在能看到从哪一步开始翻的

这一层相当于给 Skill 装了**飞行记录仪**。但它有天然上限——它能告诉你动作发生了没有，它告诉不了你做出来的东西是不是你想要的样子。

---

## 第二种评分器：评分细则检查（Rubric Check）

用 LLM 做裁判，让它来读仓库打分。很多 Skill 的验收标准本来就不是硬规则——代码结构够不够干净？组件组织是不是符合团队约定？样式配置是不是按既定风格落地？

### 评分细则（Rubric）是什么？
它是一张拆解表，把一个模糊的质量标准拆成若干个具体的、可独立判断的维度。

**例子**：把"代码质量好"拆开后可能是：
- 组件是否按功能分目录
- 是否有未使用的引用
- 样式类名是否遵循命名约定
- 配置项是否集中管理

每个维度单独打分，最后汇总。

### 关键：输出格式必须被限定
提前写好一个模板，规定大模型必须输出哪些字段（如整体是否通过、总分、每个维度的得分、每个维度为什么这么打分），大模型必须按这个模板填空，不能自由发挥。

**为什么要严格限定格式？**
- 如果只问模型"这段代码好不好"，它会给你一段散文式的评论，每次措辞不同，无法比较，无法进自动化流程
- 限定格式可以把模型的判断变成可以被程序读取的数据
- 评估结果就可以跨版本比较、长期追踪趋势、直接进持续集成流水线做自动判断

**你收集的是评估数据，不是审稿意见。评估数据是给系统用的，会持续积累。**

### 两种评分器的分工
| 评分器 | 职责 | 手段 |
|--------|------|------|
| **确定性检查** | 抓底线——基础动作有没有发生 | 纯代码规则 |
| **评分细则检查** | 抓质量——做出来的东西像不像你要的 | LLM + 锁死输出格式 |

这套组合从一开始就设计成可以接进持续集成的。一旦进了流水线，它就不再依赖某个工程师记得手动去试，它会变成**持续运行的回归守门员**。

## 扩展检查（Skill 成熟后的六类指标）

前两步解决的是"对不对"的问题，Skill 真正进入生产之后，还需要回答另一类问题：**稳不稳？贵不贵？脏不脏？**

### 1. 命令数和循环检测
运行记录里执行了多少条命令？如果一个 Skill 以前六步能完成，现在要 18 步——表面上任务还是完成了，但系统在变差。这是**行为退化最直接的信号**。

### 2. 消耗量追踪
每次运行消耗了多少 token？防止 Skill 越写越胖，每次执行越来越贵、越来越慢。

### 3. 构建检查
Skill 做完之后直接跑一次构建命令。很多项目表面上看起来完成了，一构建就露馅儿。这是**最强的端到端信号**，成本低，应该默认开启。

### 4. 运行时冒烟检查
把开发服务器拉起来，发一个请求看有没有响应。这类检查更慢，按风险来加，不是每次都全上。

### 5. 仓库清洁度检查
看任务完成之后有没有出现不该出现的文件。任务做完了不代表系统没被弄脏。

### 6. 权限回归检查
验证 Skill 是不是还能在最小权限下正常工作，防止某次改动之后它悄悄开始依赖更高权限。**权限漂移在自动化之后会变成结构性风险。**

原则：**先加快速检查，再按风险叠加慢检查，不是一次全上，是按需扩展。**

---

## 五条原则总结

1. **量真正重要的东西**——不要为了有指标而有指标，先想清楚什么变差了你会真的在意
2. **先写清可检查的定义**——验收标准模糊，评估就没有意义。先把"什么叫做对"写成可执行的判断
3. **把评估扎在真实行为上**——记录 Skill 的完整运行轨迹，围绕实际执行的动作写确定性检查
4. **规则不够时，再让模型补上**——用评分细则加锁死格式处理规则覆盖不了的质量判断，顺序不能反
5. **让真实失败去驱动样本增长**——每次手动修复都是一个信号，把它变成测试用例，让 Skill 持续把这件事做对

**真正成熟的 Skill 不只是能调用模型做事，它还得有一套评估系统，持续证明自己没有偏移方向。**
