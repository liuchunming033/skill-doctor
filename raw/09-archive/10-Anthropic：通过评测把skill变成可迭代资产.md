## 核心观点

别再凭感觉改 Skill 了。很多时候我们迭代 Skill 靠的是一种感觉——"我觉得它有用""我跑过几次效果还不错""我改了几句话，好像更顺了"。这里面有一个很大的空洞：

- 你怎么知道它真的变好了？
- 你怎么知道它没有只在刚才那个例子里变好？
- 你怎么知道它没有悄悄把另一个场景搞坏？
- 如果下一个模型版本发布了，模型自己已经能完成这件事了，你怎么判断这个 Skill 还该不该存在？

Anthropic 在重写 Skill 的生命周期——**写 Skill 不再只是写说明文档，它变成一个反复测试、度量、比较、修改的闭环。评估不再是最后验收，它变成 Skill 能持续提升的前提。**

---

## Skill 的两大分类

### 1. Capability Uplift Skills（补能力）
教 Claude 做一些原本不稳定、不会做或者很容易做错的事（如处理某类 PDF、生成某种固定结构的文档、跑一套特定的分析流程）。

**价值前提**：模型现在还不能可靠地自己完成。
**Eval 要回答**：这个 Skill 现在还带不带来真实增益？如果模型升级以后裸模型已经能做得同样好甚至更好，那这个 Skill 可能就该退役了。

### 2. Encoded Preference Skills（固化偏好）
固化的是你的偏好、你的流程、你的判断标准（如代码审查永远先看安全边界、写研究报告时必须先区分事实推断和假设、每次生成视频稿都遵循你自己的叙事节奏）。

**价值前提**：它有没有忠实执行你的工作流。
**Eval 要回答**：它还在按你的方式工作吗？

两类 Skill 不一样，但共同点是——**没有 Eval，你都只能猜。**

---

## Skill 的生命周期七步流程

### 第一步：决定 Skill 要做什么
这一步决定后面所有评估的方向。如果连这个 Skill 要解决什么任务都说不清，后面就没有办法判断它有没有成功。

### 第二步：写一个草稿
这时的 SKILL.md 还没有到成品阶段，更像一个候选方案。先把流程、约束、需要 Claude 注意的细节写出来。

### 第三步：创建少量测试提示词，马上跑起来
从这里���始，它和普通写 prompt 的习惯分开——普通做法是写完以后凭感觉试一下，Anthropic 的做法是**草稿一出现就立刻让它面对测试用例**。

### 第四步：看结果
不止看通过还是失败，还要看输出哪里对、哪里偏、哪里不稳定。也要看这个测试用例本身写得好不好——有些时候错误不在 Skill，是你根本没有把什么叫好结果写清楚。

### 第五步：展示结果
工具里有 Eval Viewer，它的作用不是炫技，而是**把输出指标、失败原因放到人面前**。因为 Skill 的质量不能只交给自动评分人，还要看它是否符合真实工作流。

### 第六步：根据反馈改 Skill
这一步进入迭代——不再停留于"感觉这句话可以写得更顺"，而是追问：哪个测试失败了？失败原因是什么？是流程漏了一步，还是约束写得太宽，还是示例不够？改完以后再跑同一批测试，确认旧问题被修掉。

### 第七步：扩大测试集
少量测试只能证明你没有在几个例子上失败，它不能证明这个 Skill 稳定。后面要把测试集扩大。

**案例**：Anthropic 的 PDF Skill 之前在处理非填充式表单时会出问题（PDF 里没有预定义的输入字段，Claude 必须把文字放到非常精确的坐标上）。如果没有 Eval，大概率只能靠偶然发现。但有了 Eval，失败模式就被隔离出来了——测试用例告诉你在这个场景里失败了。然后修复定位逻辑，修完以后再跑 Eval，确认这个失败模式被解决。

**Eval 的第一层价值**：它的目标并非给 Skill 打一个漂亮分数，而是**把失败变成可复现的对象**。只要失败可以复现，就可以定位；只要可以定位，就可以修复；只要修复后还能再测，Skill 才进入迭代闭环。

## 三个评测场景

### 1. 基准测试（Benchmark）
先判断这个 Skill 有没有必要存在——它能不能完成预期任务？相比不用 Skill 有没有带来增量？

Benchmark 会把 **with skill** 和 **without skill** 放在一起比较。如果不用 Skill 也能做到同样好，那这个 Skill 的存在价值就变弱了。如果有 Skill 明显更稳、更快、更省 token，那它就证明了自己的增量。

这也是 Capability Uplift Skill 会遇到的问题——模型会进步，今天必须靠 Skill 才能完成的任务，下一个模型版本可能自己就能完成。Eval 和 Benchmark 一起跑，就能告诉你这个 Skill 还需不需要继续存在。

### 2. 回归测试（Regression Test）
回答的是：**我改了 Skill 以后，有没有把原来能做的事弄坏？**

这对 Skill 很要命，因为你每次改 SKILL.md 都会改变 Claude 的注意力分配。你加了一条新规则，可能修好了一个场景，也可能让另一个场景变差。回归测试的作用是守住旧能力。

### 3. 版本对比（Version Comparison）
回答的是：**这一次改动到底有没有把 Skill 改好？**

通过 A/B Testing 把两个版本并排放在同一组任务上比较。你重写了一段工作流、增加了一份参考材料、删掉了一个步骤、调整了判断标准——版本对比会逼你回答：哪个版本赢在哪里？输在哪里？有没有只是在某个样本上赢了？有没有在另一个样本上退步？

---

## 核心转变

三类场景串起来，顺序很清楚：**先证明 Skill 有用 → 再保证修改没有破坏旧能力 → 最后比较不同版本决定下一轮往哪里改。**

没有评估时，你写 Skill 像是在写信条——你相信这段流程会让 Agent 更好，你相信这些规则会被遵守，但这些都没有被验证。

有了评估以后，Skill 才从"我相信"变成"我可以证明"：
- 我可以证明它在这些场景下有效
- 我可以证明它比不用 Skill 更好
- 我可以证明修改没有破坏旧能力

**成熟的 Skill 不能���靠表面上的专业感，而应该是可以被验证、被比较、被复盘、被淘汰，也可以被持续迭代的工作流资产。**

没有 Eval 的 Skill，只是一���还没被检验过的假设——它可能有用，也可能只是看起来有用。只有当你能测试它、衡量它、比较它、改进它，它才进入工程世界。
