## 核心观点

你把一个 Skill 接进了生产，它在跑任务也在完成。然后你改了一版，跑了一下，看起来没出事，就继续改。这套节奏短期能撑住，但它有一个根本性的漏洞：**你每次改完都不知道这是优化还是回归。** 你靠的是体感，不是证据。

而 Skill 一旦接入真实工作流，它会触发工具、生成文件、留下副作用。你改它的任何一个地方，整个行为边界都可能一起飘。

OpenAI 的文章《Testing Agent Skills Systematically with Evals》上来先不教你怎么写 Skill，它先做一件更基础的事：**把什么叫成功写清楚。**

---

## 一、定义 Success 的四类目标

### 1. 结果目标（Outcome）
任务有没有完成？应用能不能跑起来？这是最直觉的那一层，也是最容易被单独当成全部的那一层。

### 2. 过程目标（Process）
Codex 有没有真的调用这个 Skill？是不是按你预想的步骤和工具链在执行？

**为什么关键**：最终结果对了，不代表中间路径对了。一个 Skill 可能靠绕路完成了任务，但它走的路你根本没预期，下次换个输入就可能绕不过去。过程目标是在验证行为路径本身，不只是终点。

### 3. 风格目标（Style）
输出是不是符合你要的规范？如文件结构、组件组织方式、代码风格。

这类容易被忽略，因为它不影响能不能跑，但在真实工作流里，风格不一致会让下游的人或系统无法接受。风格目标是在保证 Skill 的输出能被整个系统消化。

### 4. 效率目标（Efficiency）
有没有 thrashing？有没有乱跑命令？有没有无意义的烧 token？

一个 Skill 如果每次都要多跑三步才能完成任务，在单次测试里可能感觉不到，但接入生产大量触发之后，这个浪费会被放大。

---

## 二、Skill 的两个独立维护面

Skill 有两个独立的维护面，可以独立失控：

1. **执行体**：instructions、工具链、步骤
2. **触发边界**：name 和 description

你改了 instructions，触发边界没变，但行为变了；你改了 description，执行体没变，但触发范围飘了。**如果只盯着执行体，触发边界的腐烂你根本看不见。**

---

## 三、三类隐藏假设（失控点）

OpenAI 把失控点归为三类：

### 1. 触发假设（Trigger Assumptions）
关于触发边界的盲区：
- 哪些 prompt 明明应该触发这个 Skill，却没有触发？
- 哪些 prompt 只是相邻需求，却把它误触发出来了？

**例子**："set up a quick react demo"应该吊起 setup-demo-app；但"add tailwind styling"只是给现有项目加样式，如果它也把 setup-demo-app 调出来了，说明 Skill 的触发边界已经糊了。

### 2. 环境假设（Environment Assumptions）
关于运行前提的盲区：
- Skill 会不会默认自己永远在空目录运行？
- 会不会默认系统里一定有 npm？
- 会不会默认用户偏好的包管理器和自己想的一样？

这类假设在本地测试时几乎不会暴露（因为本地环境刚好满足了这些前提），但换一台机器、换一个用户，这些前提就可能全部失效。**环境假设是最隐蔽的一类，因为它们不在 Skill 的代码里，它们在 Skill 的运行上下文里。**

### 3. 执行假设（Execution Assumptions）
关于执行顺序的盲区：
- Agent 会不会跳过 `npm install`，因为它误以为依赖已经装好了？
- 会不会在项目都还没建出来之前，就先去配 Tailwind？

## 四、构建回归样本集

找到这些失控点之后，下一步是把它们固定成一个回归样本集。对单个 Skill 来说，**10~20 条 prompt 已经足够高效**——重点不在数量，在这些样本能不能卡住真实的失败模式。

### 四类样本

| 类型 | 说明 | 作用 |
|------|------|------|
| **显式调用** | prompt 直接点名这个 Skill | 保证 name/description/instructions 改动后，显式调用至少不会坏掉 |
| **隐式调用** | 用户不提 Skill 名字，只描述场景 | 测试 name 和 description 是否足够强，让 Codex 能自己选中它 |
| **带上下文调用** | 任务中混入真实业务上下文 | 测 Skill 在噪声环境下还能不能被正确识别和执行 |
| **负控样本** | 不该触发这个 Skill 的请求 | 测 Skill 该不该触发——最危险的那种失控，验证它知道什么时候不该做 |

**Negative Control 是在给 Skill 划边界**：不只是验证它能做什么，还要验证它知道什么时候不该做。

---

## 五、核心转变

把这四类样本建起来，你完成的不是一套完整的评估系统，你完成的是一件更基础的事：**你把靠体感维护换成了靠最小证据闭环维护。**

这个转变比任何工具细节都重要。因为 Skill 一旦接入生产，它就不再只是 prompt 技巧，它开始变成一个要被持续验证的工作流资产。

维护这种资产至少要先回答三件事：
1. **你想让它做成什么？**
2. **它会在哪些地方失控？**
3. **你准备用哪些样本持续盯住这些失控点？**

做到这里，你才算真正从"会写 Skill"开始走向**"会维护 Skill"**。
