> 介绍了知识库的建设和维护，在软件工程领域，对知识库的核心依赖是产出能够进行AI开发的需求文档、设计文档、原型文档、技术方案和测试用例。本文介绍如何将这些工作流水线化。

> 💡 **文档目标：**用户输入一句话需求和意图，飞书项目流水线自动生成产品需求Spec、技术方案、测试用例等资料，并触发自动化门禁检测，以及人工评审。所有人确认之后进入后续流程。

## 节点1：飞书项目上创建一句话需求，触发自动化流水线

1. 飞书项目新建Issue，系统**自动生成唯一RID**
2. 仅填写极简信息：一句话需求意图，以及必要字段（优先级等）
3. 启动AI流水线，系统自动注入全局信息

## 节点2：生成Spec/PRD

1. Agent 自动从 Obsidian 全库召回和整理信息、冲突校验
2. 生成标准 Spec/PRD/html原型，关联到飞书项目Issue的对应字段

## 节点3：生成技术方案

Agent 自动从 Obsidian 全库，结合节点2的产物，生成的技术方案文档，关联到飞书项目对应字段

## 节点4：生成测试用例

Agent 自动从 Obsidian 全库，结合前面节点的产物，自动生成全套测试用例，生成的文档关联到飞书项目case字段

> 🎯 说明：节点2、3、4：
> 1. 节点对产物进行AI自动质检
> 2. 节点有干系人进行人工评审
> 3. 评审通过后，进入下一个节点
> 4. 评审不通过，可以at该节点，进行修改。

## 节点5：门禁+评审

1. 自动门禁检查前三者产物的一致性，提出comments
2. 人工评审检查自动门禁的comments和产物。
3. 评审分支：
   - 通过：填写结构化评审意见，全套产物自动回流至产品线知识库的 `specs/{{RID}}/`
   - 不通过：填写标准化修改意见，Agent自动迭代重生成
4. 评审记录自动沉淀留存

```JSON
评审人：
问题类型：业务理解偏差 / 术语不一致 / 违背历史决策 / 技术不可行 / 边界缺失 / 用例不全 / AI 幻觉
修改建议：
是否采纳：是 / 否
```

## 归档 & 版本锁定

1. 评审通过后，全套产物自动回流至产品线知识库的 `specs/{{RID}}/`
2. 自动补全双向知识库关联、元数据
3. 自动Git提交并打版本Tag（RID-v1.0）
4. 全局知识看板自动更新

## 核心优势

1. **严格技能依赖**：流水线可控、可溯源、不会乱生成、不会缺前置输入
2. **人机精准分工**：人只做创意与决策，AI承担全部结构化产出
3. **多人协同**：各节点支持多人评审，把关输出质量
4. **全链路可追溯**：原始素材→知识依据→产物→评审全程闭环
5. 全流程可度量：各环节状态自动流转，耗时准确可度量，各环节AI成本（Token）可度量。有度量就可改进。
