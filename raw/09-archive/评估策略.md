# **端到端评测VS分层评测**

![Image](https://api3-eeft-drive.feishu.cn/space/api/box/stream/download/authcode/?code=ZTNlZjAxOGNkNDQ4ZjE4MGZmNTQ0NWM4NjFmYzliNmNfMWJkZGVhNTc5YTE3Y2U1MzNhYWQ5OTdhNmVjNTk2NTJfSUQ6NzYyNTI4MDE3MDU4MDA1MzE3Ml8xNzgyMTk5MDAzOjE3ODIyODU0MDNfVjM)

在 AI Agent 的质量保障体系中，**端到端评测**与**分层评测**（也称为分段或组件评测）相结合是目前主流且科学的评估策略。这种“组合拳”的逻辑在于：**端到端评测发现问题，分层评测定位并解决问题。**

- **端到端 \(E2E\) 评测**：将系统视为“黑盒”，模拟真实用户场景（如旅行规划、代码生成），关注最终任务是否成功及用户满意度。主要用于**准入验证**（能否发布）。

- **分段评测**：将链路切开，排除上下游干扰，针对具体算子（如 Tool、RAG）进行评估。主要用于**缺陷定位与快速迭代**（该修哪里）

除此之外，还可以从**系统效率层**（资源/成本）、**会话级别**（整体轨迹）到**节点级别**（具体动作精度）进行垂直拆解，分别评测。

**端到端评测**与**分层评测**（也称为分段或组件评测）对比：

分层可以从以下三个维度进行：

## **按架构逻辑分层（工作流节点）**

这是最常见的分层方式，根据 Agent 执行任务的物理环节进行拆解：

1. **意图理解层 \(S1\)**：评估是否准确理解用户指令并打上正确的分类标签。

2. **信息检索层 \(S2\)**：针对 RAG 场景，评估返回信息的真实性、实时性和覆盖度（如 Top\-5 结果准确率）。

3. **逻辑规划层 \(S3\)**：评估 Agent 将目标分解为子任务的能力，及规划路径的合理性（如最优解命中率）。

4. **内容/文案包装层 \(S4\)**：评估输出语言的自然度、逻辑性和品牌对齐度,。

5. **用户界面与交互层 \(S5\)**：评估加载时长、地图或卡片的交互流畅度。

## **按评估粒度分层**

从宏观到微观将评测体系分为三个维度：

- **系统效率层**：关注底层基础设施，如响应延迟、Token 消耗成本、工具调用次数等,。

- **会话级别 \(Session\-level\)**：关注整个对话生命周期，包括任务成功率、轨迹质量（是否陷入死循环）及 Agent 的自省能力,。

- **节点/算子级别 \(Node\-level\)**：深入到每一个具体操作，如工具选择是否正确、参数提取是否精准、步骤效用分析等。

## **按业务维度分层**

根据业务复杂度进行垂直拆分：

- **第一层：业务场景**。描述智能体可以完成的领域（如前端系统、金融风控、医疗辅助）。

- **第二层：具体任务**。在特定场景下定义的具体功能（如测试用例生成、代码注释、SQL 生成等）。

在实际工程中，建议采用以下闭环流程：

1. **运行端到端评测**：获得整体基线分数。

2. **错误归因分析**：通过系统日志或人类抽检，识别失败案例集中在哪个环节（如 45% 是因为搜索查询质量差）。

3. **构建该环节的专项分层评测**：准备针对该环节的“黄金数据集”和评估标准。

4. **快速迭代优化**：在该层级进行高频调优，因为分层评测成本更低、反馈更快。

5. **二次端到端验证**：修复局部问题后，重新跑全链路评测，确保局部改进真正转化为了整体效果的提升。

端到端评测决定了 Agent “**能否发布**”，而分层评测决定了开发者“**该修哪里**”。

# **线下评测VS线上评测**

![Image](https://api3-eeft-drive.feishu.cn/space/api/box/stream/download/authcode/?code=NzMwZjg2YjM2OGU5ZGJiNjEzODkwMGM3NTg5ZGE5ODlfNDk2ZmU5YzYzYTU4MTQ4NzEwZTA0NTBiNjcxNWE0NGVfSUQ6NzYyNTI4MDYyNDYzNDYxMjY4M18xNzgyMTk5MDAzOjE3ODIyODU0MDNfVjM)

线下评测（通常指发布前实验、模拟、评测）与线上评测（通常指生产环境的可观测性、持续监控）在 AI Agent 的质量保障体系中承担着不同的角色。

它们的**侧重点**可以从评测环境、数据来源、核心目的及评估手段四个维度进行区分：

## **线下评测 \(Offline Evaluation\)**

线下评测主要发生在 **Agent 发布前**，是在受控环境下进行的“模拟考试”。

- **侧重点**：**基准验证与回归测试**。重点关注 Agent 在预设场景下的性能是否达标，以及新版本是否优于旧版本。

- **评测数据**：主要依赖 **“黄金数据集 \(Golden Dataset\)”**，即由专家精心策划、包含标准答案（Ground Truth）和丰富元数据的提示词与场景集。

- **核心目的**： 

    - **准入验证**：回答“系统是否可以发布？”。

    - **回归保护**：确保模型更新或 Prompt 调优后，原有能力没有退化。

    - **安全与红队测试**：在暴露给真实用户前，发现潜在的漏洞、偏见或幻觉风险。

- **典型手段**：利用模拟器模拟多轮对话、专家评分或利用 **LLM\-as\-a\-Judge** 进行自动打分。

## **线上评测 \(Online Evaluation\)**

线上评测发生在 **Agent 部署后**，是对其在“实战”中表现的持续观测。

- **侧重点**：**真实表现与业务对齐**。重点关注 Agent 在面对不可预测的真实用户输入时的适应性、稳定性和业务价值实现。

- **评测数据**：源自**生产日志 \(Production Logs\)** 和实时交互轨迹。它记录了从输入到工具调用再到最终响应的完整链路。

- **核心目的**： 

    - **监测与预警**：发现离线测试未能覆盖的边缘案例（Edge Cases）和性能漂移。

    - **链路追溯 \(Tracing\)**：通过分布式链路追踪，将用户投诉（结果标签）与具体的搜索、路由或工具调用步骤关联起来，定位根因。

    - **效果闭环**：通过真实业务指标（如“解决率”、“投诉率”、“订单转化率”）来反推 Agent 的实际价值。

- **典型手段**：基于 **OpenTelemetry \(OTel\)** 等标准的链路采集、用户反馈收集（如一键采纳率）、生产数据抽检。

核心差异总结表：

线下评测像“期末考试”，决定了 Agent **“能否发布”**；而线上评测像“实战监测”，决定了 Agent **“在实际运行中是否真的有用且安全”**。两者的闭环——即将线上发现的失败案例回流至线下扩充“黄金数据集”——是提升 Agent 效果的关键路径。

# 打分评估VS对比评估

在 AI Agent 的评测体系中，**打分评估**和**对比评估**是两种核心的评估逻辑，它们在参照系、输出形式、适用阶段及数据要求上有着本质的区别。

## **打分评估 \(Point\-wise / Absolute Scoring\)**

打分评估是对**单个候选内容进行独立评估**，通常不与其他对象进行横向对比。可以类比为“**体检报告**”——它只关注待测对象自身的指标（如血压是否达到标准），其参照系是**绝对标准**、预设的 KPI 或历史自我（如上一个版本、法律法规等）。

打分评估的评估目的是**准入验证（回答“能否发布？”），** 判断 Agent 的表现是否达到了上线要求的硬性指标（如准确率 ≥ 90%）。

**输出形式与要求：**

- **结论形式：** 二元化结论（通过/不通过）或绝对分值（如 85/100）。

- **数据要求：** 仅需待评对象的数据加上明确的阈值即可开展。

## **对比评估 \(Pair\-wise / Relative Evaluation\)**

对比评估是通过**比较两个（成对比较）或多个（列表比较）候选对象**来确定它们的相对质量或排名。可以类比为“**运动员排名**”——它不看绝对数值，而是看在同一赛场上谁表现更好。其参照系是**其他竞争对象**（如同行的竞品或不同的技术方案）。

**评估的目的包括：**

- **选型决策（回答“用 A 还是用 B？”）：** 在多个备选模型或供应商方案中挑选出最优者。

- **竞争优化（回答“如何超越竞品？”）：** 针对市场上已有的顶尖产品进行对标，寻找自身差距。

- **迭代验证（回答“比上一版改进了吗？”）：** 用于内部测试，确认新版本的模型或提示词（Prompt）优化后，效果是否优于历史版本。

**输出形式与要求：**

- **结论形式：** 排序（A \> B \> C）、相对差距（A 比 B 高 20%）或 GSB（Good/Same/Bad）结论。

- **数据要求：** 必须具备**多对象同条件数据**，即让所有模型在同一个测试集上跑出结果进行对比。

总结对比：

在实际工程中，评估策略的选择取决于评测的具体需求。通常建议在**研发早期选型**时多采用对比评估，而在**产品发布前的质量把关**阶段更多地依赖打分评估。

# 闭环策略（Evaluation → Diagnosis → Optimization）

在 AI Agent 的效果工程中，评测本身只是发现问题的第一步。为了确保每次评测都能真正转化为系统能力的提升，需要构建闭环策略：

## 闭环逻辑概览

闭环策略包括三个核心环节：

1. **评估（Evaluation）**

    - 目的：发现系统在实际任务中的表现和缺陷。

    - 数据来源：端到端评测结果、分层评测数据、线上业务指标。

    - 输出：整体得分、异常案例、失败轨迹。

2. **归因（Diagnosis）**

    - 目的：分析问题产生的根因，为优化提供明确方向。

    - 方法：

        - **分层分析**：根据架构层级（Goal/Planner/Tool/Memory/Knowledge）定位瓶颈。

        - **行为轨迹分析**：通过会话日志或链路追踪发现错误节点或决策偏差。

        - **数据切片分析**：按任务类型、场景或用户群体拆分指标，发现弱点。

    - 输出：问题列表及优先级，明确哪个模块或环节需要改进。

3. **优化（Optimization）**

    - 目的：针对归因结果改进系统性能，实现效果提升。

    - 方法：

        - **Prompt/Policy优化**：改进规划策略、工具调用指令、任务拆解逻辑。

        - **工具与技能迭代**：增强工具接口稳定性、升级技能组合能力。

        - **知识增强**：更新知识图谱、RAG检索策略、Memory策略。

        - **模型调优**：微调 LLM 或选择更优模型，提升推理、生成能力。

    - 输出：改进后的系统版本，可再次进行端到端验证。

---

## 闭环流程示意

闭环策略通常按以下步骤实施：

1. **端到端评测**
获得整体任务成功率、用户满意度等指标，确认系统是否达到发布门槛。

2. **失败案例归因**

    - 按架构层级分析每个失败案例，例如：

    ```Plain Text
    任务失败 100例：
    - 逻辑规划错误 40例
    - 工具调用失败 30例
    - RAG召回不准确 20例
    - 输出包装问题 10例
    ```

3. **分层专项评测**
针对发现的薄弱环节构建黄金数据集和测试指标，进行快速迭代优化。

4. **优化实施**
修复具体问题，优化策略或模块。

5. **二次端到端验证**
重新运行全链路评测，确保局部改进转化为整体性能提升。

6. **持续闭环**

    - 将线上发现的新问题、用户反馈或生产异常数据回流至线下评测数据集。

    - 建立周期性评估、归因、优化循环，实现持续能力提升。
