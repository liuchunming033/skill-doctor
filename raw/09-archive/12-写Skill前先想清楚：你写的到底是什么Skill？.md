## 核心观点

很多人刚开始写 Skill 的时候，会陷入一个很自然的冲动——"先写再说"。但 Claude Code 核心开发者 SACC 说：**写 Skill 最关键的步骤不是写，而是想清楚你要写的到底是什么类型的 Skill。**

不同类型的 Skill，设计思路、评估方式、维护策略都不一样。如果一开始类型没想清楚，后面越写越别扭。

---

## 四大 Skill 类型

### 1. 触发式 Skill（Trigger Skill）
**定义**：当系统检测到特定事件或条件时自动触发。

**典型场景**：
- 代码审查 Skill：每次 PR 创建时自动触发
- 部署前检查 Skill：每次部署前自动运行安全检查
- 日志监控 Skill：检测到异常模式时触发排查流程

**设计要点**：
- 触发条件要精确，避免误触发
- 执行结果要有明确的反馈机制
- 要考虑并发和重入问题

### 2. 工具式 Skill（Tool Skill）
**定义**：提供特定能力，由 Agent 在需要时自主决定是否调用。

**典型场景**：
- 数据分析 Skill：Agent 需要分析数据时调用
- PDF 处理 Skill：Agent 需要处理 PDF 时调用
- API 调用 Skill：Agent 需要与外部系统交互时调用

**设计要点**：
- name 和 description 要语义清晰，让 Agent 知道什么时候该用
- 输入输出要标准化
- 要有清晰的错误处理

### 3. 流程式 Skill（Workflow Skill）
**定义**：封装一个完整的工作流程，有明确的开始和结束。

**典型场景**：
- 站会汇报 Skill：汇总任务、代码、聊天记录，生成站会内容
- 事故响应 Skill：创建报告、同步状态、通知相关人员
- 发布流程 Skill：代码合并、构建、测试、部署、通知

**设计要点**：
- 流程步骤要清晰，但不要过度约束顺序
- 每个步骤要有明确的输入输出
- 要有状态管理（记住做到哪一步了）

### 4. 偏好式 Skill（Preference Skill）
**定义**：固化团队的偏好、规范、判断标准。

**典型场景**：
- 代码风格 Skill：规定命名规范、文件结构、组件组织方式
- 审查标准 Skill：规定审查优先级、安全边界、性能要求
- 报告格式 Skill：规定输出格式、内容结构、语气风格

**设计要点**：
- 用 gotchas 写坑点，比写教程更有价值
- 要区分"必须遵守"和"建议遵守"
- 要定期更新（团队偏好会变化）

## 如何判断你的 Skill 属于哪一类

问自己三个问题：

**1. 这个 Skill 由谁触发？**
- 由事件/条件自动触发 → 触发式
- 由 Agent 自主决定 → 工具式
- 由用户明确指令触发 → 流程式或工具式

**2. 这个 Skill 有没有固定流程？**
- 有固定流程，每次步骤基本一致 → 流程式
- 没有固定流程，每次调用方式不同 → 工具式
- 不是流程，是约束和规范 → 偏好式

**3. 这个 Skill 的核心价值是什么？**
- 自动化重复劳动 → 流程式
- 补充 Agent 能力短板 → 工具式
- 固化团队经验 → 偏好式
- 自动化响应事件 → 触发式

---

## 混合类型

很多 Skill 不是纯粹的单一类型。一个 Skill 可以同时具备多种属性：

- **工具 + 偏好**：一个 API 调用 Skill（工具式），同时规定了调用规范（偏好式）
- **流程 + 触发**：一个部署 Skill（流程式），由合并事件自动触发（触发式）
- **工具 + 流程**：一个数据分析 Skill（工具式），内部包含固定的分析流程（流程式）

关键不是找到纯类型，而是**理解你的 Skill 主要属于哪一类，然后按那一类的设计原则去主导设计**。

---

## 类型决定维护策略

| 类型 | 维护重点 | 评估重点 |
|------|---------|---------|
| **触发式** | 触发条件精确性 | 误触发率、漏触发率 |
| **工具式** | 接口稳定性 | 调用成功率、响应质量 |
| **流程式** | 流程完整性 | 完成率、步骤偏差 |
| **偏好式** | 规范一致性 | 输出是否符合规范 |

---

## 总结

写 Skill 前先想清楚类型，不是为了分类而分类，而是因为：

1. **不同类型的 Skill 设计原则不同**——把流程式当工具式写，会缺步骤；把偏好式当流程式写，会太死板
2. **不同类型的 Skill 评估方式不同**——触发式看触发率，工具式看调用质量，流程式看完成率，偏好式看一致性
3. **不同类型的 Skill 维护节奏不同**——偏好式要定期更新，工具式要关注接口变化，流程式要关注步骤漂移

**先想清楚你写的是什么类型的 Skill，再动笔。这个思考过程本身就在帮你把 Skill 做对。**
