## 核心观点

当一个团队里已经有几百个 Skills 在跑的时候，真正麻烦的问题就来了：这些东西怎么分发？怎么管理？怎么避免越做越乱？

本期继续读 Claude Code 核心开发者 SACC 的文章《Lessons from Building Claude Code: How We Use Skills》，看看 Claude 是怎么运营一整套 Skills 生态的。

---

## 一、分发策略

Claude 内部主要有两种做法：

### 方式一：放进代码库
把 Skills 放进代码库，统一跟着项目走，放在 `.cod/skills` 目录里，团队成员拉下代码就能一起用。

**适合小团队**——人少，协作范围也集中，直接放进仓库最省事。

### 方式二：做成插件放进内部市场
谁需要谁再装。

**适合大规模**——团队一大、Skill 一多，塞进仓库的每一个 Skill 都会给系统多加一层负担，最后很容易变成大家明明只需要其中一小部分，却要一起背着一整包东西走。

**市场模式更合理**：让团队自己决定装哪些、哪些先不装，整个系统会轻很多。

---

## 二、市场管理机制

很多人第一反应是"那是不是要有个审核委员会，专门批准哪个 Skill 能上线？"

**Claude 没这么做。** 他们的办法更像是**先放到试验区里跑一跑再看结果**：

1. 如果你做了一个 Skill 想让别人试用，可以先传到 GitHub 的沙盒目录
2. 再去 Slack 或论坛里告诉大家这个东西可以试
3. 接下来先看有没有人真的用起来，有没有人在持续反馈
4. 如果这个 Skill 已经开始有人持续在用，而且看起来确实有价值，它的作者就可以提 PR 把它挪进正式市场

**关键判断**：Skill 这种东西做出来太容易了，做烂也太容易了。如果没有一套筛选机制，最后市场里一定会堆满重复的、模糊的、没人维护的垃圾 Skill，真正好用的东西反而会被埋掉。

---

## 三、Skill 之间的依赖关系

比如有一个 Skill 专门负责上传文件，另一个 Skill 负责生成 CSV 并上传，后者其实就会依赖前者。

Claude 的做法不是给一套很重的依赖管理系统（声明版本、装配关系、处理一堆配置），它更像是一种**轻量协作**：

- 你只要在 Skill 里写清楚"如果要上传文件，就调用哪个 Skill"
- 只要那个 Skill 已经装好了，Claude 就会自己去调用

这套设计看起来不那么工程完备，但有一个现实好处——**简单**。因为 Skills 体系本身还在快速变化，这个阶段先让东西能接起来、能跑起来，比先上复杂机制更重要。

---

## 四、度量与数据驱动优化

一个 Skill 做出来之后，如果你根本不知道有没有人用、什么时候用、哪些本来该被调用的场景却没有被调用，那你其实没法优化。

Claude 用了 **pre-use hook** 去记 Skill 的使用情况——这个 hook 会在调用 Skill 之前触发，把时间、用户、Skill 名称、参数这些信息记下来。

有了这层记录之后，团队才能真正回答几个关键问题：
- 哪个 Skill 最受欢迎？
- 哪个 Skill 明明应该常用，但实际很少被触发？
- 还有哪些 Skill 基本没人碰，可能该下线了？

**真正被记录的不只是使用数据，更重要的是团队终于有了判断依据。** 以前可能只是凭感觉说这个 Skill 应该很有用，现在能看到真实情况，然后决定是继续优化 description、补文档，还是干脆删掉。

---

## 核心运营思路总结

Claude 这套运营思��的核心是：**先把反馈回路搭起来，不是先追求把系统设计到完美。**

| 维度 | 做法 |
|------|------|
| **分发** | 按团队规模选方式（小→代码库，大→市场） |
| **筛选** | 不靠权威拍脑袋，先看有没有人持续在用 |
| **优化** | 不靠感觉，靠数据 |
| **迭代** | 先进入真实使用，再在反馈里慢慢长出来 |

SACC 也没有把这套方法讲成什么标准答案。Claude 内部很多 Skill 一开始也就几行——说明加一个踩坑点，然后边用边补。

**重点不是先规划出一个完美版本再发布，重点是先让它进入真实使用，再在反馈里慢慢长出来。**
