## 核心观点

你开始写 Skill 了，写了几个以后发现——有时候写得很顺，有时候怎么调都不对。这背后的问题往往不是 Skill 本身写得好不好，而是你对 Skill 的理解还停留在"工具"层面，没有进入"设计"层面。

Claude Code 核心开发者 SACC 提出，写 Skill 需要经历一个心智模型的跃迁：**从"执行者"切换到"设计者"**。

---

## 执行者心态 vs 设计者心态

| 维度 | 执行者心态 | 设计者心态 |
|------|-----------|-----------|
| **关注点** | 让 Agent 完成这个任务 | 让 Agent 能稳定地在各种场景下完成任务 |
| **写 Skill 时** | 写清楚每一步怎么做 | 设计约束、边界、判断标准 |
| **遇到问题时** | 加更多指令 | 重新设计触发逻辑或结构 |
| **评估方式** | "这次跑对了" | "这个 Skill 整体上更可靠了" |

执行者心态在 Skill 数量少、场景固定的时候够用。但当 Skill 数量多起来，场景交叉起来，执行者心态就会碰到天花板——因为你每次改一个地方，都可能影响其他场景。

---

## 设计 Skill 的核心思考框架

### 1. 触发边界设计
Skill 的 name 和 description 决定了它什么时候被调用。这是 Skill 的第一道防线——**如果触发边界没设计好，Skill 要么频繁误触发，要么该触发的时候不触发。**

设计原则：
- description 要写触发条件，不是功能说明
- 用负控样本验证触发边界
- 考虑 Skill 之间的触发优先级

### 2. 约束设计
Skill 的核心不是告诉 Agent 每一步做什么，而是告诉它**边界在哪里**：
- 哪些事不能做（安全约束）
- 哪些假设不能默认（环境约束）
- 哪些顺序不能乱（执行约束）

### 3. 结构设计
Skill 是文件夹，不是文件。设计时要考虑：
- 主文件只写核心逻辑
- 参考文档、模板、脚本放在子文件里
- 配置文件独立管理

### 4. 可测性设计
在设计 Skill 的时候就要想好它怎么被评估：
- 什么样的输入算成功？
- 哪些场景最容易失控？
- 怎么证明它比不用 Skill 更好？

## 设计 Skill 时的常见陷阱

### 陷阱一：过度指令化
写得太死——"先做 A，再做 B，然后做 C"。这会让 Agent 失去灵活性，遇到边界情况时卡死。

**设计解法**：约束目标，不是约束路径。写"确保代码通过测试、规范构建"，而不是"先测试再构建"。

### 陷阱二：假设环境
默认 Agent 在某个特定目录运行、默认系统里有某个工具、默认用户偏好和设计者一致。

**设计解法**：在 Skill 开头显式检查环境假设，不满足时先做设置。

### 陷阱三：忽略遗忘
Skill 每次运行都是全新的，不记得上次做了什么。

**设计解法**：用日志记录运行历史，下次运行时读取，只处理新的变化。

### 陷阱四：没有退出策略
Skill 做完了，但环境被弄脏了——临时文件没清理、进程没关闭、配置被改了没恢复。

**设计解法**：在 Skill 设计时就定义清理步骤，确保每次运行后环境恢复干净。

---

## 从执行到设计的关键跃迁

1. **从写步骤到写约束**——不再告诉 Agent 每一步做什么，而是告诉它边界在哪里
2. **从写内容到写结构**——不再把所有内容塞进一个文件，而是设计文件夹结构
3. **从写指令到写可测性**——不再只关注"这次跑对了"，而是设计怎么持续验证
4. **从关注单次到关注系统**——不再只关心这个 Skill 好不好，而是关心整个 Skill 生态的协作

SACC 说了一句话：**"你写的不是 Skill，你写的是 Agent 的决策框架。"**

当你开始设计约束、设计触发边界、设计可测性的时候，你其实是在设计 Agent 怎么做决策。Skill 只是一个载体，真正的内容是**决策框架**。

---

## 总结

写 Skill 这件事，本质上是在做两件事：
1. **把人的经验翻译成 Agent 能理解的结构**
2. **设计一个可以被持续验证和改进的决策系统**

从执行者切换到设计者，意味着你不再问"这次跑对了吗"，而是问："如果换一个场景、换一个模型版本、换一个用户，这个 Skill 还能稳定工作吗？"

**好的 Skill 不是写出来的，是设计出来并在反馈中长出来的。**
