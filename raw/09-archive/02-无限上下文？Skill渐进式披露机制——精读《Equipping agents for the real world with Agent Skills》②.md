## 核心观点

上一期解读了 Skill 的定义（指令+脚本+资源的结构化文件夹），本期深入 Skill 的内部构成、渐进式披露的三层加载机制，以及它为什么能让 Claude 拥有近似无限上下文的工作方式。

## Skill 的文件结构

每一个 Skill 都是一个文件夹，核心文件是 `SKILL.md`——这项技能的使用说明书和入口。

在最上方是一段 **YAML front matter（元数据区）**，包含两个基本字段：
- **name**：技能名称（如 `pdf`）
- **description**：简要说明这项技能能做什么（如"从PDF中抽取文本和表单字段"）

系统在 Agent 启动时，会自动扫描所有技能的 `SKILL.md`，并把每个技能的 name 和 description 预加载到系统提示词中。这意味着 Claude 启动后就已经知道自己具备哪些技能以及这些技能大致做什么，而不需要一次性加载所有技能的全部内容。这种设计让技能可以被**索引而不被展开**。

## 渐进式披露（Progressive Disclosure）的三层加载

原文将渐进式披露比作一本组织良好的手册：**先读目录确定主题，再读相关章节掌握要点，最后只有在需要时才查阅详细的附录**。

### Level 1：YAML 元数据层
- **文件位置**：`SKILL.md` 顶部的 YAML front matter
- **内容**：技能的 name 和 description
- **加载方式**：始终加载（约消耗 100 个 token）
- **作用**：告诉模型有哪些技能以及它们是做什么的

### Level 2：正文内容层
- **文件位置**：`SKILL.md` 的 Markdown 正文部分
- **内容**：技能的核心指令与 SOP 逻辑
- **加载方式**：技能被触发时加载（平均消耗 < 5000 个 token）
- **作用**：当 Claude 决定使用这个技能时，才真正打开这部分内容

### Level 3 及以上：附属文件层
- **文件位置**：技能文件夹中除 `SKILL.md` 外的其他文件（Markdown 文档如 `forms.md`、`reference.md`；配置文件如 `.json`；脚本如 `.py`）
- **加载方式**：按需加载（理论上 token 上限是无限的）
- **作用**：Claude 只有在 SOP 指令中被要求访问这些文件时，才会读取或执行它们

## 认知与执行的分离

为什么 Level 3 可以包含大量文件甚至脚本，却不会撑爆上下文？

核心在于**认知与执行的分离**：

- **认知空间（上下文窗口）**：只包含模型推理所需的文字信息——Level 1 的元数据加上被触发技能的部分 Level 2 内容。体量小，可直接加载进上下文。
- **执行空间（文件系统和执行工具）**：包含所有外部文件、脚本和数据。任务执行发生在外部虚拟机中，Claude 不会阅读这些脚本，而是调用工具执行。

**例子**：在 PDF Skill 里，当 Claude 读到 `forms.md` 的指令"请运行 scripts/extract_fields.py"时，它不会把那份 5000 行的 Python 代码加载进上下文，而是向虚拟机发出 `run bash command` 命令。脚本运行结束后，虚拟机返回一小段结果（如"字段提取完成，结果已保存为 forms.json"），在上下文里只看到这个结果，不承担脚本执行本身。

原文写道：
> "Agents with a file system and code execution tools don't need to read the entirety of a skill into their context window. The amount of context that can be bundled into a skill is effectively unbounded."

**模型并不扩大上下文窗口，而是通过外部化执行，让技能包的体积与上下文无关。**

## 外部脚本的确定性与效率

语言模型可以写代码，但在处理排序、提取或数学运算时效率低、成本高。外部脚本执行既快速又可重复，结果可验证。因此 Skill 不仅让 Claude 知道怎么做，还确保每次都能做对。

## Skill 被触发时的动态变化

1. **初始阶段**：上下文中包含系统提示、所有技能的元数据片段（name 和 description）以及用户的初始输入
2. **技能被触发**：如用户请求"请帮我填写这份PDF表单"，Claude 判断任务与 PDF 技能相关，调用工具读取 `PDF/SKILL.md` 的正文内容
3. **按需展开附录**：如果正文中提到"请阅读 forms.md"，Claude 再次调用工具读取 `forms.md` 的内容；未被提及的 `reference.md` 仍然留在文件系统中
4. **执行外部脚本**：当 SOP 要求运行脚本时，Claude 发出执行命令，脚本在外部虚拟机中运行，只将执行结果返回上下文
5. **任务完成**：Claude 综合当前上下文中的内容（SOP 读取结果 + 脚本输出）继续生成最终响应

从始至终，上下文都只包含完成当前任务所需的信息，而技能包的完整内容则留在外部世界，随时可被访问。

## 三层逻辑总结

1. **结构上**：Skill 是一个包含 `SKILL.md` 的文件夹，其顶部元数据在系统提示中被预加载
2. **机制上**：渐进式披露让 Skill 分为三层（目录→正文→附录），Claude 按需展开，信息逐层显露
3. **认知与执行分离**：模型的思考空间始终清亮，执行空间可以无限扩展；外部脚本带来了确定性、效率和可复现性

**近似无限上下文并不是模型参数的魔法，而是一种架构思想——让模型的上下文只装载需要思考的那一页，而把真正的世界留在它随时可访问的文件系统里。**
