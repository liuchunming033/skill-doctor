# 这是你想要的"AI 驱动的产品需求工作流"吗？

> 来源：[飞书文档](https://nio.feishu.cn/wiki/C4HrwfqtQinyQ8kwp9zc09iMn2c)
> 转换时间：2026-06-05

## 方案总览

建议先看：[为什么当下大家采用Obsidian作为AICoding的ContextOS？](https://nio.feishu.cn/wiki/RU7LwZyUfijduOkEP16cM40ZnDr)

### 设计核心思想

完全遵循 Karpathy LLM-Wiki 极简知识体系（[karpathy/llm-wiki.md](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)），不自创架构、不搞复杂目录、降低团队全员理解成本。

构建一套：人负责投喂原始意图与上下文，AI 负责记忆、推理、生成、归档，人工只做决策评审的 AI 原生产品需求流水线。

### 核心分工

- 人类产品经理：只做「低摩擦原始信息投喂 + 最终决策评审」，不整理、不结构化、不写正式文档
- 产品Agent：严格按依赖顺序自动完成「Spec 生成 → 技术方案生成 → 测试用例生成」
- Obsidian + Git：唯一可信知识底座 & 版本归档仓库
- 飞书项目 / 流水线：唯一流程枢纽、门禁评审、任务流转载体

### 技术栈

- Obsidian（产品知识库）
- Wiki-Repo（代码知识图谱）
- 飞书项目流水线
- 产品 Agent（三大标准化依赖技能）
- Git 版本管控Obsidian知识库

## Obsidian 知识库目录（标准 LLM-Wiki 范式）

```Plain Text
Vault/
├── raw/          # 原始素材收件箱（LLM-Wiki 原生 Raw 层）
├── wiki/         # AI 结构化知识库（LLM-Wiki 原生 Wiki 层）
│   ├── business/ # 产品业务知识：概念、决策、用户、战略、规则
│   └── repowiki/ # 代码技术知识：技术栈、服务架构、接口、数据表、代码约束
└── specs/        # Agent 正式产物归档层（业务专属交付物）
└── CLAUDE.md     # 全局约束
```

目录职责定义：

- **/raw 原始素材层（纯输入、只读、不可篡改）**
  - 核心规则：人随意投喂，AI 只读不修改、不整理、不删除
  - 存放内容：会议纪要、用户访谈、聊天记录、灵感碎片、网页剪藏、录音转写、竞品碎片、AI 对话草稿
  - 核心特性：低摩擦输入、无分类、无结构化、无需关联 ID、永久留存原始证据

- **/wiki 结构化知识层（AI 全权维护、人只读）**
  - wiki/business 产品业务知识库：沉淀所有长期产品认知，AI 自动聚类、关联、更新、去重。包含：领域概念、历史设计决策、用户画像、产品战略、设计原则、边界约束、竞品体系。标签驱动分类：`#concept` `#decision` `#user` `#strategy` `#principle`
  - wiki/repowiki 代码知识库（内嵌式，不破坏顶层极简）：沉淀全量代码库知识图谱，为技术方案适配提供底层约束。包含：技术栈选型、微服务架构、接口定义、数据库表结构、依赖关系、权限体系、异常机制、历史技术方案、部署约束

- **/specs 正式产物归档层（流水线最终输出）**
  唯一权威交付物目录，全自动化生成，人工禁止修改，以飞书系统自动生成 Issue RID为唯一文件夹维度归档。

```Plain Text
specs/
├── ISSUE-xxxx/
    ├── prd-ISSUE-xxxx.md
    ├── tech-solution-ISSUE-xxxx.md
    ├── test-case-ISSUE-xxxx.md
    └── review-record-ISSUE-xxxx.md
```

- **CLAUDE.md** 全局约束规则（强制执行）
  1. `/raw`仅人工写入、AI只读，禁止任何结构化修改
  2. `/wiki`由AI全权维护，人工仅浏览、禁止随意修改
  3. `/specs`所有文件AI自动生成，人工禁止编辑
  4. 技术方案生成必须优先复用repowiki现有代码能力，禁止凭空设计

## 全局唯一标识规范

1. RID = 飞书项目自动生成的原生 Issue ID（如 ISSUE-8921）
2. 全程零手动填写 ID，系统全自动绑定流转
3. RID 唯一作用：
   - 飞书任务唯一主键
   - `specs` 目录产物唯一归档维度
   - 所有正式文档元数据唯一关联键
4. `raw`、`wiki` 知识库无需绑定 RID，由 Agent 根据需求语义自动检索召回关联内容

## 产品 Agent 三大标准化技能

### 通用规则：技能依赖与自动补全机制

1. 技能固定依赖链
   - Skill01：无依赖（流水线入口）
   - Skill02：强依赖 Skill01 输出
   - Skill03：强依赖 Skill01 + Skill02 输出
2. 强制调用规范
   - 不允许跳过前置技能直接执行后置技能
   - 任意后置技能被调用时，Agent 自动校验前置产物是否存在
   - 前置产物缺失 → 自动优先执行前置依赖技能，补齐后再执行当前技能
3. 全局变量：所有技能自动读取流水线全局变量：`{{RID}}、{{title}}、{{description}}、{{module}}、{{priority}}`

### Skill01：Spec 自动生成（入口技能）

- 技能 ID：SK-01-SPEC
- 技能名称：PRD & API Spec 自动生成
- 前置依赖：无
- 后置依赖：被 Skill02、Skill03 依赖
- 执行时机：流水线启动首个执行节点

**技能能力说明：**
接收飞书 Issue 基础信息，内置 Obsidian 全库语义检索，自动融合 raw 原始素材、wiki/business 产品领域知识、wiki/repowiki 代码技术约束，自动降噪、对齐历史决策、校验技术可行性，输出标准、可评审、可落地的 PRD + API Spec，为整条流水线提供唯一业务基准输入。

**检索范围：**
1. `/raw`：全部未结构化原始素材
2. `/wiki/business`：产品概念、历史决策、用户模型、产品原则、业务约束
3. `/wiki/repowiki`：现有架构、服务、接口、数据表、权限、技术栈约束

**技能标准 Prompt：**

```Plain Text

【全局上下文】RID：{{RID}}、标题：{{title}}、描述：{{description}}、模块：{{module}}、优先级：{{priority}}

【前置依赖】无（流水线入口技能）

1. 自动降噪，过滤无效、过时、闲聊内容，只保留有效业务与技术信息

## 产品需求工作流

### Step1 产品低摩擦投喂原始上下文

1. 产品日常通过各种渠道持续向 `raw/`存入所有零散原始信息
2. **零规范、零整理、零结构化、零分类**，仅做信息收集
3. 全程不创建文件夹、不关联ID、不撰写需求文档

### Step2 飞书项目上创建一句话需求，触发自动化流水线

1. 飞书项目新建Issue，系统**自动生成唯一RID**
2. 产品仅填写3项极简信息：业务模块、优先级、一句话需求意图
3. 启动AI流水线，系统自动注入全局变量

### Step3 Skill01 生成 PRD/Spec

1. Agent 自动对 Obsidian 全库做语义召回、信息整合、冲突校验
2. Obsidian信息缺失或git地址链接不上，则暂停
3. 信息完整则生成标准 Spec，关联到飞书项目字段spec

### Step4 自动链式执行后置技能（依赖自动补齐）

1. 自动执行 Skill02 生成技术方案，生成的文档关联到飞书项目design字段
2. 自动执行 Skill03 生成全套测试用例，生成的文档关联到飞书项目case字段
3. 所有产物自动带入RID���自动关联知识库元数据

### Step5 研发+测试 门禁评审（唯一人工干预节点）

1. 流水线自动展示新旧版本Diff，评审仅聚焦变更内容
2. 评审分支：
   - 通过：填写结构化评审意见，流转至归档节点
   - 不通过：填写标准化修改意见，Agent自动迭代重生成
3. 评审记录自动沉淀留存

```JSON
评审人：
问题类型：业务理解偏差 / 术语不一致 / 违背历史决策 / 技术不可行 / 边界缺失 / 用例不全 / AI 幻觉
修改建议：
是否采纳：是 / 否
```

### Step6 全自动回流归档 & 版本锁定

1. 评审通过后，全套产物自动回流至Obsidian的 `specs/{{RID}}/`
2. 自动补全双向知识库关联、元数据
3. 自动Git提交并打版本Tag（RID-v1.0）
4. 全局知识看板自动更新

## 核心优势

1. **零自创体系**：完全遵循Karpathy LLM-Wiki行业通用范式，团队无认知负担
2. **极致轻量化**：顶层仅3个目录，结构极简、极低维护成本
3. **彻底杜绝AI幻觉**：生成强绑定repowiki真实代码知识，100%适配现有架构
4. **严格技能依赖**：流水线可控、可溯源、不会乱生成、不会缺前置输入
5. **全自动自补齐**：误操作、跳过步骤也能自动修复，鲁棒性极强
6. **人机精准分工**：人只做创意与决策，AI承担全部结构化产出
7. **全链路可追溯**：原始素材→知识依据→产物→评审全程闭环
