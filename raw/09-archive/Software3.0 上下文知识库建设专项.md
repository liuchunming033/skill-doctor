# Software3.0 上下文知识库建设专项

## 背景与目标

### 1.1 项目背景

Software3.0 核心变革为AI 原生、上下文驱动、流程标准化、资产可沉淀。

当前各产品线存在知识分散、知识复用率低、新人上手成本高、跨团队协作信息不对称等问题。

为统一全链路研发知识体系，实现 "需求 - 设计 - 开发 - 测试" 全流程资产结构化沉淀、可检索、可复用、可自动化流转，特启动本次上下文知识库建设专项项目。

### 核心目标

1. 知识库建设：12 条产品线落地产品、设计、技术、测试四维完整知识库体系，补齐研发全流程知识资产；
2. 知识库消费：标准化产出 SPEC 需求规格、技术方案、测试用例三类核心研发产物；
3. 通过工作坊 + 竞赛模式，统一团队认知、沉淀方法论，为后续全面推广铺垫标准流程。
4. 识别建设过程中的痛点、卡点，为后续企业级知识中心的建设积累需求和经验。

## 实施方案

采用统一赋能 + 实战落地 + 竞赛评优模式，保证落地质量与全员参与度，逐渐形成标准化落地范式。

### 专项工作坊（统一赋能）

1. 参加人员：12条产品线核心产品、UI/UX 设计、后端 / 前端开发、测试工程师、项目负责人；
2. 培训核心内容：
   - Software3.0 知识库建设标准规范、目录结构、沉淀口径；
   - 四维知识（产品 / 设计 / 技术 / 测试）拆解方法；
   - SPEC、技术方案、测试用例标准化输出要求；
   - 实操案例分享
3. 培训目标：全员统一认知、掌握知识沉淀方法、熟练产物输出使用。

### 知识库建设大赛（实战落地）

1. 参赛单元：12条产品线；
2. 评价维度：知识库完整性、内容规范性、产物标准化程度、流水线跑通成功率、知识复用价值；具体参考下文"上下文知识库核心建设内容"。
3. 竞赛产出：完整四维知识库、三类标准化产物；具体参考下文"项目核心产出要求"和"三大标准化研发产物"。
4. 激励机制：设置标杆团队、优秀个人奖项，沉淀优秀案例。

## 上下文知识库核心建设内容

一句话原则：建设的内容是产品的决策和洞察，跳过操作和状态。决策不常变，操作天天变；洞察可以跨版本复用，状态隔一个 sprint 就过时。频繁变的内容是数据，而相对稳定的内容才是知识。

哪些内容适合加入知识库，两个核心筛选维度：
- **半衰期**：这个知识多久变一次？变了是否会让旧知识变成误导？
- **合成价值**：LLM 跨多个来源整合这类知识，能产生对你有用的洞察吗？

## 三大研发产物

这三类产物能否达到"可直接作为开发、测试唯一需求依据"的质量，取决于一件事：知识库的 ingest 质量。

知识库里有含糊的需求描述 → 生成的 SPEC 也含糊。
知识库里缺少异常场景 → 测试用例覆盖率不够。
知识库里只有架构概述 → 技术方案就只有壳。

"垃圾进，垃圾出"在知识库这里���样成立。

### 需求规格文档（SPEC）

SPEC 本质上是对已有知识的重新组织和格式化。产品定位、用户故事、功能详述、验收标准——这些恰好是你 ingest 进知识库的核心内容。知识库越完整，生成的 SPEC 越少需要人工补填。

SPEC包含：业务背景、需求目标、用户故事、功能详述、非功能需求（性能 / 安全 / 兼容）、依赖约束、验收标准、风险说明，可直接作为开发、测试唯一需求依据。

> ⚠️ 真实限制：
> - 产物需要逐节审核，主要工作是校正而不是创作。

### 技术方案文档

知识库中包含架构决策的 WHY、技术债务记录、难点方案的权衡过程。缺少代码中的完整上下文，特别是AGENTS.md或者CLAUDE.md。能生成的是技术方案的框架和架构层，细节部分仍需结合代码库AGENTS.md或者CLAUDE.md填写。

技术设计方案初稿，包含：总体架构、模块设计、数据设计、接口设计、异常处理、性能保障、上线方案、风险评估、落地排期，满足可落地、可评审、可传承要求。

> ⚠️ 真实限制：
> - 用知识库生成"架构部分 + 约束边界 + 历史决策上下文"，作为可实施的技术方案的起点，而不是完整产物。

### 标准化测试用例

知识库中的线上问题复盘、Bug 统计、高频风险点、业务异常场景梳理。这些恰好是最难靠人脑枚举、最有价值的测试用例来源。

测试用例覆盖核心业务场景、边界场景、异常场景，格式统一、步骤清晰、预期结果明确，支持自动化脚本适配与批量回归复用。

> ⚠️ 真实限制：
> - 正向核心路径用例质量高
> - 边界用例质量取决于知识库里的异常场景积累厚度
> - 产物需要人工确认审核

## 项目价值

**对于Software3.0的价值：**
1. 将隐性业务、技术、测试经验转化为可沉淀、可检索、可复用的结构化知识资产；
2. 自动化：降低文档撰写、更新、管理成本，提升需求设计阶段效率；
3. 上下文质量、一致性、完整性的极大提高，需求设计阶段产物质量的提高，有效促进软件交付的质量。

**牵引平台的价值：**
在Software3.0 产品上下文知识库过程中的痛点、卡点，整理成需求：
1. 输入给KnowledgeSight平台，促进平台建设，最终将'产品上下文知识库'的建设更加高效、高质量。
2. 输入给ModelSight，促进ASR\OCR\多模态模型的建设。
3. 推进CodeSight的落地，输出代码库的Repowiki自动集成到上下文知识库。
