## 核心观点

很多团队用上 AI 之后，会发现 AI 会放大组织已有的能力，也会放大能力缺口。如果关键经验还没有 Skill 化，AI 产出的速度越快，组织的混乱程度也会越快。

Claude Code 核心团队工程师盘了一遍 Claude 内部几百个 Skill，发现这些 Skill 大体可以落到九类里。这套分类真正有价值的地方在于：**它能帮你看清楚团队哪些能力已经沉淀成系统，哪些关键经验还停留在人脑里。**

这九类 Skill 可以收束到**四个环节**：**认知 → 生产 → 验证 → 交付**。

---

## 第一环节：认知

认知阶段要解决的是：Claude 到底知不知道自己面对的是一个什么系统？从哪里获得理解？遇到异常时从哪里开始查？

### 1. 内部库与接口文档
负责补充组织内部的专属知识。Claude 对通用技术知识本来就不差，但它不了解你们内部库、内部平台、内部命令行工具、内部设计系统到底怎么用。

**最有价值的内容**：最常用的调用方式、参考片段，以及那些公开文档里没有写但团队反复踩坑的 gotchas。

> **原则**：不要写 Claude 本来就知道的东西，要写它推不出来的东西。

### 2. 数据获取与分析
很多团队已经开始让 Agent 写代码、写文档、写方案了，但一到分析任务就开始空转，问题往往在于数据入口。

这类 Skill 的作用就是把数据源凭证、仪表板 ID、常用查询语句、分析脚本、查询模板整理成 Claude 能直接调用的东西（如漏斗分析、分群、对比、监控面板接入）。

### 3. 故障排查手册
这类 Skill 会接收一个症状（如一条 Slack 告警、一段错误签名、一个请求 ID），然后 Claude 不只是给建议，它会自己去查日志、查监控、串链路，最后输出结构化排查结果。

**特别重要**：因为它承接的是低频但认知负担极高的问题。把这类经验做成 Skill，本质上是在把资深工程师的排障思路变成组织可以复用的基础设施。

---

## 第二环节：生产

生产阶段要解决的是：Claude 怎么稳定地产出代码、流程结果和标准化产��，而不是每次都从零起步。

### 4. 代码脚手架与模板
给 Claude 一套稳定的起手式。很多事情没必要每次从零搭——新建前端组件、新开接口、新起数据库迁移、新建工作流、新拉微服务模板。

没有这类 Skill，Claude 每次都得重新决定目录结构、命名习惯、文件分层和默认骨架，最后产出的东西很容易漂。

### 5. 业务流程与团队自动化
处理高频重复、规则清楚的流程工作，如：
- **站会汇报**：把任务系统、代码仓库和聊天工具里的历史汇总起来，自动整理成站会内容
- **事故报告**：自动创建报告、同步状态、通知相关人

这类 Skill 会持续吃掉团队里那些低价值、重复性的劳动，往往是最容易被快速感知到收益的一类。

---

## 第三环节：验证

验证阶段真正关心的是：**Claude 能不能证明自己写出来的东西真的能工作。**

### 6. 代码质量与审查
负责检查代码、文档或方案本身的质量——安全风险、结构问题、命名习惯、性能隐患、团队规范审查清单。

**"Fresh eye review"**：不要只让 Claude 用刚写完代码的那套视角回头审自己。更好的做法是故意制造一个新鲜视角，让它专门来挑毛病，再把问题修掉，反复迭代。这背后是在对抗模型的自我一致性偏差。

### 7. 产品验证
验证类 Skill 值得一个工程师花一周时间专门去打磨，因为真正决定质量的不只是生成能力，还有反馈闭环。

这类 Skill 会配合无头浏览器、TMUX 等终端复用工具以及各种脚本和断言，去测试代码是否真的正确工作。

**重要技巧**：让 Claude 录制输出视频，这样你看到的就是完整执行过程——它到底测了什么、点了什么、停在了哪里、有没有真的走完关键路径。还可以把编程式断言直接放进 Skill，让 Claude 在每一步都强制检查状态。

**核心**：让 Claude 有能力证明自己做对了，而不只是交差。

---

## 第四环节：交付

交付阶段要解决的是：Claude 怎么把结果真正推进到系统里，并在后续阶段持续维护。

### 8. 持续集成与部署
负责代码合并、流水线部署、灰度回滚这一整条交付链路：
- **PR 监控**：监控合并请求状态，处理不稳定的流水线和合并冲突，条件满足后再推进自动合并
- **服务部署**：负责灰度发布，盯关键指标，在异常时自动回滚

很多团队的问题常常出在"一到部署就必须完全切回人工"，这说明 Agent 的能力链条只走到了一半。

### 9. 基础设施运维
处理基础设施和日常运维动作��尤其是高风险动作——查孤立资源、处理依赖升级、调查云资源费用异常、容器集群故障处理、执行待确认点的清理流程。

**特别强调护栏**：删资源、改环境、切流量这些事一旦失控代价很高。所以这类 Skill 更强调先把风险边界写清楚，再去谈自动化。

---

## 总结：一张组织能力地图

这九类 Skill 正好沿着一条完整的研发流程排开了：

| 环节 | 阶段任务 |
|------|---------|
| **认知** | 让 Claude 先理解系统、接上数据、掌握排障入口 |
| **生产** | 让 Claude 稳定地产出代码和流程结果 |
| **验证** | 让 Claude 不只是产出，还能做质量审查、完成产品验证 |
| **交付** | 把结果推进到代码合并、部署和后续运维里 |

把自己团队现有的 Skill 摆上去，很快就能看出来：
- 哪些地方已经沉淀下来了
- 哪些关键环节还停留在人脑里
- 哪些地方明明最需要 Skill，结果到现在还没被系统化

**团队落地 AI 研发时，真正的短板往往在于关键经验有没有被做成 Claude 能真正调用、执行、验证和复用的 Skill。**
