## 核心观点

你花两小时写了个 Skill，Claude 从来不调用；好不容易调用了，执行起来又僵化，结果不达预期。

Claude Code 核心开发者总结了构建 Skill 的九个最佳实践，本质上是在解决三个递进的问题：
1. **内容层**：Claude 能不能找到你的 Skill？找到之后能不能理解该做什么？（被看见和被理解）
2. **结构层**：Skill 能不能在不同场景下复用？会不会因为写得太死而失去灵活性？（可复用和不僵化）
3. **高级技术层**：Skill 能不能记住上次做了什么？能不能直接给 Claude 可调用的代码？能不能在需要的时候开启护栏？（有状态和可组合）

## 内容层：被看见和被理解

### 第一条：description 字段是给模型看的
很多人把 description 写成功能说明（"这是一个监控 PR 的工具"），但 Claude 启动会话时会扫描所有 Skill 的 description 来决定这个请求有没有对应 Skill。它需要的不是功能说明，而是**触发条件**——"当用户说 ... 时触发"。

**Description 决定的是什么时候该用它，这是 Skill 能不能被调用的第一道门槛。**

### 第二条：不要陈述显而易见的事情
Claude 已经知道大量通用知识。写 Skill 的时候最大的浪费就是重复它已经知道的东西。

**Anthropic 内部案例**：一个 frontend design Skill。工程师发现 Claude 生成的前端界面总是用 Inter 字体和紫色渐变，看起来很"AI味"，但客户反馈说不喜欢。于是他们写了个 Skill——这个 Skill 不教 Claude 怎么写 React（Claude 本来就会），它只告诉 Claude 一件事：**别再用 Inter 和紫色渐变了，客户不喜欢。**

Skill 的价值在于补充 Claude 默认不知道的上下文、偏好和坑。

### 第三条：构建 Gotchas 部分
Anthropic 发现任何 Skill 里信号最强的内容往往不是教程，而是 **Gotchas**（坑点）。因为教程 Claude 本来就知道很多，但坑只有你们团队真的踩过。

真正值钱的是告诉它哪里别踩。理想情况下，应该随着时间不断更新 Skill，把新踩的坑也加进去。

## 结构层：可复用和��僵化

### 第四条：使用文件系统和渐进式披露
很多人把 Skill 写成一个巨大的 Markdown 文件，所有内容都塞在里面。但 **Skill 是文件夹，不是文件**。

可以放脚本、模板、数据文件，主文件里只写核心逻辑，然后告诉 Claude 详细文档在 `references.md`，需要的时候自己去读。按需加载才能最大化利用上下文窗口。

### 第五条：仔细考虑设置流程
有些 Skill 需要用户配置（如站会要发布到哪个 Slack 频道），这个信息每个团队都不一样。用 `config.json` 存储配置，首次运行时询问，后续直接使用。

这让 Skill 从**无状态工具变成有状态助手**——用户只需要配置一次，之后每次调用都能直接用。

### 第六条：避免过度约束
很多人会写"先运行 test，再运行 lint，最后运行 build"，但这样写太死了，Claude 没有灵活性。

更好的写法是："在部署前，确保代码通过测试、规范构建，并根据情况调整顺序。"

**约束目标，而不是约束路径。** 给 Claude 它需要的信息，但也要给它适应不同情况的灵活性。

## 高级技术层：有状态和可组合

### 第七条：内存与数据存储
Skill 每次运行都是全新的，不记得上次做了什么。Anthropic 的解决方案：让 Skill 写日志——每次运行后追加一条记录，下次运行时读取日志，只报告新的变化。

Skill 不再是**无状态的脚本，而是有记忆的助手**。

### 第八条：存储脚本并生成代码
别让 Claude 每次都从头写样板代码。把稳定能力封成脚本和辅助函数，让它负责组合，而不是重造轮子。

**给 Claude 最好的工具不是更多文档，而是可调用的代码。**

### 第九条：按需 Hooks
有些规则太严格，不想一直开启，但特定场景下又需要。Anthropic 有个 `careful` Skill 会阻止危险命令（如 `rm -rf`、`drop table`）。当你调用 `careful` 时，它会注册一个 hook，只在当前会话有效，会话结束后自动失效。

## 总结

回到开头的问题：**为什么你的 Skill 不被调用？为什么调用后效果差？**

- **不被调用** → 大概率是因为你把 description 写成了摘要，而不是触发时机
- **效果差** → 大概率是因为你写了一堆 Claude 已经知道的东西，或者步骤写得太死

这背后是同一个问题：**把 Skill 当成 Prompt 来写。**

Anthropic 的核心观点是：好的 Skill 不是告诉 Claude 每一步做什么，而是给它组合的能力，让它在执行时自己决定怎么做。
