## 核心观点

前面 12 期视频分别讲了 Skill 的类型、设计、评估、运营。但如果有人问"我想从 0 到 1 给我的团队搭建一套 Skill 系统，第一步该做什么？"——这个问题值得单独讲一期。

这期是把前面分散的内容整合成一个可执行的框架，从 0 到 1 分四个阶段推进。

---

## 第一阶段：认知与定位

**目标**：搞清楚 Skill 系统要解决什么问题，以及现在团队处于什么阶段。

### 第一步：做团队能力盘点
把团队现有的工作流画出来，标记出：
- **高频重复**：每天/每周都在做的事（站会汇报、周报、代码审查）
- **认知负担高**：需要资深经验才能做的事（故障排查、架构评审）
- **交接成本高**：换人做容易出问题的事（部署流程、权限管理）

### 第二步：识别 Skill 化的优先级
不是所有工作都适合 Skill 化。优先选：
1. **规则明确**：输入输出清楚，判断标准客观
2. **频率高**：值得投入自动化
3. **影响大**：做对了能显著提升效率

### 第三步：确定 Skill 类型
根据识别出来的场景，确定要写什么类型的 Skill（触发式、工具式、流程式、偏好式）。

---

## 第二阶段：最小可行 Skill

**目标**：用最低成本跑通第一个 Skill，验证流程。

### 第一步：选一个最简单的场景
不要选最复杂的。选一个频率高、规则明确、影响可控的场景作为第一个 Skill。

### 第二步：写草稿
- 确定 name 和 description（触发条件）
- 写核心流程（约束目标，不是约束路径）
- 加 gotchas（踩过的坑）
- 写清楚输入输出

### 第三步：建最小测试集
3~5 条 prompt，覆盖：
- 1 条显式调用
- 1 条隐式调用
- 1 条带上下文调用
- 1 条负控样本

### 第四步：跑起来
让 Agent 用这个 Skill 完成真实任务。观察：
- 触发是否正确？
- 执行是否稳定？
- 输出是否符合预期？

### 第五步：迭代
根据真实反馈改 Skill。改完再跑测试集，确认旧问题被修掉。

---

## 第三阶段：扩展与结构化

**目标**：��单个 Skill 扩展到 Skill 系统，建立结构。

### 第一步：建立 Skill 目录结构
```
skills/
├── README.md          # 总览
├── code-review/       # 代码审查 Skill
│   ├── SKILL.md       # 主文件
│   ├── references.md  # 参考文档
│   └── templates/     # 模板
├── deploy/            # 部署 Skill
│   ├── SKILL.md
│   ├── config.json    # 配置
│   └── scripts/       # 脚本
└── ...
```

### 第二步：建立命名规范
- Skill 名称要语义清晰
- description 要写触发条件
- 避免名称冲突

### 第三步：建立配置管理
- 用 config.json 管理团队特定的配置
- 首次运行时询问，后续直接使用

### 第四步：建立依赖管理
- Skill 之间可以互相调用
- 在 SKILL.md 里声明依赖
- 保持轻量，不要过度工程化

---

## 第四阶段：运营与持续优化

**目标**：让 Skill 系统持续运转，而不是建完就荒废。

### 第一步：建立使用度量
- 记录 Skill 的调用次数、成功率、失败原因
- 定期 Review 使用数据

### 第二步：建立反馈回路
- 用户反馈渠道（好用的、不好用的、缺什么的）
- 定期收集和整理

### 第三步：建立维护节奏
- 偏好式 Skill：每季度更新一次
- 工具式 Skill：接口变化时更新
- 流程式 Skill：流程变化时更新
- 触发式 Skill：触发条件变化时更新

### 第四步：建立淘汰机制
- 定期检查哪些 Skill 没人用了
- 模型升级后，检查哪些 Skill 不再需要了
- 该删的删，该合并的合并

---

## 四个阶段的节奏建议

| 阶段 | 时间 | 产出 |
|------|------|------|
| **认知与定位** | 1~2 天 | 优先级列表 |
| **最小可行 Skill** | 3~5 天 | 第一个可用的 Skill |
| **扩展与结构化** | 2~4 周 | 3~5 个核心 Skill |
| **运营与持续优化** | 持续 | 健康的 Skill 生态 |

---

## 常见失败模式

### 失败模式一：一上来就想建大系统
症状：花两周设计目录结构、命名规范、配置管理，结果一个 Skill 都没写出来。

**解法**：先写一个能用的 Skill，再考虑结构化。

### 失败模式二：一次选最难的场景
症状：第一个 Skill 选了全团队最复杂的流程，写了三周还没跑通。

**解法**：从最简单的场景开始，先跑通再扩展。

### 失败模式三：写了 Skill 但不测试
症状：Skill 写完了，跑了几次感觉还行，就放那了。后来没人用了也不知道为什么。

**解法**：建最小测试集，每次改完跑回归。

### 失败模式四：没有淘汰机制
��状：Skill 越积越多，很多已经没人用了，但还留在系统里增加负担。

**解法**：定期清理，该删就删。

---

## 总结

从 0 到 1 搭建 Skill 系统，核心原则只有一条：**先跑通，再优化，再扩展。**

不要追求一步到位。先让一个 Skill 真正在团队里用起来，再慢慢加。Skill 系统的价值不在于数量多，而在于每个 Skill 都在真实地解决团队的问题。

**一个好的 Skill 系统不是设计出来的，是在使用中长出来的。**
