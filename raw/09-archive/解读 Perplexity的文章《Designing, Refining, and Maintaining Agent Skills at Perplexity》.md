# 解读 Perplexity的文章《Designing, Refining, and Maintaining Agent Skills at Perplexity》

## 一、开篇

Perplexity 的 AI Agent产品，公司内部搭建了一套严格管理的Skill资源库。里面有：

- **通用工具** — 比如操控电脑的基础功能

- **专业领域能力** — 金融、法律、医疗这些垂直行业的本事

- **各种小众功能** — 满足用户稀奇古怪的需求

有些功能平时根本用不上，但关键时刻不能掉链子。为了稳定给用户好的使用体验，Perplexity团队对待Skill质量的标准，和写正式代码的标准同等严格。

这份文档是Perplexity内部工程师统一开发、审核Skill的标准手册，现在对外公开。

## 二、到底什么是Skill？

很多人误以为Skill只是一段代码，其实完全不是。很多资深工程师写代码很厉害，但第一次写Skill总会被打回修改，因为因为许多适用于编写代码的实用模式，在创建技能时反而变成了反模式。

Skill本质是给大模型和运行环境搭建专属上下文，它的限制条件、设计逻辑和普通软件完全不一样，拿写代码的思路做Skill一定会失败。

在Perplexity的标准里，一个合格的Skill同时具备四层属性：

### 1\. Skill 是一整个文件夹，不只是单个文件

不是只靠SKILL\.md一个文件，完整Skill由多个文件配套组成，文件夹名就是Skill名称，内部标准目录：

- SKILL\.md：头部配置\+核心指令

- scripts/：AI会直接运行的代码，不用AI自己临时编写

- references/：大容量参考文档，只有需要时才加载

- assets/：模板、格式规范、固定数据

- config\.json：用户第一次使用时的初始化配置

这种中心\+分支的文件夹结构，能让Skill功能聚焦、条理清晰，复杂Skill还能做多级嵌套目录。

文章举了一个报税相关Skill，里面包含上千条税法条款，如果全部堆在同一层文件夹，AI识别效果还不如不加载这个Skill；分成三层分级目录后，AI能逐层缩小检索范围，精准调取对应法条，任务处理能力大幅提升。

但多级分层也有代价：需要人工梳理完整信息架构，配套简易检索工具、快速指引，减少AI查找资料的成本。

### 2\. Skill 是一套标准格式

根目录的SKILL\.md有强制格式要求：

1. 必须配置`name`和`description`，文件夹名称和name完全一致；

2. `name`只能全小写、用横杠分隔，不能有空格；

3. `description`是路由触发规则，不是功能说明，核心作用是告诉AI「什么场景要加载这个Skill」，所以描述开头经常写“Load when”，而不是介绍这个Skill能干什么；

头部配置还支持两个参数：

- depends：用来配置Skill之间的依赖关系，实现分层加载；

- metadata：用于测评、代码审核记录；

不同AI Agent还能自定义头部字段，也可以把额外配置单独放进json/yaml文件，避免细碎配置占用模型上下文。



### 3\. Skill 采用分层加载机制，分三层消耗token

三层加载，每层占用的token成本、加载时机完全不同：

|层级|加载内容|token消耗|加载时机|
|---|---|---|---|
|索引层|所有Skill名称\+描述|单个Skill约100token|每一次用户对话全程常驻|
|加载层|SKILL\.md完整正文|上限5000token|触发加载Skill时载入，对话全程占用|
|运行层|scripts、参考文档、模板、子Skill|无固定上限|只有Agent主动读取对应文件时才加载|

1. 索引层：所有对话一开始就写入系统提示词，AI靠这一层判断要不要调用Skill。这里文字必须极度精简，每一个token都有持续成本，能放进索引的标准极高。

2. 加载层：触发Skill后载入正文，建议控制在5000token以内。一次对话经常加载3\-5个Skill，冗余文字会同时拖累所有Skill效果，没用的内容会白白占用上下文。

3. 运行层：各类附件文件，只有用到才加载，限制最宽松，适合存放大量分支、特殊规则。

## 三、什么时候需要新建一个Skill？什么时候不用？

### 需要做Skill的情况

1. 模型原生回答会出错；

2. 同一类任务，每次输出结果不稳定，需要统一标准；

3. 专属行业、企业内部信息，不在模型训练数据里；

4. 带有主观审美、固定判断标准（比如设计师规定页面字体、排版规范，模型原生无法自主掌握）。

### 不需要做Skill的情况

1. 内容是模型本身就掌握的基础知识（比如基础git命令，模型天生会，没必要做成Skill）；

2. 内容属于通用全局规则，绝大多数对话都要用，直接放进系统提示词，不用条件加载；

3. 配套外部接口、工具频繁更新，Skill内容会快速过时，容易让AI输出错误。

### 核心准则：每一段文字都有“token成本税”

写Skill里每一句话前都自问：去掉这句话，Agent会不会出错？

如果不会，就删掉。所有用户、所有对话都会持续承担上下文token开销，Skill越精简越好。

原文引用帕斯卡名言：文字写得长，只是因为没时间精简。

优质Skill打磨起来很费时间，随手快速生成的Skill效果都会很差。相关研究也证明：让大模型自己写Skill，平均不会带来任何效果提升，模型没办法自主写出能提升自身能力的流程规则。

## 四、完整搭建Skill五步流程

做Skill不能先写正文，必须按固定顺序操作：

### 步骤0：先写测评用例（Evals）

测评用例来源：真实用户提问、过往AI出错案例、边界混淆场景（容易和其他Skill弄混的需求）。

至少准备正向用例（应该触发该Skill）\+反向用例（不该触发），反向用例参考价值更高。

核心目标：确认对应场景下Skill能正常加载。

### 步骤1：打磨description描述（最难的一步）

`description`是路由开关，写错会出现两大问题：该调用不调用、不该调用乱调用，新增Skill还会干扰其他原有Skill。

合格描述标准：

1. 以Load when开头；

2. 控制在50词以内；

3. 描述用户真实需求、用户原话，不总结Skill功能；

反面例子：介绍这个Skill能做什么；

正面例子：写用户会说出的真实诉求，比如监控代码合并的Skill，描述写“用户担心CI流程报错、需要确保代码顺利合并”。

### 步骤2：撰写SKILL\.md正文

和写给人看的文档完全相反：

1. 省略所有AI天生掌握的基础操作，不用一步步罗列基础命令；

例：不用逐条写git操作命令，直接写“把提交记录合并到干净分支，自动处理冲突，无法合并时说明原因”；

2. 重点写踩坑特例（Gotchas），告诉AI哪些错误不能犯，每次发现AI出错就新增一条；

3. 大量、多分支、大容量内容，不要放在SKILL\.md正文，拆分到scripts/references/assets文件夹，实现按需加载。

### 步骤3：合理利用分层文件夹拆分内容

1. scripts：存放固定逻辑代码，不用AI每次重新编写；

2. references：大容量参考文档，设置触发条件，满足条件才读取；

3. assets：输出固定模板，AI直接复制填充；

4. config\.json：首次使用Skill时候，用户的自定义配置信息，从这里读取；

复杂业务可以做多级嵌套目录，也可以拆分成多个独立Skill，通过depends配置依赖关系。

### 步骤4：多轮迭代优化

1. 先在分支上反复测试，不加载Skill先跑一遍基准效果；

2. 扩充测评用例，完整跑所有测试；

3. 描述里微小的文字改动，都会大幅改变触发效果，全部调整完成再提交；

4. 一次性提交完整代码\+全套测评用例，零散分步修改会增加审核负担。

### 步骤5：正式上线发布

全部测试、迭代完成后上线。

## 五、Skill长期维护方法

### 核心机制：踩坑（Gotchas）清单正向扩充（Gotchas Flywheel）

Skill上线后，以新增内容为主，尽量不大量删除原有文字：

1. AI出现新错误 → 新增一条踩坑说明；

2. 不该触发却加载 → 收紧描述，新增反向测评用例；

3. 该触发不加载 → 补充关键词，新增正向测评用例；

4. 系统全局提示词更新 → 检查Skill是否出现规则冲突、重复内容；

修改`description`这种核心路由配置时，必须配套新增对应的测评用例，不能直接改完提交。靠持续补充踩坑特例，慢慢把准确率做到极高水平。

### 多套测评套件持续校验

Perplexity会长期运行多类自动化测评：

1. 加载路由测评：检验Skill触发精准度、召回率，防止新Skill干扰原有功能；

2. 分层加载测评：确认AI用到对应场景时，会读取配套参考文件；

3. 端到端完整任务测评：完整跑一遍AI工作流程，用大模型裁判打分；

4. 多模型兼容测评：在GPT、Claude模型上分别测试，不同模型对Skill的识别逻辑差异很大。

## 七、全文总结

1. 先写测评用例再做Skill，一定要准备反向混淆场景用例；

2. description是最难写的部分，格式统一以Load when开头，每一个词都会影响AI识别；

3. 踩坑特例（Gotchas）价值最高，初期内容精简，后续随出错案例逐步扩充；

4. **新增Skill可能无意干扰线上已有的其他Skill，改动存在连锁影响；**

5. 开发、维护Skill全程配套官方分层文件夹工具，官方仓库和公开社区有大量成熟示例可以参考；

6. Skill适合自动化重复性固定工作，能大幅节省人工时间，但制作门槛高，不是所有场景都需要。
