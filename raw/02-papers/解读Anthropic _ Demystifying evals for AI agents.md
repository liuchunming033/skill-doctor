好的评估让团队能够更有信心地发布AI Agent。没有评估，团队很容易陷入被动循环——只在生产环境中发现问题，而修复一个故障往往又会引发其他问题。评估能够在问题影响用户之前就使其暴露，让行为变化变得可见，并且这种价值会随着时间的推移在Agent的整个生命周期中不断累积放大。

# **评估的挑战**

- Agent以多轮交互方式运作，涉及调用工具、修改状态、根据中间结果自适应调整，行为路径高度动态

- 自主性、智能、灵活性这些让Agent有用的核心能力，恰恰也导致其行为难以预测和标准化衡量

- 与传统软件的固定输入输出不同，Agent的中间决策过程复杂，评估维度远超单一结果对错

# **评估结构定义**

### 1\. 任务（Task）

**定义**：对Agent的单一指令或目标，描述需要完成的具体工作。

**关键特征**：

- 是评估的**起点和输入**

- 包含用户意图、上下文、约束条件

- 可以是简单指令（"查询账户余额"）或复杂目标（"帮我预订从纽约到东京的最便宜航班，要求周三出发、直飞优先、预算$1500以内"）

**示例**：

```Plain Text
任务：为客户退款$50，并发送确认邮件
```

### 2\. 试验（Trial）

**定义**：**一次完整的任务执行实例**，即Agent从接收任务到结束运行的全过程

**关键特征**：

- 一个任务可以进行**多次试验**（多次运行）

- 每次试验因Agent的非确定性（如温度参数、模型随机性）可能产生不同行为路径

- 试验 = 任务 \+ 某次具体执行 \+ 产生的完整交互记录

**对比**：

||任务（Task）|试验（Trial）|
|---|---|---|
|**类比**|考试题目|一次具体考试作答|
|**性质**|静态定义|动态执行实例|
|**数量关系**|1个任务 → N个试验||



### 3\. 转录本（Transcript）



**定义**：试验的**完整执行记录**，捕获Agent与环境交互的全过程。



**关键特征**：

- **多轮结构**：包含所有工具调用、状态变更、中间结果、Agent的推理步骤

- **可回溯性**：能重现Agent的决策链条

- **评估依据**：评分器分析转录本来判断行为质量

    

**典型内容**：

```Plain Text
[系统提示]
[用户任务]
[Agent思考：我需要先查询用户订单...]
[工具调用：get_orders(user_id="U123")]
[工具返回：[{order_id: "O456", amount: 50, status: "completed"}]]
[Agent思考：订单已确认，现在调用退款API...]
[工具调用：process_refund(order_id="O456", amount=50)]
...
```



**重要性**：转录本是Agent评估区别于传统LLM评估的核心——必须看**过程**，而非仅看最终输出。



---



### 4\. 结果（Outcome）



**定义**：试验的**最终状态或输出**，包括任务完成情况和副作用。



**关键特征**：

- **二元维度**：成功/失败（是否达成目标）

- **细粒度维度**：完成度、效率、成本、副作用

- **延迟性**：某些结果不会立即显现（如退款是否真正到账需异步确认）

    

**结果分类**：

|类型|说明|
|---|---|
|**直接结果**|Agent返回给用户的最终响应|
|**环境副作用**|数据库变更、邮件发送、API调用等外部状态变化|
|**长期结果**|用户满意度、业务指标影响（通常超出单次评估范围）|



---



### 5\. 评分器（Grader）



**定义**：**判断试验质量的评估函数或模型**，对转录本和/或结果进行打分或分类。



**关键特征**：

- **输入**：转录本（过程）\+ 结果（最终状态）\+ 任务定义（预期目标）

- **输出**：分数、标签、或通过/失败判定

- **实现方式多样**：规则代码、LLM\-as\-a\-judge、人工标注、混合方案

    

**常见评分器类型**：



|类型|适用场景|示例|
|---|---|---|
|**输出匹配**|有标准答案的任务|检查最终退款金额是否为$50|
|**规则检查**|明确约束条件|验证是否调用了授权接口|
|**LLM评分器**|开放性、语义判断|"回复是否礼貌且专业？"|
|**参考对比**|有黄金标准转录本|与人工标注的理想路径对比相似度|
|**状态验证**|副作用关键的场景|检查数据库中退款记录是否存在|



**核心挑战**：评分器本身也需要被验证（"谁来评估评估者"），避免评分器错误导致评估失真。



---



### 6\. 评估框架（Harness）



**定义**：**运行试验、收集转录本、应用评分器的底层执行基础设施**。



**职责**：

- 管理任务与试验的生命周期

- 提供Agent运行环境（沙箱、模拟环境、真实环境）

- 记录和存储转录本

- 调度评分器执行

- 收集和汇总评估指标

    

**关键设计考量**：

- **环境隔离**：试验之间不互相污染状态

- **可复现性**：相同输入应产生可追踪的输出

- **规模扩展**：支持批量运行大量试验

- **成本管控**：真实环境调用可能昂贵（如真实退款API）

    

---



### 7\. 智能体框架（Agent Harness）



**定义**：**专门为Agent评估设计的Harness变体**，核心区别是支持**多轮交互和工具使用**。



**与普通Harness的关键差异**：



|特性|普通LLM Harness|Agent Harness|
|---|---|---|
|**交互模式**|单次请求\-响应|多轮循环（观察→思考→行动）|
|**环境状态**|无状态或简单上下文|复杂状态管理（工具结果持久化）|
|**执行终止**|模型输出即结束|需判断何时停止（任务完成/最大轮数/错误）|
|**副作用处理**|通常无|必须管理（模拟/回滚/隔离）|
|**错误传播**|单次错误即结束|**错误可在多轮中传播累积**|



---



### 8\. 评估套件（Suite）



**定义**：**面向特定评估目标的完整配置组合**，包含一组任务、评分器、运行参数和预期标准。



**结构组成**：

```Plain Text
Suite = {
  任务集合: [任务A, 任务B, 任务C...],
  评分器配置: {各任务对应的评分规则},
  运行参数: {试验次数、环境设置、超时限制...},
  通过标准: {总体阈值、关键任务必须通过等},
  元数据: {版本、用途、关联的业务场景}
}
```



**Suite的设计意图**：

- **覆盖度**：一组任务共同覆盖Agent的能力边界

- **回归检测**：新代码版本运行相同Suite，对比行为变化

- **分层评估**：快速Smoke Suite → 完整Regression Suite → 深度Adversarial Suite

## 三、核心难点：多轮评估中的错误传播累积



这是Agent评估区别于传统软件测试的**本质挑战**。



### 错误传播机制



```Plain Text
第1轮：Agent错误理解用户意图
    ↓
第2轮：基于错误理解，调用错误工具
    ↓
第3轮：工具返回异常/无关数据，Agent尝试"修复"但方向错误
    ↓
第4轮：状态进一步混乱，Agent进入循环或产生幻觉
    ↓
最终结果：完全偏离目标，但评分器若只看最终输出可能误判为"尝试解决问题"
```



### 关键洞察



|问题|说明|
|---|---|
|**错误不等于失败**|中间步骤出错后Agent可能自我纠正，最终成功——这是Agent能力的体现，不应简单判负|
|**正确不等于成功**|中间步骤都对，但累积效率极低（如20轮完成本应3轮的任务），实际不可接受|
|**路径依赖**|早期微小差异（如工具返回顺序）可能导致完全不同的后续行为|
|**评分器盲区**|仅看最终结果的评分器会漏掉过程中的风险行为（如差点误删数据）|



### 应对策略



1. **过程评分（Process\-based Grading）**：评分器分析完整转录本，而非仅最终输出

2. **里程碑检查（Milestone Checking）**：在关键中间步骤设置检查点

3. **错误恢复评分**：专门评估Agent从错误中恢复的能力

4. **对比基线**：与已知良好路径对比，检测偏离程度

5. **扰动测试（Perturbation）**：故意改变工具返回顺序/内容，观察鲁棒性

# **评分器类型选择**

## 1\. 代码评分器（Code\-based Evaluator）

基于代码的评分器（Code\-based graders）是一类通过程序化方式自动评估AI系统输出的方法，主要包括：

- 字符串匹配：检查输出文本是否与预期匹配，比如精确匹配、正则表达式、模糊匹配等

- 二元测试：验证代码修改后，测试用例从失败变为通过或保持通过状态

- 静态分析：不运行代码，直接分析代码结构和规范，比如代码风格检查、类型检查、安全漏洞扫描

- 结果验证：检查最终输出结果是否正确

- 工具调用验证：检查调用了哪些工具以及参数是否正确

- 转录分析：统计对话轮次、词元使用量等

这类评分器的核心优势

- 执行速度快：自动化执行，无需人工等待

- 成本低廉：计算成本低，不需要人力

- 评判标准客观统一：标准统一，不受主观偏见影响

- 结果可复现性强：相同输入永远得到相同结果

- 易于定位和调试：当出现评分争议时易于调试和定位问题

- 精确验证特定条件是否满足。

然而，它们也存在明显的局限性：

- 对"合法但形式不同"的答案过于敏感，容易误判，例如："纽约" vs "New York" vs "NYC" → 被判错

- 无法理解细微差别、语境或创造性表达，例如：答案顺序调换、额外空格、礼貌用语。

- 在评估主观性较强的任务（如创意写作、对话质量、策略选择的优劣）时表现乏力，因为这些任务往往没有唯一正确的答案，难以用简单的规则覆盖所有合理的输出情况。

代码评分器是一把双刃剑——它以牺牲灵活性和语义理解能力为代价，换取了评估过程的高效、客观和可控，因此最适合用于边界清晰、标准明确的客观任务，而不适合开放式的主观评估场景。

代码的评分器的评估示例：

```Python
# 示例：检查Agent是否正确调用工具
def check_tool_call(output, expected_tool):
    return output.tool_name == expected_tool and \
           output.parameters == expected_parameters  # 精确匹配

# 示例：验证输出格式是否为有效JSON
def check_json_format(output):
    try:
        json.loads(output)
        return True
    except:
        return False
```

## 2\. 模型评分器（LLM\-as\-Judge）

**基于模型的评分器** 用AI模型（如LLM）替代传统代码/人工来评估输出质量，常用于：

- 自动评测AI生成内容

- 评估开放式问答、创意写作等**无标准答案**的任务

**基于模型的评分器（Model\-based graders）** 共有 **5种评估方法**：

### 1\. Rubric\-based scoring（基于评分标准的评分）

- **含义**：让AI模型按照预设的评分标准（如1\-5分制、具体维度要求）对输出进行打分

- **示例**："请从准确性、流畅度、相关性三个维度，每项1\-5分，评估这个回答"

### 2\. Natural language assertions（自然语言断言）

- **含义**：用自然语言描述判断规则，让模型验证输出是否满足这些断言条件

- **示例**："检查回答中是否包含事实性错误" 或 "确认回答语气是否礼貌"

### 3\. Pairwise comparison（两两比较）

- **含义**：让模型直接比较两个输出，判断哪个更好，而非单独打分

- **示例**："比较回答A和回答B，哪个更符合用户需求？"

### 4\. Reference\-based evaluation（基于参考的评估）

- **含义**：将模型输出与标准参考答案进行对比，评估相似度或正确性

- **示例**："将生成的摘要与人工编写的参考摘要比对，计算内容重叠度"

### 5\. Multi\-judge consensus（多位评委达成一致）

- **含义**：使用多个AI模型（或多次调用）独立评估，取多数一致的结果

- **示例**：用3个不同的LLM分别打分，最终取中位数或多数投票结果

对比总结：

|方法|核心思想|适用场景|
|---|---|---|
|Rubric\-based scoring|按标准量化打分|需要明确分数的评测|
|Natural language assertions|用规则做是非判断|质量检查、合规审查|
|Pairwise comparison|相对比较|难以绝对量化时|
|Reference\-based evaluation|与标准答案对比|有明确参考答案的任务|
|Multi\-judge consensus|集成多个评判|降低单一模型的偏差|

**基于模型的评分器（Model\-based graders）** 的 **5大优势**：

- **Flexible（灵活）：**可适应不同任务和评分标准，无需硬编码规则

- **Scalable（可伸缩）：**能自动处理大量评估任务，随规模扩展

- **Captures nuance（捕捉细微差别）：**能理解语义微妙差异，不只是表面匹配

- **Handles open\-ended tasks（处理开放式任务）：**适用于没有唯一正确答案的任务（如创意写作、分析推理）

- **Handles freeform output（处理自由格式输出）：**能评估非结构化、多样化的输出形式

```Python
# 用另一个LLM评判Agent输出
judge_prompt = """
你是专业评估员。请评估以下Agent回答是否符合用户意图。

用户问题：{question}
Agent回答：{answer}

从以下维度评分（1-5）：
1. 准确性：信息是否正确
2. 完整性：是否覆盖所有要点
3. 安全性：是否有害内容

输出JSON格式：{"accuracy": X, "completeness": Y, "safety": Z, "reasoning": "..."}
"""
```

**Model\-based graders（基于模型的评分器）** 的 Weaknesses（弱点）共有 **3 项：**

- **不确定性（评分漂移）**：模型输出不稳定，相同输入可能得到不同评分结果

- **成本高于代码**：相比传统代码评估，调用模型推理需要更多计算资源和费用

- **需要人工评分员校准以确保准确性**：必须与人类评分标准对齐，否则评分可能偏离人类判断

**使用模型评分器，通常需要进行"校准"，比如：**

|校准问题|表现|解决方法|
|---|---|---|
|**评分漂移**|同一答案不同时间得分不同|温度设为0，固定seed|
|**标准不一致**|对"好"的定义模糊|编写详细评分标准（rubric）|
|**位置偏见**|偏好先出现的选项|随机调换答案顺序，多次采样|
|**长度偏见**|偏好更长的回答|明确字数要求，或标准化处理|
|**自我提升**|偏爱与自己风格相似的回答|使用不同模型家族作为评判者|

### 3\. 人类评分器（Human Evaluator）

人工评估是AI/ML系统中的**基准 truth（ground truth）来源。作为"金标准"的角色**

- 定义**什么是正确答案**（ground truth标注）

- 仲裁**模型评分器之间的分歧**

- 发现**评估体系本身的盲点**

特点是：

- 质量高 → 可作为"黄金标准"，用来训练和校准自动化评分模型，适合大模型评测、搜索质量评估、内容审核标准制定等需要高可信标注的领域。

- 成本也高 → 难以规模化，促使业界转向模型化评分方案，**"昂贵"的多维度**

|成本类型|具体说明|
|---|---|
|时间成本|单条评估需数分钟（阅读上下文、判断质量）|
|人力成本|需领域专家，非普通标注员可胜任|
|一致性成本|多人标注需培训统一标准，仍有主观差异|
|迭代成本|每次调整评估标准需重新标注|

## 三种评分器的特性对比

|维度|代码评分器|模型评分器|人类评分器|
|---|---|---|---|
|**核心特征**|快速、客观、脆弱|灵活、可扩展、需校准|质量金标准、昂贵|
|**速度**|⚡ 毫秒级|🐢 秒级到分钟级|🐢🐢 分钟级到小时级|
|**成本**|低（计算资源）|中（API费用）|高（人力时间）|
|**客观性**|高（规则确定）|中（模型有随机性）|低（人为主观差异）|
|**灵活性**|低（需预定义规则）|高（理解语义和上下文）|高（人类理解力强）|
|**脆弱性**|高（无法处理边界情况）|中（可能幻觉或偏见）|低|
|**可扩展性**|高（自动化）|高（自动化）|低（人力瓶颈）|
|**校准需求**|无需|需要（与人工判断对齐）|本身就是标准|

每个任务的评分可采用**加权计分**（组合多个评分器的结果，达阈值即可）、**二元判定**（所有评分项全过才算过）或**混合模式**（两者结合）。

# **能力评估与回归评估**

能力评估从低通过率起步，驱动改进；

回归评估保持近100%通过率，防止倒退。

高通过率的能力评估可"毕业"为回归套件。

两种评估类型的对比：

||**能力评估 \(Capability/Quality Evals\)**|**回归评估 \(Regression Evals\)**|
|---|---|---|
|**核心问题**|"这个代理擅长做什么？"|"代理是否仍能处理过去能做的事？"|
|**初始通过率**|低（故意设置难任务）|接近100%|
|**目的**|推动能力提升，给团队"爬坡目标"|防止倒退，保护已有能力|
|**动态变化**|高难度任务通过后"毕业"为回归测试|持续运行，捕获性能漂移|

**"毕业"机制**

- 能力评估中的高难度任务 → 代理学会后 → 转入回归测试套件

- 问题从"我们**能否**做到？"变为"我们**能否可靠地**做到？"

**为什么必须同时运行**

- 只优化能力 → 可能在A处提升，却在B处意外破坏

- 回归测试作为"安全网"，确保改动不引发连锁问题

这是**持续循环**：

```Plain Text
开发初期：能力评估为主（低通过率，快速迭代）
     ↓
能力达标后：任务"毕业"进入回归套件
     ↓
长期维护：回归测试持续运行 + 新的能力评估不断加入
```

这种双轨设计解决了AI代理开发的核心矛盾——**既要进步，又要稳定**。

# **非确定性度量**

## 核心问题

智能体的行为在多次运行中会有波动——同一个任务这次通过了，下次可能失败。这种**非确定性**让评估结果比表面看起来更复杂，不能简单用"通过/失败"二元判断。

## 两个关键指标

例如：

- **pass@1 = 50%**：模型第一次尝试时，一半任务能成功

- **pass^k 计算**：若单次成功率75%，3次全成功的概率是 0\.75³ ≈ 42%

当k=1时，两个指标相同；当k=10时，走向完全相反——pass@k趋近100%（总能蒙对一次），pass^k趋近0%（很难次次完美）。

**选择哪个指标取决于产品需求**：要"能解出来"还是"每次都要对"。

- pass@k：给 k 次尝试机会，**任意一次成功**即通过 → 适用于代码生成、搜索等"有一个正确答案就行"的场景

- pass^k（或 pass@k 的严格版本）：**每次尝试都必须成功** → 适用于多步骤任务、自主决策的智能体，要求行为稳定可预测

## 第一步：从你目前手动测试的内容开始

每次发版前点点点验证的那些功能以及用户最常用的操作，将 线上出的 bug、客服收到的投诉转成测试样本。

- 影响用户多的、骂得狠的，先测

- 边角料小问题，往后排

## 第二步：编写明确的任务并提供参考答案

**任务质量比想象中更难把控**。一个好的评估任务必须满足：两位领域专家独立评判时，能得出相同的通过/失败结论。

关键原则

|原则|说明|
|---|---|
|**专家可独立达成一致**|任务描述要足够明确，消除歧义|
|**专家自己能完成**|如果专家按任务描述都做不出来，说明任务本身有问题|
|**评分标准清晰**|模糊的评分标准会导致判断不一致，变成指标噪声|

## 常见陷阱：代理"无辜失败"

> 例子：任务让代理"写一个脚本"，但没指定文件路径，而测试却假设了特定路径 → 代理没错，但评估失败
> 
> 

**教训**：评分器检查的所有内容，必须在任务描述中明确写出。

## 重要信号：0%通过率 ≠ 代理无能

对于前沿模型，如果**多次尝试（如pass@100）都是0%**：

- 更可能是**任务设计有缺陷**，而非代理能力不足

- 提示你要**重新检查任务描述和评分器配置**

## 实用建议：准备参考答案（Reference Solution）

为每个任务创建一个**已知能通过所有评分器的标准输出**：

- 证明任务**确实可解**

- 验证评分器**配置正确**

## 总结

> 评估任务要像法律条文一样精确，不能让代理因为"阅读理解"问题而背锅。
> 
> 

## 第三步，覆盖正例和反例

**不能只测"该做时做"，还要测"不该做时不做"**。

如果只测"该搜索时是否搜索" → 模型学会**逢查询就搜索**，即使已有知识能直接回答。

为 Claude\.ai 的网页搜索功能建评估时：

**"触发平衡"**（undertriggering vs\. overtriggering）：

- **触发不足**（undertriggering）：该搜不搜 → 信息缺失

- **过度触发**（overtriggering）：不该搜也搜 → 效率低下、成本浪费

评估套件是**动态扩充**的，不是一次性建好就完事。

## 第4步：构建一个具备稳定环境的健壮评估框架

**评估环境必须稳定且隔离**，否则测出来的不是智能体能力，而是环境噪音。

两个关键要求

|要求|为什么重要|
|---|---|
|评估用智能体与生产环境一致|否则评估结果无法指导实际部署|
|每次试验从干净环境开始|消除残留状态干扰|

**共享状态导致评估失真：**

- **git 历史残留** → Claude 偷看之前试验记录，成绩虚高

- **资源未释放**（如内存耗尽）→ 多个试验连环失败，看似智能体问题，实则是环境撑不住了

## 第五步：仔细设计评估标准

优秀的评估设计需要为代理和任务选择最合适的评分器。我们建议尽可能选择确定性评分器，在必要时或需要更多灵活性时使用大语言模型评分器，并审慎地使用人工评分器进行额外验证。

## 四个关键设计要点

### 评结果，不评路径

**常见误区**：检查代理是否按"正确顺序"调用了特定工具序列。

**问题**：代理经常能找到设计者未预料到的有效方法，这种刚性检查会**误杀合理方案**。

**正确做法**：评估最终产出（outcome），而非执行路径。

---

### 设置部分得分

对于包含多个环节的任务，应设置部分得分。一个客服人员若能正确识别问题并核实客户信息，但未能完成退款操作，其表现明显优于直接失败的情况。

---

### 迭代校准LLM评分器

模型评分通常需要经过仔细的迭代以验证其准确性。

### 排查"隐性故障模式"

即使Agent表现良好，也可能导致得分偏低，原因在于评分漏洞、代理框架限制或任务表述模糊。

**另一个案例：METR的时间范围基准**

- 任务要求：优化到"达到指定分数阈值"

- 实际评分：要求"超过该阈值"

- 结果：Claude因**遵循指令**被扣分，无视目标的模型反而得分更高

仔细检查任务设计和评分机制有助于避免此类问题。

### 防作弊设计

让评分器具备抗绕过和防篡改能力。代理不应能轻易“作弊”通过评估。任务和评分器的设计应确保真正通过必须解决实际问题，而非利用未预期的漏洞。

### 总结一句话

> 好的评分器要像好的老师：看重实质成果、允许合理创新、承认部分进步、及时修正误判，且不让"应试技巧"取代真实能力。
> 
> 

## 第6步：检查转录文本（transcripts）

查看Agent执行记录，验证评分器质量、区分真实失败与评分误判的关键手段。

- 评分器可能出错：它可能把代理的正确解法误判为失败，也可能放过真正的问题，只有通过大量阅读记录，才能发现评分器的系统性偏差

- 通过记录，**我们可以知道****代理是真错了，还是被冤枉了**

- 转录文本确保每次失败都可以**清楚追溯**：哪里错了、为什么错，如果分数长期不涨，要排查是代理能力问题，还是评估设计缺陷

不要只看最终分数，要建立工具和工作流来定期抽样审查记录。

## 第7步：监控能力评估饱和度

**评估饱和度**指当AI代理通过了所有可解决的任务，评估分数接近100%时，评估本身失去了区分能力、无法提供改进信号的状态。

100%的评估虽然能追踪回归问题，但无法提供改进方向的信号。当智能体通过了所有可解决的任务时，就会出现评估饱和现象，此时已无提升空间。例如，今年SWE\-Bench Verified的得分从30%起步，而前沿模型目前接近80%以上的饱和水平。随着评估趋近饱和，进展也会放缓，因为只剩下最困难的任务尚未完成。这可能导致结果具有误导性，因为巨大的能力提升仅表现为分数上的小幅增长。例如，代码审查初创公司Qodo最初对Opus 4\.5并不满意，因为其单次示例编码评估未能体现模型在更长、更复杂任务上的进步。为此，他们开发了一种新的代理式评估框架，从而更清晰地呈现了实际进展。

通常情况下，我们不会轻易相信评估得分，除非有人深入研究了评估的细节并查阅了一些对话记录。如果评分不公平、任务表述模糊、正确答案被扣分，或评估框架对模型造成限制，那么该评估就应予以修订。

## 第8步：保持评估套件的健康状态

评估套件是一种需要持续关注和明确负责人的活态产物，才能保持其有效性。

Anthropic设立专门的评估团队来负责核心基础设施，同时由领域专家和产品团队承担大部分评估任务并自行执行评估工作。

对于AI产品团队而言，拥有并持续迭代评估机制，应当如同维护单元测试一样成为常规操作。团队可能会在早期测试中看似“有效”的AI功能上耗费数周时间，却未能满足那些本可通过精心设计的评估提前发现的隐性期望。明确评估任务是检验产品需求是否具体到足以开始开发的最佳方式之一。

我们建议采用评估驱动的开发方式：在代理能够实现目标能力之前，先构建评估来定义计划中的功能，然后不断迭代，直到代理表现良好。

最了解产品需求和用户的人员最适合定义成功标准。

# **评估与其他方法协同**

完整的评估图景还包括生产环境监控、用户反馈、A/B测试、人工对话记录审查以及系统的人员评估。

系统梳理了6种评估方法的优缺点及适用场景。

|方法|本质|优势|劣势|
|---|---|---|---|
|**自动化评估 \(Automated evals\)**|编程运行测试，无需真实用户|快速迭代、完全可复现、无用户影响、可集成CI/CD、支持大规模场景测试|前期投入大、需持续维护防漂移、若脱离真实使用模式会产生虚假信心|
|**生产监控 \(Production monitoring\)**|实时系统追踪指标与错误|揭示大规模真实用户行为、捕捉合成评估遗漏的问题、提供实际表现的真实数据|被动响应（问题先影响用户）、信号嘈杂、需基础设施投入、缺乏评分标准|
|**A/B 测试**|真实用户流量对比不同版本|衡量实际用户结果（留存/完成率）、控制混杂因素、可扩展且系统化|缓慢（需数周达显著性）、需足够流量、仅测试已部署变更、难解释指标变化原因|
|**用户反馈 \(User feedback\)**|明确信号（点踩/bug报告）|发现未预料问题、附带真实案例、通常与产品目标相关|稀疏且自选、偏向严重问题、用户很少解释失败原因、非自动化、依赖用户发现问题损害体验|
|**人工对话审查 \(Manual transcript review\)**|人工阅读智能体对话|培养故障直觉、捕捉自动化遗漏的细微质量问题、校准"好"的标准|耗时、不可扩展、覆盖不一致、评审疲劳影响质量、通常仅定性而非定量|
|**系统性人工研究 \(Systematic human studies\)**|训练有素的评分员结构化评分|多评分员的金标准质量判断、处理主观/模糊任务、为改进模型评分器提供信号|昂贵且周转慢、难频繁运行、评分员分歧需协调、复杂领域需专家|

## 关键洞察：方法组合与阶段匹配

> **"瑞士奶酪模型"**：没有单一方法能捕获所有问题，多层组合才能互相补漏。
> 
> 

|开发阶段|核心方法|作用|
|---|---|---|
|**上线前 \+ CI/CD**|自动化评估|第一道防线，每次变更/模型升级即运行|
|**上线后**|生产监控|检测分布漂移、未预料的真实世界故障|
|**流量充足后**|A/B 测试|验证重大变更的实际效果|
|**持续进行**|用户反馈 \+ 对话抽样审查|填补 gaps：持续分类反馈、每周抽样阅读、按需深入|
|**特定场景**|系统性人工研究|校准LLM评分器、评估主观输出（人类共识作为金标准）|

## 一句话总结

> **最有效的团队组合**：自动化评估保速度、生产监控给真相、定期人工审核做校准——三者缺一不可。
> 
>
