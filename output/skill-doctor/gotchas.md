# Gotchas —— Skill Doctor的踩坑记录

## 坑点1：description写成功能说明

**现象**：description写成"这是一个PDF处理的Skill"，而不是触发条件

**问题**：Agent不知道什么时候触发，Skill被埋没

**诊断方法**：
- 检查是否包含"这是一个"、"该Skill用于"等表述
- 检查是否缺少用户会说的关键词
- 检查长度是否超过100字符

**正确示例**：
```
当用户说处理PDF、填写PDF表单、或提取PDF内容时触发。用于填写填充式表单。
```

**错误示例**：
```
这是一个帮助用户处理PDF文件的Skill，可以填写表单、提取文本、转换格式等功能。
```

---

## 坑点2：gotchas.md是空壳

**现象**：创建了gotchas.md，但里面只有"暂无坑点"或几个字的说明

**问题**：最有价值的内容缺失，Skill实用性差

**诊断方法**：
- 检查gotchas.md文件大小是否<500字节
- 检查是否只有"暂无"、"TODO"等占位符
- 检查是否缺少"现象-问题-解决"结构

**正确做法**：
- 记录团队踩过的真实坑点
- 每个"坑点"包含：现象、问题、解决
- 随时间持续扩充

---

## 坑点3：单文件堆砌所有内容

**现象**：SKILL.md超过500行，所有内容都塞在一个文件里

**问题**：违反渐进式披露原则，context过载

**诊断方法**：
- 检查SKILL.md行数是否>300
- 检查是否有references目录或附属文件
- 检查是否有[链接](xxx.md)引用

**正确做法**：
- 主文件控制骨架，<200行
- 详细内容拆分到附属文件
- 使用`[详细说明](examples.md)`引用

---

## 坑点4：过度约束执行路径

**现象**：写成"第一步运行命令A，第二步运行命令B，第三步..."

**问题**：给Agent的死板路径，失去灵活性

**诊断方法**：
- 检查是否出现"第一步"、"第二步"、"Step 1"等表述
- 检查是否缺少决策点（"判断"、"选择"、"根据"）
- 检查是否只有流程没有目标

**正确做法**：
- 约束目标，不约束路径
- 给Agent足够的自由度
- 描述决策框架，不是执行步骤

---

## 坑点5：重复Agent已知知识

**现象**：Skill里写"什么是git"、"git commit的作用"、"如何写commit message"

**问题**：浪费context，Agent本身就知道这些

**诊断方法**：
- 检查是否出现"什么是"、"介绍"、"教程"等词汇
- 检查是否讲基础知识，而不是团队特有知识
- 检查是否缺少"团队规范"、"公司特有"等内容

**正确做法**：
- 补充Agent不知道的（团队规范、公司特有流程）
- 不教Agent已知知识（基础概念、通用方法）
- Gotchas最有价值

---

## 坑点6：缺少config.json持久化

**现象**：Skill每次都要重新配置，没有config.json

**问题**：无法记住用户偏好，每次重新询问

**诊断方法**：
- 检查是否有config.json文件
- 检查config.json是否为空或只有默认值
- 检查Skill是否需要用户配置但缺少配置文件

**正确做法**：
- 创建config.json存储团队偏好
- 包含：max_length、default_language、team_rules等
- Skill运行时读取配置，不是每次询问

---

## 坑点7：需要脚本但缺失

**现象**：Skill需要执行稳定能力（如PDF提取），但没有脚本

**问题**：每次让Agent重新写代码，不稳定且重复

**诊断方法**：
- 检查Skill是否涉及稳定能力（数据转换、文件处理）
- 检查是否有scripts目录
- 检查脚本是否完整（有注释、有错误处理）

**正确做法**：
- 封装稳定能力为脚本
- 脚本放在scripts/目录
- Skill中调用脚本，不是让Agent写代码

---

## 坑点8：Hooks配置错误

**现象**：PreToolUse hook配置格式错误，或matcher写错

**问题**：hook不生效，监控失败

**诊断方法**：
- 检查hooks是否在SKILL.md frontmatter或settings.json
- 检查matcher是否正确（"Bash"、"Skill"）
- 检查command路径是否正确

**正确格式**：
```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Skill",
      "hooks": [{"type": "command", "command": "~/.claude/hooks/log-skill.sh"}]
    }]
  }
}
```

## 坑点9：Markdown格式不规范

**现象**：YAML frontmatter缺失、表格格式错误、代码块没有语言标注

**问题**：影响可读性，甚至导致解析错误

**诊断方法**：
- 检查是否有YAML frontmatter（name、description）
- 检查表格是否对齐
- 检查代码块是否标注语言（```python）

**正确做法**：
- 必须有YAML frontmatter
- 表格列对齐
- 代码块标注语言类型

---

## 坑点10：缺少下节预告或参考资料

**现象**：课程类Skill没有"下节预告"，或缺少参考资料

**问题**：用户不知道下一步学什么，缺少延伸阅读

**诊断方法**：
- 检查是否有"下节预告"或"参考资料"
- 检查参考资料是否使用[[wikilink]]格式
- 检查是否有后续学习路径

**正确做法**：
- 课程类Skill必须有下节预告
- 所有Skill应有参考资料
- 使用[[wikilink]]链接相关概念
