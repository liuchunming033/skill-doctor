# Agent 评估实战课 —— 从"靠感觉"到"靠证据"

> **课程定位**：面向 AI Agent 开发者和团队的工程实战课。学完之后，你能从"跑通一次就觉得没问题"的体感开发，升级到"用数据证明 Agent 能做什么、不能做什么"的工程化开发。
>
> **总时长**：~150 分钟（按每章 4-5 分钟估算）
>
> **前置知识**：了解 Agent 基本概念（Prompt / Tool / Memory），有至少一次 Agent 开发经验。

---

## 课程结构总览

```
模块一：破冰 —— 为什么你需要评估（3 章）

```

# 模块一：破冰 —— 为什么你需要评估

## 第 1 章：你的 Agent 开发是"瞎"的吗？

| 项目 | 内容 |
|------|------|
| **核心问题** | 没有评估的 Agent 开发为什么是在"蒙眼开车"？ |
| **核心内容** | 三个真实翻车场景：（1）改了 Prompt 自以为更好，上线后差评翻倍；（2）Agent 在 10 次测试中 4 次给出完全不同的答案；（3）修复了 Bug A，引发了 Bug B。引出核心论点：Agent 开发已从"代码逻辑"变为"概率行为"——你改的不是代码，你改的是概率分布。 |
| **开场白** | "你刚调了一版 Prompt，感觉 Agent 回答顺眼多了。你把新版本推上线，心满意足地下班。第二天早上打开电脑——用户投诉翻了三倍。发生了什么事？" |
| **时长** | 4 分钟 |
| **参考资料** | [[AgentEvaluation]] § 为什么 Agent 评估比传统 LLM 评估更难？ |

## 第 2 章：Agent 评估为什么比传统软件测试难？

| 项目 | 内容 |
|------|------|
| **核心问题** | 传统软件测试的方法论为什么不能直接套用到 Agent 上？ |
| **核心内容** | Agent 三个独特挑战：（1）**非确定性**——同一 prompt 多次执行结果不同，"跑通一次"不代表"稳定能跑"；（2）**黑盒化**——模型升级/Prompt 微调/工具链变化都可能导致行为漂移，肉眼难以察觉；（3）**错误级联放大**——一次任务涉及几十步工具调用，前序步骤的小偏差沿链路逐级放大。对比传统软件：固定输入→固定输出 vs 相同输入→不同输出。 |
| **开场白** | "传统软件测试有个美丽的前提：同样的输入，永远得到同样的输出。Agent 的世界里没有这个前提。" |
| **时长** | 4 分钟 |
| **参考资料** | [[AgentEvaluation]] § Agent 评估 vs 传统软件测试；[[AgentEffect]] § 与传统软件测试的本质区别 |

## 第 3 章：评估的核心结构 —— 一张图看懂 8 个概念

| 项目 | 内容 |
|------|------|
| **核心问题** | 聊 Agent 评估之前，先统一语言：任务、试验、转录本、评分器、结果到底是什么？ |
| **核心内容** | 核心公式：**Eval = Agent 输入 → 执行 → 捕获 Trace + 产物 → 一组检查规则 → 可对比的分数**。8 个概念的定义与关系：Task（考试题目）、Trial（一次作答）、Transcript（答题过程记录）、Outcome（最终答案）、Grader（阅卷老师）、Harness（考场）、Agent Harness（Agent 专用考场）、Suite（完整考试方案）。一张关系图理清链路。 |
| **开场白** | "你去看 Anthropic 的评估文章，会被 8 个概念砸晕。但其实它们之间的关系特别简单——就像一场考试。" |
| **时长** | 5 分钟 |
| **参考资料** | [[AgentEvaluation]] § 评估的核心结构 |

# 模块二：指标 —— 怎么量化"好"

## 第 4 章：客观指标 vs 主观指标

| 项目 | 内容 |
|------|------|
| **核心问题** | 评估 Agent 时，哪些指标能用代码算？哪些必须靠人判断？ |
| **核心内容** | 客观指标：BLEU/ROUGE/METEOR/BERTScore、Accuracy/F1/EM——成熟度高、稳定性好、但无法捕捉微妙质量差异。主观指标：满意度/可信度/回答完整性——任务相关性强、但评估成本高。两者各自的适用场景和局限性。核心观点：**学术优先客观，企业必须结合主观，最佳实践是两者组合**。 |
| **开场白** | "BLEU 分数 0.85，Agent 表现一定好吗？我见过 BLEU 0.9 的翻译把'How are you'翻成'怎么是你'。" |
| **时长** | 4 分钟 |
| **参考资料** | [[MetricsDesign]] § 客观指标 vs 主观指标 |

## 第 5 章：指标设计三原则 —— 可量化、可自动评测、可归因

| 项目 | 内容 |
|------|------|
| **核心问题** | 设计出来的指标怎么保证能落地？不是写出来好看的？ |
| **核心内容** | 三原则递进逻辑：（1）**可量化**——把抽象概念转为可计算的数值，否则"为什么好、好多少"无法回答；（2）**可自动评测**——支持自动化才能融入 CI/CD 持续跑，否则每次发版前人工测一轮成本爆炸；（3）**可归因**——指标下降时能定位到哪个模块出了问题（是意图理解？还是检索？还是生成？），否则指标只是报警但无法指导修复。三原则关系：**可量化 → 可自动评测 → 可归因**。 |
| **开场白** | "很多团队设计了一堆指标，看起来很专业。但三个问题暴露真相：能量化吗？能自动跑吗？出了问题知道该修哪吗？" |
| **时长** | 5 分钟 |
| **参考资料** | [[MetricsDesign]] § 指标设计三原则 |

## 第 6 章：五大评估维度 —— 功能、过程、效率、鲁棒、体验

| 项目 | 内容 |
|------|------|
| **核心问题** | 评估 Agent 应该从哪几个维度入手？优先级怎么排？ |
| **核心内容** | 腾讯 TEG 实践总结的五大维度：**(P0) 功能正确性**（做对了吗？）、**(P1) 过程质量**（过程合理吗？中间步骤有风险行为吗？）、**(P1) 效率与成本**（划算吗？20 轮才完成本该 3 轮的任务不可接受）、**(P0) 鲁棒性与安全**（靠谱吗？换个问法还灵吗？有越狱风险吗？）、**(P2) 体验与对齐**（好用吗？语气得当吗？）。为什么"过程质量"在 Agent 评估中如此特殊——只看最终结果会漏掉"差点误删数据"这种风险行为。 |
| **开场白** | "评估和考试不一样。考试只看最终答案，Agent 评估还得看你是光明正大做出来的，还是偷看了隔壁的答案。" |
| **时长** | 5 分钟 |
| **参考资料** | [[AgentEvaluation]] § 五大评估维度 |

# 模块三：评分器 —— 谁来打分

## 第 7 章：代码评分器 —— 快、便宜、但脆弱

| 项目 | 内容 |
|------|------|
| **核心问题** | 什么场景下可以直接用代码判断 Agent 表现？代码评分的边界在哪？ |
| **核心内容** | 三种确定性评测方法：（1）**Rule-based**——nDCG/MRR/Accuracy/Precision/Recall/F1，适用于排序质量、分类匹配等有标准答案的任务；（2）**语义相似度**——BERTScore/Cosine Similarity，适用于问答/摘要/改写；（3）**二进制测试**——编译通过？文件存在？工具调用正确？Token 不超标？优点：毫秒级、零人力、完全可复现。缺点：对"合法但形式不同"的变体脆弱。案例：Agent 答案"纽约"被判定错误因为预期是"New York"。 |
| **开场白** | "代码评分器就像一个严格的会计——你多写了一个空格它都觉得你错了。但有时候，这种严格恰恰是你最需要的。" |
| **时长** | 5 分钟 |
| **参考资料** | [[GraderTypes]] § 确定性评分器；§ 评测方法细分 |

## 第 8 章：模型评分器（LLM-as-Judge）—— 灵活但需要校准

| 项目 | 内容 |
|------|------|
| **核心问题** | 当代码搞不定时，怎么用 LLM 当裁判？LLM 裁判可靠吗？ |
| **核心内容** | 五种评估方法：（1）Rubric-based scoring（按评分标准打分）；（2）Natural language assertions（自然语言断言）；（3）Pairwise comparison（成对比较）；（4）Reference-based evaluation（参考答案评估）；（5）Multi-judge consensus（多评委共识）。三种评测模式：单模型评分、多模型投票、成对比较。优势：灵活、可扩展、捕捉细微差别、处理开放式输出。劣势：非确定性、比代码评分贵、需要与人工校准。 |
| **开场白** | "让一个 LLM 当裁判去评判另一个 LLM 的回答——这听起来像让一个学生去批改另一个学生的试卷。能行吗？" |
| **时长** | 5 分钟 |
| **参考资料** | [[GraderTypes]] § 模型评分器；§ LLM-as-Judge 的三种评测模式 |

## 第 9 章：人工评分器 —— 黄金标准但稀缺

| 项目 | 内容 |
|------|------|
| **核心问题** | 什么时候必须用人来打分？怎么用最少的人力撬动最大的评估价值？ |
| **核心内容** | 三种用途：（1）**建立 Ground Truth**——定义"什么是正确答案"；（2）**仲裁分歧**——模型评分器之间出现矛盾时的人工裁决；（3）**发现盲点**——发现评估体系本身的缺陷。两种组织方式：专家评测（质量高但成本高规模有限，适用于法律/医疗等专业场景）vs 众包评测（成本较低可扩展，但需设计明确评分标准）。核心原则：**人类是稀缺资源，只用于建立标准和处理真正模糊的案例，不做常规重复劳动。** |
| **开场白** | "人工评估是黄金标准。但黄金很贵——你的评估预算里，只有 5-10% 的样本配得上它。" |
| **时长** | 4 分钟 |
| **参考资料** | [[GraderTypes]] § 人工评分器；§ 人工评测的组织方式 |

## 第 10 章：评分器选择决策树 + 三层流水线

| 项目 | 内容 |
|------|------|
| **核心问题** | 面对具体评估任务，怎么在三类评分器之间做选择？怎么把它们串成流水线？ |
| **核心内容** | 决策树：能用代码精确判断？→ 代码评分器。不能？→ 能定义清晰评分标准？→ LLM-as-Judge。不能？→ 人工评分器。三层流水线架构：**Layer 1 代码过滤层**（100%样本，零成本，格式/工具调用/精确匹配 → 失败直接淘汰）→ **Layer 2 模型质量层**（100%样本，成本可控，语义质量多维度评估，低置信度触发人工）→ **Layer 3 人工仲裁层**（5-10%样本，边界案例+定期抽检校准+初始标注）。执行口诀：**"能写规则的不写 prompt，能用 prompt 的不用人。"** |
| **开场白** | "如果考试有 100 道题，你让老师批改 100 道？不——你用机器批 90 道选择题，剩下的 10 道论述题才请老师出马。Agent 评估也一样。" |
| **时长** | 5 分钟 |
| **参考资料** | [[GraderTypes]] § 三层评估流水线；§ 推荐组合策略 |

# 模块四：校准 —— 让 LLM 裁判可信

## 第 11 章：Trace 分析实操 —— 从转录本中提取评估信号

| 项目 | 内容 |
|------|------|
| **核心问题** | 评分器打出了 85 分，但你真的理解这 85 分代表什么吗？Trace 里藏着评分器看不到的信息。 |
| **核心内容** | 在进入校准之前，必须先学会读 Trace——因为校准 LLM 裁判的前提是你知道"什么是真正的好"。Trace 分析的三个层次：（1）**表面层**——工具调用序列是否正确？参数是否准确？（2）**效率层**——Agent 绕了多少弯路？本该 3 步完成的用了 20 步？有没有死循环？（3）**风险层**——Agent 中间有没有危险操作（如差点误删数据、查询了不该查的信息）即使最终结果是正确的？四个关键洞察：**错误 ≠ 失败**（中间出错但自我纠正成功 = Agent 能力的体现）、**正确 ≠ 成功**（中间都对但效率极低 = 实际不可接受）、**路径依赖**（早期微小差异导致完全不同的后续行为）、**评分器盲区**（仅看最终结果的评分器会漏掉过程中的风险行为）。实操演示：打开一段 JSONL Trace，逐行解读。 |
| **开场白** | "评分器告诉你 Agent 拿了 85 分。但你看一眼 Trace，发现 Agent 在第三步差点把数据库删了，第五步绕了 10 个圈子才找到答案——这 85 分的水分有多大？" |
| **时长** | 5 分钟 |
| **参考资料** | [[Trace]]；[[AgentEvaluation]] § 四个关键洞察 |

## 第 12 章：LLM-as-Judge 的五种偏差

| 项目 | 内容 |
|------|------|
| **核心问题** | LLM 裁判自带哪些偏见？怎么发现并消除它们？ |
| **核心内容** | 五种系统性偏差拆解：（1）**评分漂移**——同一答案不同时间得分不同（修：温度=0，固定seed）；（2）**标准不一致**——对"好"的定义模糊（修：编写详细rubric）；（3）**位置偏见**——偏好先出现的选项（修：随机调换顺序多次采样）；（4）**长度偏见**——偏好更长的回答（修：明确字数要求或标准化）；（5）**自我提升**——偏爱与自己风格相似的回答（修：使用不同模型家族作为评判者）。每种偏差配一个真实翻车案例。 |
| **开场白** | "LLM 裁判和人类裁判一样有偏见。但 LLM 的偏见更隐蔽——它不会告诉你'我不喜欢这种风格'，它直接给你打低分。" |
| **时长** | 5 分钟 |
| **参考资料** | [[GraderTypes]] § LLM-as-Judge 校准：五种偏差与对策 |

## 第 13 章：校准实操 —— 黄金测试集 + 相关性验证

| 项目 | 内容 |
|------|------|
| **核心问题** | 具体怎么校准一个 LLM 评分器？从零到可用的完整流程是什么？ |
| **核心内容** | 三步校准流程：（1）收集 100-500 条人工标注建立"黄金测试集"——需要领域专家，不只是随便标注；（2）计算模型评分与人工评分的 Spearman 相关性；（3）迭代优化评分标准直到相关性 > 0.8。关键细节：什么是一致率 ≥ 85%？为什么用 Spearman 而非 Pearson？校准不是一次性的——模型升级后重新校准、评分标准改了重新校准、每季度定期抽检。 |
| **开场白** | "校准不是一劳永逸的事情。你今天校准好的 LLM 裁判，下个模型版本可能就变了——就像你换了新眼镜，整个世界都歪了一度。" |
| **时长** | 5 分钟 |
| **参考资料** | [[GraderTypes]] § 校准流程 |

## 第 14 章：LLM 非确定性根源 —— Temperature、Top-P 与多模型对抗

| 项目 | 内容 |
|------|------|
| **核心问题** | 为什么同一个 LLM 同一条 Prompt 跑三次得到三个不同结果？怎么降低这种随机性？ |
| **核心内容** | 随机性机制拆解：自回归生成 → Temperature 缩放原始 logits → Softmax 转概率分布 → Top-P 筛选候选词集 → 随机采样。每一步随机差异通过链式法则指数级放大。不同 LLM 结果不同的三个根源：分词器差异（"unstable"→"un"+"stable" vs "unstable"）、训练参数差异（千亿参数从不同数据学到的计算模式）、RLHF 指令微调差异。**多模型对抗方案**：2-3 个不同 LLM 执行相同 Prompt，统计聚合（求均值/多数投票），代价是 2-3 倍耗时，适用于对时间不敏感的长时评测。 |
| **开场白** | "你写好了 Prompt，信心满满地点下运行。第一次：85 分。第二次：72 分。第三次：91 分。是 Prompt 写错了？还是 LLM 在逗你？答案比你想的更底层。" |
| **时长** | 5 分钟 |
| **参考资料** | [[GraderTypes]] § LLM 非确定性根源与多模型对抗 |

# 模块五：策略 —— 怎么组织评估

## 第 15 章：端到端评测 vs 分层评测

| 项目 | 内容 |
|------|------|
| **核心问题** | 应该测整个 Agent 系统还是拆开测每个组件？什么时候用哪种？ |
| **核心内容** | 端到端（E2E）：黑盒视角，模拟真实用户场景，关注"最终任务是否成功"→ 决定"**能否发布**"。分层评测：白盒视角，排除上下游干扰，按三种维度拆解 — （1）**架构逻辑**（意图理解层→信息检索层→逻辑规划层→内容包装层→UI交互层）、（2）**评估粒度**（系统效率→会话级别→节点级别）、（3）**业务维度**（业务场景→具体任务）。最佳实践闭环：E2E 找到问题 → 分层定位根因 → 专项迭代优化 → 二次 E2E 验证。一句话："E2E 告诉你该不该修，分层告诉你该修哪。" |
| **开场白** | "全链路测试告诉你车跑不快。但到底是发动机的问题还是轮胎的问题？你把整辆车拆了也找不到答案——除非你让修车师傅单独测每个部件。" |
| **时长** | 5 分钟 |
| **参考资料** | [[EvaluationStrategy]] § 端到端评测 vs 分层评测 |

## 第 16 章：线下评测 vs 线上评测

| 项目 | 内容 |
|------|------|
| **核心问题** | 为什么线下测得好好的，一上线就翻车？怎么堵上这个漏洞？ |
| **核心内容** | 线下评测 = "模拟考试"：依赖黄金数据集（Ground Truth），受控环境，目的是准入验证 + 回归保护 + 安全红队测试。线上评测 = "实战监测"：依赖生产日志和实时交互轨迹（OpenTelemetry 等标准链路采集），目的是监测预警 + 链路追溯 + 效果闭环（解决率/投诉率/转化率）。核心差异表：环境/数据来源/核心目的/典型手段。关键闭环：**线上发现的失败案例回流到线下扩充黄金数据集**，这是 Agent 效果持续提升的核心路径。 |
| **开场白** | "模拟考次次满分，高考却翻车。Agent 也一样——离线测试通过的 Agent，上线后面对真实用户的各种骚操作，分分钟崩溃。" |
| **时长** | 5 分钟 |
| **参考资料** | [[EvaluationStrategy]] § 线下评测 vs 线上评测 |

## 第 17 章：打分评估 vs 对比评估

| 项目 | 内容 |
|------|------|
| **核心问题** | 什么时候看绝对分数就够了？什么时候必须横向对比？ |
| **核心内容** | 打分评估 = "体检报告"：参照系是绝对标准、预设 KPI 或历史自我，输出通过/不通过或绝对分值，适用于准入验证（能发布吗？）。对比评估 = "运动员排名"：参照系是其他竞争对象（竞品/替代方案），输出排序（A>B>C）或 GSB（Good/Same/Bad），适用于选型决策（用 A 还是 B？）和迭代验证（比上一版改进了吗？）。关键区别：打分只需待评对象 + 阈值，对比必须多对象同条件数据。研发早期选型用对比，发布前质量把关用打分。 |
| **开场白** | "体检报告说你血压 120，你需要一个参照——是超标了（标准是 140），还是比上次高了（上次是 110）？两种参照，两种完全不同的行动。" |
| **时长** | 5 分钟 |
| **参考资料** | [[EvaluationStrategy]] § 打分评估 vs 对比评估 |

## 第 18 章：能力评估 vs 回归评估 —— "毕业"机制

| 项目 | 内容 |
|------|------|
| **核心问题** | 同一个测试跑了一百次终于过了，然后呢？怎么让它不退化？ |
| **核心内容** | 能力评估（Capability Eval）="目标山峰"：起步通过率 20-50%，回答"Agent 能把什么做好？"。回归评估（Regression Eval）="防御漂移"：应接近 100% 通过率，回答"原有能力是否还在？"。"**毕业**"机制：能力评估通过率稳定到 100% 后"毕业"为回归评估，进入 CI 每次代码变更都跑。举例：SWE-bench Verified 一年内从 40% 提升到 >80%，接近饱和 → 从能力评估变为回归评估。 |
| **开场白** | "能力评估就像驾照考试——反复练、反复考，直到通过。回归评估就像年检——每年确认你还是那个合格的司机。问题是：拿到驾照之后，你是越开越好，还是越开越野？" |
| **时长** | 5 分钟 |
| **参考资料** | [[CapabilityVsRegressionEval]]；[[AgentEvaluation]] § 能力评估 vs 回归评估 |

## 第 19 章：闭环策略 —— 评估→归因→优化

| 项目 | 内容 |
|------|------|
| **核心问题** | 评估跑完了，报告生成了，BadCase 列表出来了——然后呢？怎么把评估转化成真正的系统改进？ |
| **核心内容** | 闭环五步：（1）端到端评测获得整体基线 →（2）失败案例归因（按架构层级分类：逻辑规划错误 40% / 工具调用失败 30% / RAG 召回不准 20% / 输出包装问题 10%）→（3）对薄弱环节构建专项分层评测和黄金数据集 →（4）针对性优化（Prompt/工具/知识/模型）→（5）二次端到端验证确认局部改进转化为整体提升 → 持续闭环（线上新问题回流线下评测集）。核心思想：**评测不是终点，优化才是目的。没有闭环的评估只是一份体检报告，有闭环的评估才是持续迭代引擎。** |
| **开场白** | "大多数团队陷入了'评估沼泽'：每个月跑一轮评估，每月都出一份 BadCase 报告，每个月都没人改。评估变成了形式主义。" |

# 模块七：框架 —— 评估基础设施

## 第 22 章：评估框架 —— Agent Harness 与普通 Harness 的本质区别

| 项目 | 内容 |
|------|------|
| **核心问题** | 为什么不能拿普通 LLM 的评估框架直接套用到 Agent 上？ |
| **核心内容** | Agent Harness 五维对比普通 LLM Harness：（1）交互模式：单次请求-响应 vs 多轮循环（观察→思考→行动）；（2）环境状态：无状态 vs 复杂状态管理（工具结果持久化）；（3）执行终止：模型输出即结束 vs 需判断何时停止（任务完成/最大轮数/错误）；（4）副作用处理：通常无 vs 必须管理（模拟/回滚/隔离）；（5）错误传播：单次错误即结束 vs 错误可在多轮中传播累积。核心洞察：**评估时评估的是"框架 + 模型"的组合，而非单独模型**。同一模型在不同框架下表现可能差异巨大。 |
| **开场白** | "你以为换一个评估框架跑同样的 Agent，结果会差不多？大错特错。框架本身就是变量。" |
| **时长** | 4 分钟 |
| **参考资料** | [[EvalHarness]] § Agent Harness vs 普通 LLM Harness |

## 第 23 章：环境隔离与工程可靠性

| 项目 | 内容 |
|------|------|
| **核心问题** | 评估环境不干净会导致什么诡异问题？长时间运行的评估任务怎么保证可靠性？ |
| **核心内容** | 环境隔离三大陷阱：（1）**git 历史残留**——Agent 偷看之前试验的记录成绩虚高（Claude 内部真实案例）；（2）**资源未释放**——内存耗尽导致多个试验连环失败，看似 Agent 问题实则环境撑不住；（3）**文件残留**——上次试验的临时文件被本次 Agent 读取利用。工程可靠性设计：**Checkpoint 中断恢复**——每个阶段上报心跳，定时轮询识别中断，按阶段续跑而非从头开始；**并发限流**——QPM（控制线程并发数）+ TPM（输入+思考输出 Token 都计入，用 max_tokens 限制）。 |
| **开场白** | "Claude 的工程师曾经发现一个诡异的现象：某个 Agent 的评估成绩好得离谱。调查之后发现——Agent 在偷偷读之前试验留下的 git 历史。AI 学会作弊了。" |
| **时长** | 5 分钟 |
| **参考资料** | [[EvalHarness]] § 环境隔离；§ 长时间运行任务的可靠性 |

## 第 24 章：Trace 与可观测性 —— 不看过程只看结果就是瞎子

| 项目 | 内容 |
|------|------|
| **核心问题** | 如果 Agent 只在最后输出一个结果，你怎么知道中间发生了什么？怎么给过程打分？ |
| **核心内容** | Trace（执行轨迹）= Agent 结构化日志，包含：系统提示、用户任务、Agent 思考（CoT）、每次工具调用（名称+入参+返回值+时间戳）。输出格式要求：JSONL（每行一个 JSON 事件，可逐行解析）。Trace 是 Agent 评估区别于传统 LLM 评估的核心——**必须看过程，而非仅看最终输出**。如果 Agent 只输出最终回答而无可解析中间过程，则只能做结果评估——大量过程信息（如"差点误删数据"的风险行为）会丢失。过程评估的三种方式：里程碑检查、错误恢复评分、与已知良好路径对比检测偏离。 |
| **开场白** | "Agent 最后输出'已为您退款 50 元'。只看结果，完美。但是你看 Trace：它先删了用户的订单记录，又查了三次物流 API，最后才执行退款。这个过程你满意吗？" |
| **时长** | 5 分钟 |
| **参考资料** | [[Trace]]；[[AgentEvaluation]] § 错误传播累积机制 |

# 模块八：进阶 —— 上下文工程与深度思考

## 第 25 章：上下文工程（上）—— 四种长上下文失效模式

| 项目 | 内容 |
|------|------|
| **核心问题** | 给 LLM 裁判的 Prompt 总共大几万 Token——你以为信息越多越准确？结果 LLM 反而判断错误？ |
| **核心内容** | 上下文工程 = Prompt Engineering 的超集：不仅关注"怎么说"，更关注"给什么信息"。三个底层局限：注意力权重稀释（无关信息瓜分关键信息权重）+ KV 缓存 FIFO 淘汰（早期信息丢失）+ RoPE 位置编码模糊（超长文本中相近位置难以区分）。四种失效场景：（1）**上下文冲突**——近因效应（关注尾部）vs 首因效应（关注头部）的注意力竞争；（2）**上下文混淆**——张冠李戴，相似 ID 或实体被 LLM 搞混；（3）**上下文干扰**——无关信息通过注意力机制和激活值传播污染整个计算图；（4）**上下文丢失**——窗口开头信息被边缘化或淘汰。每种模式配真实案例。 |
| **开场白** | "LLM 号称支持百万 Token 窗口。你把 50 条服务记录、20 条规则、10 个范例全扔进 Prompt，信心满满地点运行——LLM 把所有 ID 都搞混了。窗口大 ≠ 脑子好用。" |
| **时长** | 5 分钟 |
| **参考资料** | [[ContextEngineering]] § 四种上下文失效场景 |

## 第 26 章：上下文工程（下）—— 五种应对策略

| 项目 | 内容 |
|------|------|
| **核心问题** | 怎么治理上下文，让 LLM 在长 Prompt 场景下保持推理准确性？ |
| **核心内容** | 五种策略：（1）**逻辑拆分**——将复杂任务从一次 LLM 调用拆为多次，每次只处理单一子任务（如十几种根因归类从一次混判改为每种独立判）；（2）**内容精简**——Prompt 控制在 5w token 以内，重点规则放头部（利用首因效应），去除冗余字段（ID/时间戳/状态等），长 ID 映射为短编号；（3）**强制约束**——明确输出格式（纯净 JSON、禁止代码块、禁止非标准格式）；（4）**Few-Shot**——添加示例作为"触发器"引导 LLM 激活推理模式，而非教它新知识；（5）**首因效应利用**——最重要规则放最前面，不是放最后。核心原则：**降低 Token 量 ≠ 降低信息量——去冗余、做压缩、拆逻辑。** |

## 第 27 章：深度思考模式 —— 什么时候该开启？

| 项目 | 内容 |
|------|------|
| **核心问题** | 深度思考模式能提升推理准确性，但也让响应慢 N 倍、成本高 N 倍。到底什么时候该开？ |

# 模块九：实战 —— 从零到一 + 案例 + 总结

## 第 28 章：Swiss Cheese Model —— 没有银弹，只有千层饼

| 项目 | 内容 |
|------|------|
| **核心问题** | 那么多评估方法，该用哪个？答案：全都要，但要懂怎么组合。 |
| **核心内容** | Swiss Cheese Model（瑞士奶酪模型）：每一层评估方法都有漏洞（像奶酪上的洞），但多层叠加就能互相补漏。五层组合：**自动化评估**（上线前 + CI/CD，第一道防线）→ **生产监控**（上线后，检测分布漂移和未预料的真实故障）→ **A/B 测试**（流量充足后，验证重大变更）→ **用户反馈 + 对话抽样**（持续进行，填补 gap，分类反馈、每周抽样阅读）→ **系统性人工研究**（特定场景，校准 LLM 评分器、评估主观输出）。最有效团队的组合公式：**自动化评估保速度 + 生产监控给真相 + 定期人工审核做校准。** |
| **开场白** | "世界上没有完美的评估方法，就像瑞士奶酪没有没洞的那一片。但你把五片叠在一起，光就穿不过去了。" |
| **时长** | 4 分钟 |
| **参考资料** | [[AgentEvaluation]] § Swiss Cheese Model |

## 第 29 章：pass@k 与 pass^k —— 非确定性场景下的统计指标

| 项目 | 内容 |
|------|------|
| **核心问题** | 同一个任务 Agent 执行 10 次只成功 7 次，这算通过还是没通过？ |
| **核心内容** | 两个互补指标：（1）**pass@k**——k 次试验中至少 1 次通过的概率 → 衡量"**峰值能力**"（Agent 能不能做好？）。k 越大，pass@k 越高，但边际递减。（2）**pass^k**——k 次试验全部通过的概率 → 衡量"**稳定性**"（Agent 能不能每次都做好？）。k 越大，pass^k 越低，暴露非确定性问题。实践指导：能力评估阶段关注 pass@k（探索上界），回归评估阶段关注 pass^k（守住下界）。举例：pass@10 = 95%，pass^10 = 30% → Agent 有能力做好，但极其不稳定，需要工程优化而非能力提升。 |
| **开场白** | "Agent 测评不是'做过一次就过了'。同一道题做 10 次，10 次都做对——这才是真正的过关。做 10 次只对 3 次——不是运气好叫能力强，是概率在帮你。" |
| **时长** | 5 分钟 |
| **参考资料** | [[passAtK]] |

## 第 30 章：实战案例 —— 阿里云"AI 评测 AI"智能客服运营 Agent

| 项目 | 内容 |
|------|------|
| **核心问题** | 前面讲的这么多理论，真实落地时是什么样子？会遇到哪些坑？ |
| **核心内容** | 阿里云智能客服团队案例拆解：（1）**目标**——用更大的 LLM + 深度思考模式，去评测较小的 LLM 客服 Agent；（2）**架构**——评估（四维度规则：语义相关性/表达质量/内容合规性/信息完整性）→ 诊断（十几种根因归类：粗召遗漏/精排问题/生成幻觉等）→ 优化（自动生成相似问 + 互联网检索补充知识）；（3）**工程挑战**——正向 vs 负向评判混合策略、两级业务品类/场景映射解决业务强依赖、多 LLM 对抗降低单一模型偏差、上下文工程（逻辑拆分 + 内容精简 + 强制约束 + Few-Shot）、Checkpoint 中断恢复、并发限流；（4）**实际效果**——BadCase 发现准确率 85%+，根因归类准确率 80%+。 |
| **开场白** | "理论讲了一堆，现在看看真刀真枪的实战。阿里云团队搭了一个'AI 评 AI'的运营系统——让 AI 当客服的质检员。这个系统踩过的坑，比我们上一节课学的还多。" |
| **时长** | 5 分钟 |
| **参考资料** | [[AutoEvalAgent]]；[[摘要-ai评测ai-智能客服运营agent]] |

## 第 31 章：从零到一 —— Anthropic 的 8 步评估建设路线图

| 项目 | 内容 |
|------|------|
| **核心问题** | 我现在就要开始搭 Agent 评估体系，第一步做什么？ |
| **核心内容** | Anthropic 8 步路线图详细拆解：**Step 0**：尽早开始——20-50 个简单任务即可起步，别等到"准备好"；**Step 1**：从手工测试转化——把团队正在做的手工测试固化为可复现任务；**Step 2**：写无歧义任务——附带参考解法，让不同评估者能得出一致结论；**Step 3**：构建平衡用例集——正例 + 负例，覆盖常见和边界情况；**Step 4**：构建稳健框架——环境隔离 + 确保每次试验从干净环境开始；**Step 5**：精心设计评分器——优先确定性，必要时 LLM-as-Judge；**Step 6**：读转录本！——验证评估是否在测量真正重要的东西，这是最容易被跳过的步骤；**Step 7**：监控能力评估饱和——通过率稳定 100% 后毕业为回归评估；**Step 8**：通过开放贡献和维护保持评估套件健康。 |
| **开场白** | "别等了。你不需要 1000 个任务才能开始。20 个真实场景的任务 + 一套评分规则 = 你的第一版评估套件。Anthropic 也是这么开始的。" |

## 第 32 章：课程总结 —— 从"靠感觉"到"靠证据"的能力跃迁

| 项目 | 内容 |
|------|------|
| **核心问题** | 学完这门课，你获得了什么能力？接下来该做什么？ |
| **核心内容** | 回顾课程核心脉络：**感觉驱动**（跑通一次就上线，出问题靠用户反馈）→ **指标驱动**（有量化标准，知道"好"是什么）→ **评分器驱动**（代码+LLM+人工三层组合，能自动化打分）→ **策略驱动**（端到端/分层/线下/线上/打分/对比/闭环，评估体系完整）→ **工程化驱动**（环境隔离/Checkpoint/限流/上下文工程/深度思考，评估基础设施可靠）。三个核心转变：（1）从"跑通一次就够了"到"稳定的统计证据"；（2）从"感觉变好了"到"可对比的分数"；（3）从"评估是验收环节"到"评估是迭代引擎"。行动建议：本周就选 20 个真实任务，写第一版评估套件。 |
| **开场白** | "32 章，150 分钟。如果只记住一句话，记住这个：Agent 开发的质量不取决于你写了多聪明的 Prompt，而取决于你有多了解它到底做对了什么、做错了什么。" |
| **时长** | 4 分钟 |
