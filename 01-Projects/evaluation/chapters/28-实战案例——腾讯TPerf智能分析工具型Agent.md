## 引言

阿里云的对话型 Agent 评估，最大的难点是"没有标准答案"——什么是好的回复，需要 LLM 裁判来判断。

腾讯 TPerf 遇到的问题恰恰相反。

TPerf 是腾讯云的性能压测平台，它的智能分析 Agent 任务是：**分析性能数据，找出瓶颈，给出优化建议**。

这类任务有客观标准：分析步骤对不对？参数提取准不准？报告结论是否基于数据？

答案是确定的——这是工具型 Agent 评估最大的优势，也是最大的陷阱：**因为标准客观，你反而会把要求提得很高，然后发现 Agent 达不到。**

TPerf 团队的经验是：先用扣分制暴露最薄弱的环节，再用 0 容忍策略守住关键底线。

---

## 💡 什么是工具型 Agent 评估的核心难题？

工具型 Agent 的评估重点和对话型不同：

| 维度 | 对话型（阿里云） | 工具型（TPerf） |
|------|-----------------|----------------|
| 核心问题 | 回答质量够不够好？（语义判断） | 步骤对不对？结果准不准？（逻辑判断） |
| 评分器 | LLM 裁判为主 | 代码规则为主 |
| 最大挑战 | 没有标准答案 | 如何对比复杂的步骤序列 |
| 稳定性要求 | 允许一定波动 | 关键决策 0 容忍 |

工具型 Agent 的评估难不在"怎么打分"，而在：

1. **步骤序列比对**：Agent 的执行路径是动态的，两次跑出来的步骤顺序可能不完全一样——哪些差异是可接受的，哪些是错误？
2. **关键决策的容忍度**：如果 Agent 给出了错误的性能优化建议，工程师按这个建议操作会出事——这类 Agent 的错误代价极高。

---

## 🔧 核心方法一：100 分扣分制

TPerf 用的不是加分制（答对一条加多少分），而是**从 100 分开始扣**。

```
总分 100 分
├── 步骤正确性：满分 10 分（LCS 比对，扣分）
│   ├── 工具调用序列是否与基线一致
│   └── 工具参数是否正确
├── 效率：满分 10 分（扣分）
│   ├── Token 消耗是否超基线 1.5 倍
│   └── 工具调用次数是否超基线
└── 报告质量：满分 80 分（扣分）
    ├── 数据准确性（分析数据是否与原始数据一致）
    ├── 分析深度（是否覆盖关键瓶颈）
    ├── 结构完整性（报告格式是否完整）
    └── 建议可执行性（优化建议是否具体可操作）
```

为什么用扣分制，不用加分制？

加分制天然有"及格就好"的倾向——Agent 只要做到基本步骤，就能积累到分数，很难暴露真正的薄弱点。

扣分制不同：**Agent 最薄弱的地方会被集中扣分，帕累托效应更明显**——哪个维度扣分最多，哪个维度就是优化优先级最高的。

## 🔧 核心方法二：LCS 序列比对（解决步骤评估难题）

工具型 Agent 的执行路径是一个序列：

```
基线路径：[查询 CPU 负载] → [查询内存使用] → [调用 DB 分析工具] → [生成报告]
本次执行：[查询内存使用] → [查询 CPU 负载] → [调用 DB 分析工具] → [生成报告]
```

这两个序列哪个更好？顺序不同，但内容一样。

暴力比对会说"路径不一致，扣分"——但实际上顺序不同只是实现细节，不影响最终分析结果。

TPerf 用的是 **LCS（最长公共子序列）算法**：

```python
# 伪代码：LCS 比对
baseline_steps = ["查询CPU", "查询内存", "DB分析", "生成报告"]
actual_steps   = ["查询内存", "查询CPU", "DB分析", "生成报告"]

lcs_length = LCS(baseline_steps, actual_steps)  # = 3 (查询CPU, DB分析, 生成报告)
step_score = (lcs_length / len(baseline_steps)) * 10  # = 7.5 分
```

LCS 比对的好处：**允许步骤顺序有合理变动，只检查关键步骤是否缺失或替换**。

实践中 TPerf 会设定哪些步骤是"强制步骤"（缺失即重大扣分），哪些是"可选步骤"（缺失小扣分）。

## 🔧 核心方法三：0 容忍稳定性策略

Ch25 讲了 pass@k 和 pass^k 两个指标——能力验证阶段用 pass@k，稳定性验证阶段用 pass^k。

TPerf 做了一个更严格的选择：**关键决策类 Agent，不是取平均分，不是多数投票，而是 0 容忍——任一 Trial 不达标，整个用例不通过。**

用公式表达：

```
通过条件 = (Trial 1 通过) AND (Trial 2 通过) AND (Trial 3 通过)
```

这等价于强制 pass^k = 1.0，即必须 k 次全部通过。

为什么这么严苛？

**因为性能分析 Agent 的建议是用来指导工程师操作的。** 一次错误建议，工程师按指令操作，可能造成生产事故。这类 Agent 不能"平均而言是对的"，它必须"每次都对"。

这个策略不适合所有 Agent。TPerf 自己给出了适用条件：
- 关键决策类 → 0 容忍（pass^k = 1.0）
- 辅助分析类 → 90% 通过率即可（pass^k ≥ 0.9）
- 探索型任务 → 至少 1 次通过（pass@k ≥ 1）

## 📋 四种执行模式

TPerf 的评估系统设计了四种模式，对应不同研发阶段的需求：

| 模式 | 适用场景 | 运行范围 | 耗时 |
|------|---------|---------|------|
| 快速分析 | 单次 Prompt 调试 | 单模型 × 单用例 × 1 次 Trial | 秒级 |
| 全量测评 | 发布前验证 | 全模型 × 全用例 × 多次 Trial | 3-5 小时 |
| 按模型筛选 | 模型升级验证 | 指定模型 × 全用例 × 多次 Trial | 30-60 分钟 |
| 重新评分 | 评分规则迭代 | Trace 已缓存，只重跑评分器 | 分钟级 |

**重新评分模式**是其中最精妙的设计——Trace 存档后不需要重跑 Agent，只需重跑评分器。

当你发现评分规则有问题（比如步骤比对太严格，顺序差异也被扣分了）时，直接修改评分规则，对历史 Trace 重新打分，就能知道这条规则的调整是否改善了结果。

这让**评分器的迭代速度**和 Agent 本身的迭代解耦了——不需要每次修改规则都重跑所有 Agent。

## 🎯 本节核心观点

**工具型 Agent 评估的优势：客观标准存在，代码规则为主，可信度高。**

**工具型 Agent 评估的难点：步骤序列比对、关键决策的容忍度设定。**

**TPerf 的三个核心方法：**
1. 扣分制 → 暴露最薄弱环节（帕累托效应更明显）
2. LCS 序列比对 → 允许步骤顺序灵活，但关键步骤不能缺
3. 0 容忍稳定性 → 关键决策类 Agent 一次失败即全部不通过

**四种执行模式的设计思路：把"跑 Agent"和"运行评分器"解耦——Trace 缓存后，评分规则可以独立迭代。**

**关键问题问自己：你的 Agent 是辅助类还是决策类？这决定你该用 pass@k 还是 pass^k=1.0。**

---

## 实操

**第一步**：借鉴 TPerf 的扣分制，为你的 Agent 设计一张 100 分评分表：
- 列出 3-5 个评估维度
- 为每个维度分配权重（合计 100 分）
- 标注每个维度的扣分规则（满足什么条件扣多少分）

**第二步**：你的 Agent 的执行是否有固定的步骤序列？如果有，标注哪些是"强制步骤"（缺失大扣分），哪些是"可选步骤"（缺失小扣分或不扣分）。

**第三步**：你的 Agent 属于什么类型？
- 如果是关键决策类 → 考虑 0 容忍策略（pass^k = 1.0）
- 如果是辅助分析类 → 设定通过率阈值（如 pass^k ≥ 0.9）

**第四步**：你的评估流程中，是否支持"重新评分"模式？Trace 是否有持久化存档？如果没有，这是你现在可以立刻改进的地方。
