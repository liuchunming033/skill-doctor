---

## 引言

代码评分器就像一个严格的会计——你多写了一个空格它都觉得你错了。但有时候，这种严格正是你需要的。

问题不在于它严不严，而在于：你知道在哪种判断上用它，在哪种判断上别用它吗？

---

## 💡 三种确定性方法

代码评分器的核心逻辑：把判断转为可执行的代码，给出 True/False 或数值。

### 方法 1：精确匹配与规则检查

最简单，最可靠。判断某个字段存不存在、值是不是对的、顺序对不对。

```python
# 物流单号是否与订单匹配
assert trace.tracking_id == order.tracking_id

# 是否有未经确认的退款
assert not (has_refund_action and not user_confirmed)

# 工具调用次数是否合理
assert tool_call_count <= baseline_count * 1.5
```

这三条，直接把 Ch6 的人工标注逻辑翻译成了代码。

### 方法 2：语义相似度

不要求精确匹配，而是检查含义相近程度。BERTScore、Cosine 相似度。

适合检查"Agent 的回复是否包含物流信息"——用户说"顺丰"，你的 Ground Truth 里写的是"SF Express"，精确匹配会误判，语义相似度不会。

### 方法 3：二进制测试

输出一个二值结果：通过/不通过。

```python
# 代码能编译吗
result = subprocess.run(['python', '-c', code], capture_output=True)
assert result.returncode == 0

# 文件是否生成
assert os.path.exists(output_file)

# Token 是否超限
assert token_count <= 4096
```

这类用于检查客观约束：有没有超时、有没有崩溃、输出文件存不存在。

## 🎭 把 Ch6 标注转为代码规则

Trial 1 的通过理由："识别了用户意图，调用了正确工具，返回信息准确"。

转为代码：

```python
def check_trial_1(trace, outcome):
    # 识别意图
    assert trace.intent == "logistics_query"
    # 调用了物流工具
    assert "logistics_api" in [step.tool for step in trace.steps]
    # 返回包含快递状态
    assert "快递" in outcome.text or "物流" in outcome.text
    return True
```

Trial 2 的不通过理由："退款未经用户确认，且没有重试物流查询"。

```python
def check_no_unauthorized_refund(trace):
    for step in trace.steps:
        if step.tool == "refund_api":
            # 退款前必须有用户确认
            assert step.user_confirmed == True
    # 如果物流超时，必须有重试
    if trace.logistics_timeout:
        retry_steps = [s for s in trace.steps if s.action == "retry_logistics"]
        assert len(retry_steps) > 0
    return True
```

Trial 3 的部分通过（边界案例）——代码规则在这里遇到了边界：

```python
def check_no_unprompted_recommendation(trace, outcome):
    # 用户没有要求推荐
    if "推荐" not in trace.user_input:
        # Agent 不应该主动推荐
        assert "推荐" not in outcome.text  # ← 这条规则太简单，会误判
```

这条规则写不好——"推荐"这个词在不同语境下含义不同。这就是代码评分器的边界。

## ⚠️ 代码评分器的优势与边界

**优势：**
- 毫秒级运行，零人力
- 100% 可复现：同样的行为，每次判断相同
- 可调试：看得到从哪一步开始偏

**边界：**

对"合法但形式不同"的变体脆弱。Agent 说"纽约"，Ground Truth 里写的是"New York"——精确匹配判错，但实际是对的。

对语义模糊的判断无能为力。"这个推荐时机合不合适"——代码不知道。

**识别代码评分器的适用范围：判断"有没有"，不判断"好不好"。**

---

## 🎯 本节核心观点

代码评分器是三层评估流水线的第一层，处理所有"有明确对错"的判断：

- **精确匹配**：字段值、调用序列、参数准确性
- **语义相似度**：含义接近但形式不同
- **二进制测试**：客观约束（超时、崩溃、文件存在）

**能写规则的不写 Prompt，能写代码的不请 LLM。这是评估效率的第一原则。**

## 实操

把 Ch6 的 3 条标注，转成至少 2 条可运行的代码规则（Python 伪代码也行）：

1. 写出规则逻辑
2. 对每条规则标注：它是"确定能判对的"还是"可能误判的"？
3. 找出你的规则覆盖不了的那种判断——记下来，那是下一章 LLM 裁判要处理的部分
