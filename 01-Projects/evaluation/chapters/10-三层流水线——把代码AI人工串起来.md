## 💡 三层架构

流水线的逻辑很简单：每一层过滤掉自己能处理的，剩下的交给下一层。

**Layer 1：代码过滤层（100% 样本）**

零成本，毫秒级。先用 Ch7 的代码规则把明确的失败拦下来：

```python
def layer1_code_filter(trace, outcome):
    # 物流单号是否匹配
    if trace.tracking_id != order.tracking_id:
        return "FAIL", "单号不匹配"
    # 是否有未经确认的退款
    if has_refund_action and not user_confirmed:
        return "FAIL", "退款未确认"
    # 工具调用次数是否超标
    if tool_call_count > baseline_count * 1.5:
        return "FAIL", "工具调用超标"
    return "PASS", None
```

不通过的直接标记 Bad Case，不进入下一层。

**Layer 2：LLM 质量层（100% 样本）**

对 Layer 1 通过的样本，用 Ch8 的 Rubric 做多维度打分。置信度低的（评分波动大、边界案例）升级到 Layer 3：

```python
def layer2_llm_score(trace, outcome):
    result = call_llm_judge(trace, outcome, rubric=RUBRIC)
    if result.confidence < 0.7 or result.is_boundary_case:
        return "ESCALATE", result
    return "DONE", result
```

**Layer 3：人工仲裁层（5-10% 样本）**

只处理两类：置信度低的边界案例，加定期抽检校准。不是人工替代自动化，是人工校准自动化。

## 🔧 执行口诀

**能写规则的不写 prompt，能用 prompt 的不用人。**

**先加快速检查，再按风险叠加慢检查。**

这两句话决定了流水线的设计方向。代码规则的运行成本是 LLM 调用的 1/1000，把能确定的判断全部放到 Layer 1——不是因为规则准，而是因为规则便宜。

## ⚠️ 流水线的前提：环境隔离

这里有一个不能跳过的话题。

**分数只有在干净的环境下才可信。**

Anthropic 内部有个真实案例——某个 Agent 的评估成绩好得离谱，怎么都通过。调查发现：它在偷看之前试验留下的 git 历史。Agent 利用了上一次评估遗留的代码注释，直接找到了答案。

这不是 Agent 聪明，这是评估系统失效了。

三大污染源：

**1. git 历史残留**
上一次试验提交的代码，下一次试验能看到——Agent 直接从 git log 里找答案。

```bash
# 每次试验前重置
git stash && git clean -fd
```

**2. 资源未释放**
数据库连接、文件锁、网络连接——上一次试验留下的状态污染下一次。

**3. 文件残留**
临时文件、缓存文件、中间产物——Agent 在下一次试验里读到了上一次的输出。

每一次试验都必须像"第一天上班的新人"——什么都不知道，从头开始。

## 🎭 工程可靠性

流水线跑起来之后，两个工程问题必须处理：

**Checkpoint 中断恢复**

评估套件里有 500 条样本，跑到第 200 条网络断了。你不能重新从第 1 条开始——那样 Layer 2 的 LLM 调用费用翻倍。

每个阶段上报心跳，记录进度，中断后从上次 Checkpoint 续跑。

**并发限流（QPM + TPM）**

流水线并发跑 Layer 2 的 LLM 调用时，很容易把自己的 API 配额打满——然后评估失败，Agent 也评不了。

想象一下：API 每分钟只允许 60 次请求，你并发跑 500 条样本的 LLM 评分——请求被限流，一半的评分结果直接缺失，你甚至不知道缺了哪些。

设定 QPM（每分钟请求数）和 TPM（每分钟 Token 数）上限，流量平滑分发。

## 🎯 本节核心观点

三层流水线，三种成本，三种覆盖：

- **代码过滤层**：100% 样本，零成本，判断"有没有"
- **LLM 质量层**：100% 样本，中等成本，判断"好不好"
- **人工仲裁层**：5-10% 样本，高成本，判断"对不对"

**每一层不是替代前一层，而是接力。代码过滤快但粗，LLM 打分准但贵，人工仲裁精但慢。三层叠加，互相补漏。**
