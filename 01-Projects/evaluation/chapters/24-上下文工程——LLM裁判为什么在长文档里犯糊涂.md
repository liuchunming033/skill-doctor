---

## 引言

LLM 号称支持百万 Token 窗口。你把 50 条记录、20 条规则、10 个范例全扔进 Prompt——LLM 把所有人的 ID 都搞混了。

窗口大 ≠ 脑子好用。

工程师在构建 LLM 裁判时，最常犯的错是"信息越多越准确"。你觉得给 LLM 更多上下文，它判断会更可靠。结果是——上下文一超过某个量，LLM 开始张冠李戴，把 A 的问题归到 B 头上，把首段的规则忘在脑后，把中段的例子和末段的数据搅在一起。

这不是模型的问题，是你的 Prompt 设计问题。

> **上一章教你给评估搭了路线图，这一章解决路线图上的第一个减速带**：你把所有信息一股脑塞给 LLM 裁判，它反而变傻了。瘦身不是为了省钱——是为了让裁判别犯糊涂。

---

## 💡 四种上下文失效场景

**失效 1：上下文混淆（张冠李戴）**

最常见，最难察觉。

你的评估 Prompt 里有 50 条客服对话记录，每条有独立的 session_id。LLM 在处理第 40 条时，偶尔会把第 38 条的错误归到第 40 条身上——因为两个 ID 长得相似（session_abc123 和 session_abc124），在 50 条的长上下文中"混"在一起了。

评分结果看起来"差不多"，但仔细核查你会发现：大量的错误归因不是 LLM 判断力不行，是它把账记错了。

**失效 2：上下文丢失（开头信息被边缘化）**

研究证明，LLM 对 Prompt 开头和结尾的注意力远高于中段。你在 Prompt 第一段写了最重要的评分规则，LLM 跑到第 30 条对话时，那条规则已经"淡出视野"了。

结果：越靠后的对话，越容易忽略前面的关键约束。

**失效 3：上下文冲突（近因 vs 首因）**

你的 Few-Shot 范例在 Prompt 中段，评分规则在 Prompt 开头。当范例和规则略有出入时，LLM 倾向于跟随"更近的那个"——Few-Shot 范例覆盖了开头的规则。

你以为 Few-Shot 在帮你，实际上 Few-Shot 在悄悄改写你的规则。

**失效 4：上下文干扰（无关信息污染判断）**

每条对话记录里带了 20 个 metadata 字段：创建时间、设备类型、用户 ID、标签……这些字段和"回复是否得体"完全无关，但它们占了大量 Token，把真正有用的信息稀释了。

LLM 的注意力是有限资源。无关信息越多，有用信息越难被关注到。

## 🔧 真实案例：从 72% 到 89%

一个评估 Prompt 的实际情况：

```
评估对象：50 条客服对话记录
每条记录包含：session_id、时间戳、用户ID、设备类型、
              完整对话内容、物流单号、API调用记录……
评分规则：20 条
Few-Shot 范例：10 个

总 Token 量：约 15,000 Token
评分准确率：72%（与人工评分对比）
```

具体失效表现：LLM 把 session_abc123 的物流查询超时错误归到了 session_abc124 上——两个 ID 相邻，内容相似，在长上下文中发生了混淆。

**瘦身方案三步走：**

**第一步：删冗余字段**

每条对话记录从 200 Token 压缩到 80 Token——删掉时间戳、设备类型、用户 ID 等与评分无关的字段。只保留：session_id、对话内容、相关 API 调用结果。

**第二步：长 ID 映射短编号**

```python
# 处理前
session_id: "session_abc123"
session_id: "session_abc124"

# 处理后
S1 → session_abc123
S2 → session_abc124
```

这一步消灭了"张冠李戴"的根源。

**第三步：拆分复合指令**

原来一次 LLM 调用评两件事："功能是否正确" + "回复是否得体"。拆成两次独立调用，每次只评一个维度。

**结果：评分准确率从 72% → 89%。**

## 🎭 五种应对策略

**策略 1：逻辑拆分**

把"评 N 条记录"拆成 N 次独立调用，每次只处理 1 条。代价是调用次数增加，优势是完全消除上下文混淆。

如果不能拆到 1 条，至少把复合任务拆开——功能评估一次调用，体验评估一次调用。

**策略 2：内容精简（控制在 5 万 Token 内）**

实践建议：单次 LLM 裁判调用控制在 5 万 Token 以内。每条记录只保留评分必要的字段，其余全删。

能用 ID 替代长字符串的，全部替代。

**策略 3：强制约束输出格式**

要求 LLM 裁判输出纯净 JSON，而不是自然语言。JSON 格式强制了 LLM 结构化思考，减少模糊归因：

```json
{
  "session_id": "S1",
  "score": 3,
  "reason": "未按规则重试超时请求"
}
```

**策略 4：Few-Shot 放对位置**

Few-Shot 范例要么放在 Prompt 最开头（利用首因效应），要么放在被评估内容的紧前方（利用近因效应）。

放在 Prompt 中段是最坏的选择——既不够近，也不够先。

**策略 5：利用首因效应**

最重要的评分规则放在 Prompt 第一段，不是最后一段。LLM 对第一段的记忆远比中段稳定。如果你有 3 条核心规则和 17 条辅助规则，把 3 条核心规则提到最前面，单独成段，加粗标注。

## ⚠️ 核心认知

降低 Token 量 ≠ 降低信息量。

删掉的是冗余字段、无关 metadata、过长的 ID 字符串。保留的是真正参与评分的内容。

绝大多数评估 Prompt 的问题不是信息不够，是信息太多——而且质量参差不齐，真正有用的淹没在噪音里。

**瘦身不是损失，是聚焦。聚焦才能让 LLM 把有限的注意力放在真正重要的地方。**

## 🎯 本节核心观点

**四种失效场景：**
- 上下文混淆：张冠李戴（相似 ID 在长上下文中混淆）
- 上下文丢失：开头规则被遗忘
- 上下文冲突：Few-Shot 覆盖了评分规则
- 上下文干扰：无关字段稀释注意力

**五种策略：**
1. 逻辑拆分（1 条 = 1 次调用）
2. 内容精简（控制 5 万 Token 内，删冗余字段）
3. 强制约束（纯净 JSON 输出）
4. Few-Shot 位置优化（首位 or 紧前方）
5. 首因效应（核心规则放第一段）

**黄金原则：窗口大 ≠ 脑子好用。评估 Prompt 的质量，不取决于塞了多少信息，而取决于去掉了多少干扰。**

## 实操

对你的 LLM 裁判 Prompt 做一次"瘦身体检"：

**第一步**：数一下当前评估 Prompt 的 Token 量。超过 5 万了吗？

**第二步**：检查每条记录里的字段——哪些字段和评分标准无关？删掉它们。计算瘦身后每条记录的 Token 量（目标：压缩 30% 以上）。

**第三步**：看你的 session_id 或其他 ID——是否超过 10 个字符？如果是，写一个映射表，把它们替换成 S1、S2、S3 这样的短编号。

**第四步**：回跑同一批评估任务，对比瘦身前后的评分一致性。
