## 🔍 前半部分：Bad Case 系统化反推 —— 从"这个 case 没过"到"这个能力短板"

### 为什么逐个修是死循环？

你有 50 条 Bad Case。逐条看：

- case 1："用户骂人时 Agent 还在推销新品"
- case 2："用户说'我要投诉'被当成'我要查询'"
- case 3："物流 API 超时后直接退款"
- case 4："多轮对话中忘了用户前面说的订单号"
- ……

看起来毫不相关。一条一条修？修到第 5 个你会发现前 3 个的修复互相冲突——让 Agent 对情绪敏感了，它却把正常的催促也当成投诉，通过率反而下降。

**单个 Bad Case 是症状，失败模式才是病因。**

---

### 反推三步法

**第一步：归类抽象——按失败模式分类，不是按表面现象**

把 50 条 Bad Case 摊在桌上，不看"这条说了什么"，看"这条为什么失败"：

| 失败模式 | 典型案例 | 占比 |
|---------|---------|------|
| 意图识别错误 | 情绪型输入（骂人/说反话/表达愤怒）被当成常规查询 | 45% |
| 工具调用失败 | API 超时/参数错误，缺乏重试机制 | 30% |
| 上下文管理 | 多轮对话中丢失前面提到的订单号、偏好 | 15% |
| 回复质量 | 语气不当/多余推销/信息不完整 | 10% |

归类的关键：**合并同类项**。你看到的不再是 50 个孤立的失败，而是 4 种失败模式。

**第二步：帕累托定位——20% 的失败模式贡献 80% 的 Bad Case**

画出帕累托图：意图识别错误（45%）+ 工具调用失败（30%）= 75% 的问题。

这意味着：**修好前两种失败模式，就能消灭 75% 的 Bad Case。** 不需要修 50 个，只需要解决 2 个系统性问题。

**第三步：反推策略优化——修一个失败模式，消灭一片 case**

不是"在 Prompt 里加一条规则处理骂人场景"（这是修单个 case），而是问：**为什么意图识别模块整体上对负面情绪不敏感？**

定位到系统性问题：意图识别模块的处理顺序是"先判断意图→再生成回复"，没有"先检测情绪→再判断意图"的步骤。用户的愤怒情绪被正常路由到了查询流程。

反推优化：在 Prompt 中增加"先检测情绪→再判断意图"的推理顺序。优化后意图识别类 Bad Case 减少 65%。

**关键洞察：修一个推理顺序（一行 Prompt 改动），消灭了 45% × 65% ≈ 30% 的全部 Bad Case。**

### 与 Ch17 闭环的关系

Ch17 的闭环策略是宏观流程：E2E 评测 → 失败归因 → 专项优化 → 二次验证。

Bad Case 反推三步法就是其中"失败归因"这一步的详细操作手册。两者配合使用：
- 闭环策略告诉你**什么时候做**归因分析
- Bad Case 反推告诉你**怎么做**归因分析

现在就做：

1. 收集你 Agent 的 10 条失败案例（如果没有，回看 Ch1 盲测和后续各章的实操记录）

## 🏢 后半部分：阿里云"AI 评测 AI"——对话型 Agent 的真实落地

### 💡 为什么对话型 Agent 的评估最难？

对话型 Agent 的评估有一个根本性难题：**没有标准答案**。

用户问"我的快递怎么还没到？"——"快递预计明天到""帮您加急处理了""正在为您查询"都是合法回复。哪个最好，取决于情境、情绪、业务规则。

这和工具型 Agent（步骤对不对？）、内容生成型 Agent（图好不好看？）都不同——对话的好坏本身就是一个需要语义理解的判断。

这意味着：代码评分器（Ch7）只能处理格式和边界，语义质量必须靠 LLM 裁判（Ch8）——而 LLM 裁判的可靠性又是最难保证的。

阿里云面对的就是这个问题。

---

## 🏗️ 架构：评估-诊断-优化三段闭环

阿里云的系统用三段架构解决这个问题：

```
线上客服对话
      ↓
【评估】—— 用大 LLM 对小 LLM 的回答打分
      ↓
【诊断】—— 找出 BadCase 的根因（属于哪类失败模式）
      ↓
【优化】—— 自动生成改进数据，补充知识库
      ↓
【二次验证】—— 优化后回归，确认问题已修复
```

表面上看，这和 Ch17 的闭环策略是一样的——发现→归因→优化→验证。

但阿里云做了一件额外的事：**优化这一步本身也是自动化的**。诊断到"知识库缺乏 X 场景信息"时，系统会自动在互联网上检索相关数据，补充到知识库里，再触发重新评估。

这不是纯粹的评估系统，而是一个**自愈的 Agent 运营系统**。

## 🔧 五大工程技巧

### 技巧一：正负向混合评判（解决 LLM 裁判的单边偏差）

Ch11 讲过 LLM 裁判的位置偏见——先看到正面内容，后面的负面问题容易被忽视。

阿里云的解法：**强制 LLM 先找优点，再找缺点，最后综合权衡**。

```
评判 Prompt 结构：
1. 先列出这条回复做得好的地方（优点 2-3 条）
2. 再列出这条回复的问题点（缺点 1-3 条）
3. 综合上述分析，给出 1-5 分的整体评分
```

这个顺序不是随意设计的——它强迫 LLM 在脑子里同时持有正反两面的信息，再做综合判断，而不是看到前几句好就直接打高分。

实测效果：Bad Case 的召回率从 68% 提升到了 85%+。

---

### 技巧二：两级业务类型/场景映射（解决泛化失效）

通用的评分标准在特定业务场景下会失效。

比如：快递时效类问题的回答要求精准（"预计 3 天内到"比"快了"好），但商品咨询类问题可以更宽泛（"这款适合你"比"规格参数是 XXX"更有体验感）。

用同一套 Rubric 评所有场景，LLM 裁判会判错。

阿里云的解法：**两级映射**：
- **一级**：业务大类（物流/退款/商品/活动）
- **二级**：具体场景（物流-查询/物流-催单/物流-投诉）

每个具体场景对应一套专属 Rubric。评估前，先判断这条对话属于哪个场景，再调用对应 Rubric 打分。

这样 LLM 裁判不再用一把尺子量所有事情，而是对不同场景分别做专业判断。

---

### 技巧三：多 LLM 对抗（解决单一模型偏差）

Ch11 讲的多模型对抗在这里真实落地了。

阿里云用了 2-3 个不同模型家族的 LLM 同时评判同一条对话，取多数投票结果。

关键点：**必须选不同家族的模型**。用两个 Claude 对抗，因为训练数据相似，偏差方向也会相似，对抗价值不大。用 Claude + GPT + 千问，三个模型的偏差方向各不相同，多数投票才能真正降低单一偏差的影响。

代价：3 倍 API 成本，3 倍响应时间。只在关键质量判断环节开启，格式校验等用代码规则处理。

---

### 技巧四：上下文工程（解决 LLM 裁判在长对话里犯糊涂）

Ch23 的内容在这里用到了。

客服对话往往有 10-20 轮，加上评分规则、Few-Shot 示例，一次评估请求很容易超过 5 万 Token。这时候上下文混淆（session ID 混淆、前后轮信息错位）的风险急剧上升。

阿里云的处理：
1. **逻辑拆分**：功能正确性和体验质量分两次独立调用评估，不合并
2. **内容精简**：对话中的 metadata（时间戳、设备信息）在不影响评判的情况下剥离
3. **强制约束**：输出格式锁死为 JSON，`{score: int, reason: str, evidence: str[]}`
4. **Few-Shot 前置**：把示范案例放在 prompt 最前面，利用首因效应

效果：长对话场景下的评分一致性从 72% 提升到 89%（参考 Ch23 的案例数据）。

---

### 技巧五：Checkpoint 中断恢复 + 并发限流

Ch10 提到过这两个工程细节，在阿里云的系统里都真实用到了。

**Checkpoint 中断恢复**：大批量评估（比如一次处理 500 条对话）时，如果中途 LLM API 超时或系统崩溃，不能从头重跑——会浪费大量 API 配额。阿里云在每 50 条对话评估完成后写一次 Checkpoint，中断后从上次 Checkpoint 续跑。

**并发限流（QPM + TPM）**：评估系统自己也在调用 LLM，如果并发太高，会把 API 配额耗尽，影响主系统（AI 客服）的正常服务。阿里云设置了独立的评估专用 API Key，并限制了每分钟请求数（QPM）和 Token 数（TPM）。

这两个细节在系统小规模验证时不会暴露，但一旦评估规模扩大，就会变成拦路虎。

## 📊 实际效果

阿里云这套系统的量化结果：

| 指标 | 数值 |
|------|------|
| Bad Case 发现准确率 | 85%+ |
| 根因归类准确率 | 80%+ |
| 人工标注量减少 | ~70% |

这意味着——以前需要 10 个质检员每天手工审核的工作量，现在 AI 系统每天自动完成，人工只需处理 AI 拿不准的边界案例（约 15-20% 的样本量）。

## 🎯 本节核心观点

**对话型 Agent 评估的核心挑战：没有标准答案——语义质量判断必须靠 LLM 裁判，而 LLM 裁判的可靠性是最大风险点。**

**阿里云的三段架构：评估→诊断→优化，形成自愈的 Agent 运营系统。**

**五大工程技巧（以及对应的前置章节）：**
1. 正负向混合评判 → 解决位置偏见（Ch11）
2. 两级业务场景映射 → 解决泛化失效（Ch12）
3. 多 LLM 对抗 → 解决单一模型偏差（Ch11）
4. 上下文工程 → 解决长对话混淆（Ch23）
5. Checkpoint + 限流 → 解决大批量工程可靠性（Ch10）

**关键洞察：校准比数量重要，不是评估覆盖多少条对话，而是评估准不准。BadCase 发现准确率 85% 的意义在于：你找出的问题是真问题，而不是评估系统自己出错了。**
