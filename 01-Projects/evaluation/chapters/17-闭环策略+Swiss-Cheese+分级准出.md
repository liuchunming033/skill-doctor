---

## 引言

世界上没有完美的评估方法，就像瑞士奶酪没有没洞的那一片。

但你把五片叠在一起，光就穿不过去了。

很多团队的状态是这样的：跑完评估、看到 BadCase、改了 Prompt、觉得好多了——然后点了发布。两周后，线上出了问题，翻找日志发现：那个 BadCase 根本没修干净，而且这次出问题的场景在测试集里压根没有。

评估不是"一次性检验"。它是你和 Agent 之间的持续对话。

---

## 💡 闭环五步：评估的价值在推动改进

不改进的评估，叫记账。

**闭环五步：**

```
Step 1: E2E 评测 → 找到整体通过率
Step 2: 失败归因 → 这些 BadCase 因为什么失败的？
Step 3: 专项优化 → 针对最大的失败模式，改
Step 4: 分层验证 → 只测改动涉及的那一层
Step 5: E2E 二次验证 → 整体通过率有没有提升？
```

以客服 Agent 为例：

E2E 评测发现整体通过率 75%，看起来还行。但归因分析一拆开：

| 失败模式 | 占比 |
|---------|------|
| 逻辑规划错误（API 超时无重试） | 40% |
| 工具调用失败（物流 API 参数错误） | 30% |
| RAG 检索问题 | 20% |
| 结果包装问题 | 10% |

**不要乱枪打鸟——先打最大的那个。**

针对"API 超时无重试"写补丁，专项跑工具调用层测试，通过了，再回到 E2E 验证——通过率从 75% 升到 88%。

剩下的 12% 失败，继续下一轮闭环。

## 🔧 Swiss Cheese：五层叠加，互相补漏

没有任何一层评估能覆盖全部问题。CI 自动化会漏掉"看起来正确实则有风险"的输出，生产监控会漏掉低频但严重的场景，人工评审会漏掉系统性的统计偏差。

**Swiss Cheese Model：每一层都有洞，但叠在一起，光穿不过去。**

```
第一层：CI 自动化评估
  → 每次代码提交自动触发，三层流水线（代码规则 + LLM 裁判 + 回归套件）
  → 速度快，覆盖广，适合高频迭代
  → 漏掉：主观质量、边缘场景
  → 最小实现成本：GitHub Actions 免费额度 + 一个定时脚本

第二层：生产监控
  → 线上真实用户行为，发现测试集没有覆盖的场景
  → 漏掉：低流量的小概率事件（可能发生了但采样没抓到）
  → 最小实现成本：每日导出 10 条对话，人工扫一眼（Ch14 的三种方法）

第三层：A/B 测试
  → 将新版本灰度到 5-10% 用户，比较关键指标变化
  → 漏掉：短期 A/B 测不出的长尾效应
  → 最小实现成本：需要 >1000 日活才有统计意义，<1000 日活可跳过此层

第四层：用户反馈
  → 用户投诉、评分、流失信号——真实用户的感知
  → 漏掉：沉默用户（有问题但懒得反馈）
  → 最小实现成本：对话界面加 👍👎 按钮，每周导出

第五层：人工抽检研究
  → 月度随机抽 30-50 条，深度阅读 Trace，发现系统性盲点
  → 漏掉：无法规模化
  → 最小实现成本：每月 2 小时，一个人，30 条 Trace
```

**画出你的五层图：哪些层已有覆盖，哪些层还是空的？**

没有团队能同时铺满五层。但你得知道自己漏了哪里。只有 CI 自动化的团队，风险在于：通过率很高，但生产里悄悄有人不满意。只有人工抽检的团队，风险在于：抽到的样本代表性不够，而且扩展不了。

## 🎭 分级准出：你的风险级别决定验证力度

不是每次改动都需要全量测试。**问题是：你怎么知道这次改动是几级风险？**

```
L1 — 低风险（Prompt 微调、回复措辞调整）
  → 验证要求：单工具回归测试 + 结果评判
  → 不需要：全量 E2E，不需要 A/B 测试
  → 上线条件：回归通过，LLM 裁判评分 ≥ 基线

L2 — 中风险（模型版本升级、主要业务逻辑变更）
  → 验证要求：多工具链测试 + 关键场景全量测试 + GSB 对比
  → 不需要：全量对抗任务
  → 上线条件：E2E 通过率 ≥ 上版本，关键场景无退化

L3 — 高风险（框架重写、核心架构调整、新能力引入）
  → 验证要求：全量测试 + 对抗任务 + 人工一致性校准 + 灰度发布
  → 上线条件：E2E 通过率明显高于上版本，人工评审一致率 ≥ 85%，灰度阶段无异常
```

**怎么判断这次改动是 L1/L2/L3？**

一个实用标准：

- 改动范围只影响**单一模块**，且有回归覆盖 → L1
- 改动跨越**多个模块**，或影响核心业务逻辑 → L2
- 改动影响**整体架构**，或引入以前没有的能力 → L3

**不要把 L3 改动用 L1 标准发布。** 很多生产事故不是因为改动太大，是因为验证力度跟不上改动力度。

## ⚠️ 一个常见错误：评估通过了就是可以上线了

通过了什么？通过了你的测试集。

你的测试集覆盖了多少真实场景？有没有对抗任务？有没有情绪输入？有没有长尾场景？

**评估的质量决定"通过"的含义。** 一个设计糟糕的测试集给你 95% 通过率，和一个设计完善的测试集给你 80% 通过率，后者包含的信息量更多，更值得信任。

核心公式记住这一句：

**自动化保速度 + 监控给真相 + 人工做校准 + 风险定门禁。**

---

## 🎯 本节核心观点

**闭环五步：** E2E 评测 → 失败归因 → 专项优化 → 分层验证 → E2E 二次验证

**Swiss Cheese 五层：**
- CI 自动化（速度快，覆盖广）
- 生产监控（真实用户，发现盲点）
- A/B 测试（灰度对比，验证变化）
- 用户反馈（感知信号，不可替代）
- 人工抽检（深度洞察，发现系统性问题）

**分级准出 L1/L2/L3：** 风险级别决定验证力度，不是每次改动都要全量测试。

**评估不是终点，是迭代的引擎。一次通过不是真通过，稳定通过才算数。**

## 实操

基于 Ch13 你定位的问题（物流 API 超时不重试），写一条完整闭环计划：

**第一步**：这次改动属于 L1/L2/L3 哪个级别？判断依据是什么？

**第二步**：对应该级别，你需要跑哪些验证？（代码规则层？E2E 层？还是需要 GSB 对比？）

**第三步**：你的 Swiss Cheese 五层里，目前已有哪几层？缺失的层在哪里？

如果你发现缺失的层正好是"生产监控"——说明你现在其实没有能力在线上问题变大之前发现它，这本身就是一个优先级最高的技术债。
