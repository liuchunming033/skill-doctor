## 引言

30 章（含 Ch0 全景图），~165 分钟。

如果只记住一句话：**Agent 开发的质量不取决于你写了多聪明的 Prompt，而取决于你有多了解它做对了什么、做错了什么。**

回到 Ch1 那个"同一个问题跑 3 次"的你。

那时候的结论是：感觉还行，偶尔会犯一些错误，应该问题不大。

这个结论有什么问题？

- "感觉"不能被复现
- "偶尔"不知道是 5% 还是 40%
- "应该没问题"没有任何证据支撑

你没法用这个结论做决定——上不上线？优化哪里？这版比上版好还是坏？都答不上来。

**这就是感觉驱动的 Agent 开发的根本困境：你不知道它有多好，所以你不知道什么时候够好了，也不知道哪里还不够好。**

评估不是负担——它是你唯一能信任的镜子。

## ✅ 毕业清单

你的 Agent 评估系统达到什么程度，才算"毕业"？

**基础层（必须）：**
- [ ] 有 10+ 条经人工确认的 Ground Truth 用例
- [ ] 有至少 2 条自动运行的代码规则（格式/边界检查）
- [ ] 有 1 个校准过的 LLM 裁判 Rubric（Spearman 相关性 > 0.7）
- [ ] 有 Trace 记录（每次运行可追溯）
- [ ] 区分了能力评估和回归评估阶段

**工程层（应该）：**
- [ ] 三层流水线可自动运行（代码→LLM→人工仲裁触发条件）
- [ ] 评估环境有隔离（每次 Trial 从干净状态启动）
- [ ] Bad Case 有分类管理（按失败模式，不是按单个 case）
- [ ] 有明确的准出标准（L1/L2/L3 对应不同上线条件）

**进阶层（有更好）：**
- [ ] pass@k 和 pass^k 都在监控

## 💡 四个关键转变

**转变一：从"跑通一次"到"统计证据"**

跑通一次不是证明，是采样。同一个任务跑 10 次，计算 pass@k 和 pass^k，才是对 Agent 能力的有效陈述。非确定性系统需要统计性结论，不是点估计。

**转变二：从"感觉变好了"到"可对比的分数"**

"感觉这版比上版好"没有信息量。"这版的 GSB 胜率是 73%，上版是 61%，提升显著"才有信息量。可对比的分数让你知道：好在哪里、好多少、值不值得上线。

**转变三：从"评估是验收"到"评估是迭代引擎"**

把评估放在最后做，是在等待失败。评估放在开发循环里，是在用数据驱动每一次迭代。从"写 Prompt → 看效果 → 凭感觉改"升级为"写 Prompt → 跑测评 → 读数据 → 有针对性地改"。

**转变四：从"修单个 Bad Case"到"反推系统性问题"**

一个一个修 Bad Case，修到第 5 个会发现前 3 个的修复互相冲突。把 50 个 case 摊在桌上，找失败模式，找 Top 20% 的系统性问题，修一个消灭 20 个——这是质量工程的正确姿势。

## ⚠️ 永远记住这一条

**永远不要依赖单一评估方法。**

代码评分器有盲区（抓不到语义），LLM 裁判有偏差（近期偏见/评委效应），人工评估有疲劳（标注一致性随时间漂移）。

三层结合：代码评分器处理客观逻辑，LLM 裁判处理语义质量，人工抽检校准对齐——每一层补充另一层的盲区。没有完美的评估方法，只有互补的评估体系。

## 🎯 课程核心观点总结

**认知层：**

> Agent 评估的本质，是把"感觉这版还行"变成"这版在 87 个测试任务上的通过率是 91.3%，比上版提升 8.2%，主要进步来自意图识别类任务，在多轮上下文场景仍有待提升"。

**方法层：**

> 没有标准答案时用 GSB 对比；有参考解时用 MOS/Likert 打分；有基准路径时用 LCS 比对；都有时叠加三层。方法选择跟着场景走，不跟着习惯走。

**工程层：**

> 测试集是资产，要版本控制。评分器是工具，要持续校准。Trace 是证据，要认真读。Bad Case 是信号，要系统反推。评估套件是基础设施，不是一次性项目。

**哲学层：**

> 评估不是给 Agent 打分，是在和 Agent 对话——它告诉你它做错了什么，你告诉它应该怎么做。这个对话永远不会结束，但每一轮都让 Agent 更可靠一点。
