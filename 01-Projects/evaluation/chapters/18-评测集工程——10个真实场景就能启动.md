
> 学习目标：掌握从 0 搭建评测集的四种方法，理解"三层架构"的设计逻辑，学会用混合策略从 10 条真实场景快速启动——不要等"准备好"，这件事永远没有准备好的那天。

---

## 引言

垃圾进，垃圾出。但别因此不敢开始——从 10 条真实用户问题出发，你的评测集就比 90% 的团队更靠谱了。

很多团队卡在"数据不够多"这个借口里，迟迟不启动评估。等收集到 200 条，他们再等 500 条；等 500 条齐了，觉得测试场景不够全……最后 Agent 发布了，评测集还在"建设中"。

**评测集不是工程完工的产物，它是和 Agent 一起生长的系统。10 条够了，先跑起来。**

---

## 💡 从真实用户日志里挖出第一批数据

最好的评测数据从来不是设计出来的，是从真实用户那里挖出来的。

以客服 Agent 为例：打开最近一周的对话日志，挑出频率最高的那 10 条用户提问：

```
"我的订单怎么还没到？"
"能改成顺丰发货吗？"
"这个商品质量有问题，怎么退？"
"我想查一下订单 12345 的物流"
"为什么我的快递一直显示在运输中？"
...
```

这 10 条，比你绞尽脑汁设计的 100 条测试用例更接近真实。它们代表了用户最高频的需求场景，Agent 如果连这 10 条都处理不好，其他都是浮云。

**但真实数据不能直接用——需要清洗。**

清洗三步：
1. 去敏感信息（手机号、地址、真实订单号替换为占位符）
2. 标准化表述（删掉和任务无关的寒暄、重复词）
3. 标注正确答案（预期行为是什么？调用哪个 API？回复什么内容？）

---

## 🔧 三层架构：从简单到复杂，分层管理

有了数据之后，不要把所有用例堆在一起跑。按复杂度分三层：

```
Layer 1 — 单工具任务（200-500 条）
  → 纯查物流、纯查订单状态、纯退款确认
  → 每条只涉及单一 API 调用，逻辑简单
  → 这是你的基础盘，通过率低于 90% 就别谈 L2

Layer 2 — 多工具链任务（100-200 条）
  → 查物流 + 推荐替代品
  → 查订单 + 判断是否符合退款条件 + 发起退款
  → 涉及多步工具调用和状态传递

Layer 3 — 对抗任务（50-100 条）
  → 物流 API 超时后 Agent 会怎么处理？
  → 用户骂人时 Agent 还能保持专业吗？
  → 多个订单号混在一句话里，Agent 能分清吗？
```

**一个关键判断：L1 没过不要跑 L2。L2 没过不要发布。**

三层架构让你的精力配得上优先级——先把基础做稳，再考虑复杂场景，最后才是边界测试。

## 🎭 四种构建方法：用对场景，省十倍时间

不是所有数据都要靠人工一条一条写。四种方法各有用武之地：

| 方法 | 质量 | 速度 | 成本 | 最适用场景 |
|------|------|------|------|-----------|
| 人工构造 | ★★★★★ | ★★☆☆☆ | 高 | 高风险边界用例 |
| 日志挖掘 | ★★★★☆ | ★★★☆☆ | 中 | 冷启动，第一批数据 |
| 数据增强 | ★★★☆☆ | ★★★★☆ | 低 | 扩充多样性 |
| 合成数据 | ★★☆☆☆ | ★★★★★ | 极低 | 快速填充稀缺场景 |

**实践中的正确姿势：混合策略。**

- 用日志挖掘拿到第一批 10-20 条真实数据，建立基础
- 用数据增强扩充多样性（同一个意图，10 种不同表达方式）
- 用合成数据补充 L3 对抗场景（真实日志里罕见，但必须覆盖）
- 用人工构造确认高风险边界（多订单混淆、情绪爆发场景）

合成数据有一个硬门槛：**必须人工审核再入库。** 让 LLM 生成测试用例没问题，但生成的用例如果有逻辑漏洞，就会污染你的评测集，让评估结果完全失真。

---

## ⚠️ 评测集的四项质量要求

不是有数据就是好数据。判断一批测试用例质量好不好，看四个维度：

**1. 高质量问题**：每条用例的输入清晰、无歧义，预期输出有明确判断标准。

**2. 业务真实**：来自真实场景，不是你想象用户"应该"问的问题。

**3. 多样性覆盖**：同一类意图覆盖多种表达——"查物流"可以是"我的快递到哪了？""帮我看看订单状态""能追踪下运输进度吗？"

**4. 无歧义表达**：每条用例只有一个"正确答案"的解释方向。如果一条用例两个人标注会给出不同的预期输出，这条用例要么修改，要么删掉。

## 🎯 本节核心观点

**三层架构：**
- Layer 1（单工具，200-500条）→ 基础盘
- Layer 2（多工具链，100-200条）→ 集成测试
- Layer 3（对抗边界，50-100条）→ 鲁棒性验证

**四种构建方法：** 日志挖掘启动 → 数据增强扩充 → 合成数据补稀缺 → 人工构造守边界

**评测集的终极原则：垃圾进垃圾出，但 10 条真实数据就能启动——不要等"准备好"。**

---

## 实操

从你的 Agent 的用户日志或历史对话中挖 5 条真实问题。清洗后：

**第一步**：对每条写出预期行为（调用什么工具？返回什么内容？）。

**第二步**：按三层架构标注每条属于 Layer 1/2/3。

**第三步**：检查你的 5 条用例——有没有"无歧义"的？有没有"多样性"问题（5 条都是同一类型的意图）？如果有，补充至少 1 条 Layer 3 对抗用例。
