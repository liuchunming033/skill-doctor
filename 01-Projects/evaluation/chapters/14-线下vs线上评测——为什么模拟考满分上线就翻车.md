
> 学习目标：理解线下评测与线上评测的本质差异，掌握"线上失败案例回流线下"的闭环机制，学会识别测试集的覆盖盲点。

---

## 引言

模拟考次次满分，高考却翻车。

Agent 也一样——真实用户不会按你的测试用例说话，他们会骂人、会打错字、会说反话。

你搭好了测试集，Agent 通过率 100%。信心满满上线——用户投诉翻倍。发生了什么？

---

## 💡 线下评测：模拟考试

**线下评测 = 受控环境里的模拟考试。**

你手动设计输入，你知道预期输出，你控制测试条件。优点是可重复、可比较——同一套题，不同版本的 Agent 都跑一遍，分数可以横向对比。

客服 Agent 线下测试 100 条用例：

```
查询订单物流      ✅  通过
更改收货地址      ✅  通过
申请退款         ✅  通过
...
总通过率：100%
```

满分。信心拉满。

上线后，真实用户说："我的快递卡在杭州三天了，你们到底送不送？？"

Agent 理解为"查物流"→正常回复物流状态。用户越看越生气。

**问题：线下用例只覆盖了"标准查询"场景，没有覆盖"带情绪的投诉"场景。**

---

## 🔧 线上评测：实战监测

**线上评测 = 真实生产环境里的持续监测。**

不是你设计输入，是真实用户在说话。优点是覆盖你想不到的情况，缺点是你无法控制——用户什么时候说什么，你不知道。

线上监测发现的三类问题，在线下几乎发现不了：

**1. 边缘输入**：用户打错字、说方言、混用中英文——"查一下俺的快递"、"check my order 1234"。

**2. 情绪输入**：用户愤怒、沮丧、讽刺——这时候 Agent 如果还在"正常回复"，是激化矛盾。

**3. 长尾场景**：低频但真实发生的场景——"我的订单被合并了，三个单号，分别查一下"。这种场景你线下可能一条都没写。

---

## 🎭 核心闭环：线上失败回流线下

这是线下和线上评测真正的价值所在：

```
线下评测（模拟考）
    ↓ 通过，上线
线上监测（实战）
    ↓ 发现新的失败模式
线下扩充测试集（把真实失败案例加进去）
    ↓ 回归验证
重新上线
```

**线上发现的失败案例回流到线下扩充黄金数据集——这是 Agent 效果持续提升的核心路径。**

没有这个闭环，你的测试集会越来越脱离真实用户，评估结果越来越失真。

---

## 🔧 线上监测的三种最小化实现

"线上持续监测"听起来需要一套复杂的系统——但它可以从三件简单的事开始：

1. **对话界面加 👍👎 按钮**：最简单的用户反馈收集——不需要日志系统，不需要数据分析，一个按钮就够。每周导出 👎 的对话，分析最近一周新增的失败模式。

2. **每日抽取 10 条真实对话，人工标注**：不需要全量分析，10 条足够让你感知 Agent 的真实表现。把有问题的对话加进线下测试集。

3. **监控日志中关键字段频率**：追踪"退款""投诉""转人工"等敏感词的出现频率——如果某天突然翻倍，线上可能出了你线下没测到的问题。

这三种方式，成本递增但精度递增。起步用第 1 种，稳定运行后加第 2 种，关键业务叠加第 3 种。

> 线上监测是 Ch17 Swiss Cheese 第二层的具体实现——两者对应同一个防御位置。

## ⚠️ 一张清单：线下能测 vs 线上才知道

| 类别 | 线下能测 | 线上才知道 |
|------|---------|----------|
| 标准场景 | ✅ | ✅ |
| 边缘输入 | 部分（需要主动设计） | ✅ 天然覆盖 |
| 情绪输入 | 需要特意添加 | ✅ 真实发生 |
| 长尾场景 | ❌ 你不知道它存在 | ✅ 用户帮你找 |
| 并发压力 | 部分（需要模拟） | ✅ 真实峰值 |

---

## 🎯 本节核心观点

- **线下评测**：准入验证 + 回归保护。回答"现在这版能上线吗？"
- **线上评测**：监测预警 + 效果闭环。回答"上线后有没有出问题？"

两者的关系不是替代，是互补：**线下保住底线，线上发现盲点，线上的失败案例回流补强线下。**

100% 线下通过率 ≠ 上线安全。你的测试集覆盖不了你没想到的场景。

---

## 实操

列出你 Agent 的两份清单：

**"线下能测的"**：你现在测试集里已经有的场景类型。

**"只有线上才知道的"**：你认为真实用户可能说但测试集里没有的场景（至少想 3 个）。

然后问自己：**你最近一次用户投诉，在你的线下测试集中有对应的用例吗？**

如果没有——这条投诉现在就应该进测试集。
