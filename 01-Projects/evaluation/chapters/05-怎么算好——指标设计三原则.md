## 引言

数字不等于质量。但没有数字，你什么都改不了。

有一个 Agent 团队定义"回复长度 > 50 字"为高质量标准——结果 Agent 学会把每句话加上无意义的客套和重复，把 30 字就能说清楚的事硬拉到 80 字。分数满分，用户投诉翻倍。

**你量什么，Agent 就优化什么。所以要小心你量的是什么。**

## ⚠️ 主观分的死穴

你在 Ch4 给五个维度打了主观分——"功能正确性感觉是 7 分"。

这个分数的问题：

- 明天换个人打，得 5 分
- 放进代码里跑不了
- 分数低了，不知道该改哪里
- 分数高了，不知道是真的好还是侥幸好

**主观判断是评估的起点，不是终点。**

## 🔧 指标设计三原则

以客服 Agent 的"功能正确性"为例，一步步走：

### 原则 1：可量化

把抽象概念转为可计算的值。

"功能正确性感觉不错" → **"物流单号是否与订单匹配？（是/否）"**

不是形容词，是可以计算的判断条件。可量化的指标，不同的人/系统运行，得到同样的结论。

---

### 原则 2：可自动评测

支持自动化，才能融入 CI/CD。

"物流单号是否与订单匹配？" → `assert trace.tracking_id == order.tracking_id`

写成一段脚本，每次 Agent 跑完自动执行。不需要人盯着，不需要人工判断。

可自动评测 = 每次提交代码就能自动验证 Agent 没有退化。

---

### 原则 3：可归因

指标下降时，能定位到哪个模块出了问题。

物流单号匹配失败 → 排查顺序：接口层（物流 API 返回格式是否变了？）→ 工具层（Agent 有没有正确传参？）→ 模型层（LLM 有没有正确提取单号？）→ Prompt 层（提取逻辑是否清晰？）。

**可归因的指标，让优化有方向。不可归因的指标，只能让你焦虑。**

---

## 📐 三原则的递进关系

```
可量化 → 可自动评测 → 可归因
```

三个不是并列的，是递进的：

- 不可量化的东西，写不成代码
- 不能自动跑的代码，进不了 CI
- 没有归因能力的指标，降了也不知道改哪里

设计一个指标，按这三个问题逐一检查：这个值能算出来吗？能自动算吗？降了知道查哪吗？

## 🎭 客观指标 vs 主观指标

| 类型 | 举例 | 适用场景 |
|------|------|---------|
| 客观 | 单号匹配准确率、工具调用次数、Token 消耗 | P0 功能验证、效率监控 |
| 主观 | 回复是否自然、推荐是否合适 | 体验对齐、过程质量 |

**两种都需要，最佳实践是组合。**

客观指标把 P0 问题自动拦住，主观指标目前先用人工处理，方法篇 Ch8 会教你用 LLM-as-Judge 来自动化主观判断。

用客观指标管确定性问题，用主观指标管模糊问题——分工明确，互不替代。

## 🎯 本节核心观点

指标设计三原则递进：

- **可量化**：抽象概念转为可计算的值
- **可自动评测**：写成代码，进 CI/CD
- **可归因**：指标下降，能定位到哪个模块

**一个通不过三原则的指标，要么没用，要么会让你越看越迷糊。**

从你最薄弱的那个维度开始，按三原则设计第一个真实的评估指标。

---

## 实操

从 Ch4 五个维度中选你最薄弱的那个，按三原则设计一个指标：

1. **可量化**：这个维度的"好"如何表达成可计算的条件？
2. **可自动评测**：这个条件怎么写成 Python/Bash 代码？（伪代码也行）
3. **可归因**：如果这个指标降了，你会首先去查哪里？

写完之后检查：这个指标能进你的 CI/CD 吗？如果不能，卡在哪一步？
