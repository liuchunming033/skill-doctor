
> 学习目标：掌握基线评估方法（Baseline-Based Evaluation），理解"跑一次人工确认即为基线"的核心逻辑，学会用过程/结果/效率三维度对比检测退化——让没有标准答案的任务也能被量化评估。

---

## 引言

你没有标准答案。但你跑一次，确认不错，把结果存成文件——这就是你的标准答案。

下次跑出不一样的，就是退化。

很多工程师遇到"开放型输出"就放弃评估，理由是"这种任务没有唯一正确答案，怎么评？"——代码重构建议、性能瓶颈分析、技术方案对比……这类任务的输出确实没有教材上的标准答案。

但"没有唯一答案"不等于"无法评估"。你的判断本身就是标准。把它固化下来，就是基线。

---

## 💡 基线评估的核心思路

基线评估（Baseline-Based Evaluation）回答的不是"这个输出对不对"，而是"这个输出和上次一样好吗？"

**工作流程：**

```
Step 1：首次执行 + 人工确认
  → 运行一次 Agent 任务
  → 人工判断：方向正确？关键内容覆盖了？
  → 确认后，把这次执行快照存为基线

Step 2：后续每次执行，与基线对比
  → 过程对比：关键步骤有没有缺失？顺序有没有偏离？
  → 结果对比：关键分析维度有没有遗漏？
  → 效率对比：Token 消耗是否超基线 1.5 倍？

Step 3：发现偏离时，触发人工复查
  → 偏离 = 退化信号，不是必然退化
  → 人工确认偏离方向：更好了？更差了？还是只是风格变了？
```

这套逻辑的前提是：**你第一次的人工确认是可信的。** 如果基线本身就跑偏了，后续所有对比都会失去意义。

## 🔧 案例：代码分析 Agent 的基线建立

一个代码分析 Agent，任务是"分析这个模块的性能瓶颈"。

没有标准答案——这个模块有 5 个潜在问题，不同工程师可能关注的重点不同。

**第一次执行，建立基线：**

```
输入：src/utils/cache.py 模块
输出：
  - 识别出 3 个性能瓶颈：
    1. 缓存命中率检测逻辑冗余（关键）
    2. 数据库查询未走索引（关键）
    3. 日志写入同步阻塞（中等）
  - 工具调用序列：read_file → analyze_complexity → query_db_schema → 生成报告
  - Token 消耗：2,847

人工确认：
  ✅ 两个"关键"瓶颈确认真实存在
  ✅ 调用序列合理
  → 存为基线 baseline_cache_analysis_v1.json
```

**两周后，模型版本升级，回跑同一任务：**

```
第二次执行结果：
  - 识别出 2 个瓶颈（日志写入未提及）
  - 工具调用序列：read_file → analyze_complexity → 生成报告（跳过了 query_db_schema）
  - Token 消耗：1,923

三维度对比：
  过程：缺少 query_db_schema 步骤 → 偏离 ⚠️
  结果：遗漏了"数据库未走索引"这个关键瓶颈 → 退化信号 🔴
  效率：Token 消耗降低 32% → 改善 ✅
```

结论：效率提升，但核心分析能力退化。模型升级导致了步骤跳跃，遗漏了关键检查。

**这就是基线评估的价值——不是给你一个绝对分数，而是精确告诉你哪里偏了。**

## 🎭 三维度对比：怎么判断偏离的方向

基线对比的结果不是非黑即白的。偏离可能是好事，也可能是坏事。

**过程偏离**

步骤缺失、顺序颠倒、新增了基线里没有的步骤——都是过程偏离。

判断：这个步骤为什么被跳过？是省掉了冗余步骤（好）？还是遗漏了必要检查（坏）？

**结果偏离**

输出的关键维度发生变化——基线里有的结论不见了，或者出现了基线里没有的新结论。

判断：遗漏了关键内容（坏）？还是新发现了之前没注意到的问题（可能是好）？

**效率偏离**

Token 消耗超过基线 1.5 倍，或者降低超过 50%——都值得关注。

超出太多：Agent 在绕路，重复调用，或者陷入了某种循环。
降低太多：Agent 走捷径了，可能跳过了必要步骤（结合过程偏离一起判断）。

## ⚠️ 基线更新的时机

基线不是永远固定的。以下三种情况必须更新基线：

**1. Agent 核心逻辑发生变更**
如果你修改了 Prompt 或工具调用逻辑，旧基线对比没有意义——先确认新版本的输出，再把新确认的结果存为新基线。

**2. 模型版本升级**
新模型的输出风格和逻辑路径可能有系统性变化，旧基线会产生大量"误报退化"。升级前先跑人工确认，更新基线。

**3. 用例描述修改**
用例的输入或期望发生变化时，基线同步更新。否则你在用旧标准评新任务。

**每次更新都必须走人工确认，不能用 AI 自动生成基线。** 基线的可信度，完全来自那一次人工判断的质量。

---

## 🎯 本节核心观点

**基线评估三步：**
1. 首次执行 + 人工确认 → 存为基线
2. 后续执行与基线三维度对比（过程/结果/效率）
3. 偏离触发人工复查，判断是进步还是退化

**基线更新条件：** Agent 逻辑变更 / 模型升级 / 用例修改

**核心价值：把"没有标准答案的任务"转化为"可量化对比的任务"——让开放型输出也能被评估。**

**黄金原则：基线的质量决定评估的质量。跑偏了存的基线，等于把错误固化成了标准。**

## 实操

从 Ch19 你展开的 4 条用例中，选一条没有唯一标准答案的（比如"回复是否得体"或"分析是否到位"）：

**第一步**：执行一次 Agent，人工判断输出方向是否合理（判断依据写 1-2 句话）。

**第二步**：把这次执行的完整快照存下来：输入、工具调用序列、输出、Token 消耗。这就是你的第一版基线。

**第三步**：再跑一次，与基线对比三个维度：
- 过程：步骤顺序一样吗？有缺失或新增步骤吗？
- 结果：关键内容覆盖度一样吗？
- 效率：Token 消耗差异超过 50% 吗？
