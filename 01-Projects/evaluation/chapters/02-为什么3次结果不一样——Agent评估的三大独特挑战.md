---

## 引言

传统软件测试有个美丽的前提：同样的输入，永远得到同样的输出。

你刚跑完的盲测已经证明——Agent 的世界里没有这个前提。

这不是 Bug。这是 Agent 的本质特征。

---

## 💡 三大独特挑战

你在 Ch1 看到的三次不同结果，不是随机偶然。它们来自三个系统性原因。

### 挑战 1：非确定性

LLM 本身就是个概率系统。

自回归生成的链条是这样的：每次生成下一个 token 时，模型先算出所有可能 token 的得分（logits），然后做温度缩放和概率转换，最后从概率分布里随机采样。

就像每次掷一枚偏重的骰子——你改的是骰子的重量分布，不是某次的结果。

一次 Agent 任务可能涉及几十次这样的采样——每次的微小差异通过链式法则指数级放大。

所以"跑通一次"不代表"稳定能跑"。你没见过的那两种行为，一直都在。

---

### 挑战 2：黑盒化

你看到的是输入和输出。中间发生了什么，你不知道。

更可怕的是：模型升级、Prompt 微调、工具链变化——任何一个都可能导致行为漂移，而且肉眼难以察觉。

举一个真实的场景：你把 system prompt 里的"请礼貌回复"改成"请简洁回复"——你觉得只是改了一个措辞。结果某类用户的投诉率上升了 20%，因为"简洁"在投诉场景下被理解成了"冷漠"。但你没有打开过任何日志，你甚至不知道这件事发生了。

**这就是黑盒化的危险：你改了，你以为你改了，但你不知道它变了。**

---

### 挑战 3：错误级联放大

一次任务涉及的工具调用可能多达几十步。

第一步的小偏差，传递到第二步，放大。传递到第三步，继续放大。等到最终结果出来，你已经不知道问题出在哪了。

客服 Agent 那次"直接退款"就是这样发生的——不是因为退款逻辑出了问题，是因为物流 API 超时触发了一个异常分支，而那个分支没有重试机制。

---

## 🎭 对比：传统软件 vs Agent

| 维度 | 传统软件 | Agent |
|------|---------|-------|
| 输入→输出 | 固定输入 → 固定输出 | 相同输入 → 不同输出 |
| 失败可见性 | 报错明确，栈追踪清晰 | 行为偏差，无报错 |
| 测试策略 | 写一次，跑通，提交 | 多次运行，统计证据 |
| 优化方式 | 修代码 | 改概率分布 |

这不是"Agent 有缺陷"。这是 Agent 的本质——你改的不是代码，你改的是概率分布。**这意味着你不能再靠"看代码逻辑"来判断 Agent 好不好——你只能靠统计数据。**

---

## 🚨 关键认知

如果你只跑了一次，你看到的是概率分布的一个样本，不是 Agent 的真实能力。

这就是为什么 Ch1 的盲测要跑 3 次——你需要的不是一个答案，你需要的是一个分布的样子。

**非确定性不是需要消灭的敌人。它是需要管理的现实。**

---

## 🎯 本节核心观点

三个独特挑战：
- **非确定性**：链式随机放大，跑通一次 ≠ 稳定能跑
- **黑盒化**：行为漂移肉眼不可见
- **错误级联**：前序偏差沿链路放大，溯源困难

这三个挑战解释了你在 Ch1 看到的一切差异。

---

## 实操

回看 Ch1 的 3 次盲测结果，逐一标注：

- 哪次差异是**非确定性**造成的？（同一逻辑，随机结果不同）
- 哪次是**黑盒行为**？（中间步骤发生了你没预期的事）
- 有没有**错误级联**？（前面一步出问题，后面跟着崩了）

**示范标注**：以 Ch1 客服 Agent 为例——Trial 2 中"物流 API 超时→直接退款"是错误级联，因为超时是触发条件，导致后续的异常分支被激活。Trial 3 中"物流正确但多推荐了新品"是黑盒行为——你不知道 Agent 为什么在物流查询场景里调用推荐 API。

把标注结果保存下来。后面方法篇的所有工具，都是为了处理这三个挑战。
