# Agent 评估实战课 —— 从"靠感觉"到"靠证据"

> **课程定位**：面向 AI Agent 开发者和团队的工程实战课。学完之后，你能从"跑通一次就觉得没问题"的体感开发，升级到"用数据证明 Agent 能做什么、不能做什么"的工程化开发。
>
> **总时长**：~165 分钟（每章约 5 分钟，共 30 章 + 导言）
>
> **前置知识**：了解 Agent 基本概念（Prompt / Tool / Memory），有至少一次 Agent 开发经验。
>
> **贯穿案例**：你手头正在开发的那个 Agent 就是这门课最好的教材。每一章的"实操"都围绕你自己的 Agent 展开——学一章，动手改一章。

## 课程结构总览

**30 章，~165 分钟。不教概念，教证据。**

大多数 Agent 教程教你写 Prompt。这门课教你写评估——因为你改的不是代码，你改的是概率分布。

# 第一篇：认知篇 —— 从一次盲测开始

> **核心理念**：我见过太多人学评估，第一件事就是背概念。错了。这门课反着来——你先跑盲测，跑出差异了，再用专业语言描述你看到的东西。Agent 的不确定性不是理论，是你亲手跑出来的。
> **贯穿案例**：你手头正在开发的那个 Agent。每章的"实操"都围绕它展开。

## 第 1 章：盲测你的 Agent —— 同一个任务跑 3 次

| 项目 | 内容 |
|------|------|
| **核心问题** | 如果不看理论，直接让你对自己的 Agent 做一次"盲测"——同一个任务跑 3 次，你能发现什么？ |
| **案例** | 一个客服 Agent，用户问"我的订单怎么还没到？"。跑 3 次：第一次查物流返回"已发货"，第二次建议"联系快递员"，第三次直接退款。同一个人问同一个问题，Agent 给了三个完全不同的答案——哪一个是对的？ |
| **核心内容** | 盲测三步：（1）选一个你 Agent 最常处理的任务；（2）用完全相同的输入跑 3 次；（3）记录每次的结果差异（不只是最终答案，还包括中间过程）。典型的发现：答案不同、工具调用不同、推理路径不同、甚至一次成功一次失败。**关键认知：如果你只跑了一次，你不会知道你的 Agent 还有另外两种行为。** |
| **实操** | 对你的 Agent，选一个高频任务，固定输入，跑 3 次。记录：3 次结果是否一致？中间步骤是否一致？有哪次是你没想到的行为？ |
| **开场白** | "别急着学理论。先打开你的 Agent，同一个问题问 3 遍。看看 3 次的答案——一样吗？如果不一样，你之前是怎么判断它'好不好'的？" |
| **时长** | 5 分钟 |
| **参考资料** | [[AgentEvaluation]] § 为什么 Agent 评估比传统 LLM 评估更难？ |

## 第 2 章：为什么 3 次结果不一样 —— Agent 评估的独特挑战

| 项目 | 内容 |
|------|------|
| **核心问题** | 你刚跑完盲测，发现了差异。这些差异不是 Bug——它们暴露了 Agent 评估的三个底层挑战。 |
| **案例** | 接 Ch1 的客服 Agent 盲测结果：第一次查物流 API 返回"运输中"→回复"您的快递预计明天到"；第二次物流 API 超时→Agent 直接退款（跳过确认环节）；第三次查物流成功但 Agent 顺便推荐了新品（行为一致但路径不同）。三次暴露了三个不同的问题。 |
| **核心内容** | 三个独特挑战：（1）**非确定性**——同一 prompt 多次执行结果不同。自回归生成 → Temperature 缩放 logits → Softmax → Top-P 筛选 → 随机采样。每一步随机差异通过链式法则指数级放大。"跑通一次"不代表"稳定能跑"；（2）**黑盒化**——模型升级/Prompt 微调/工具链变化都可能导致行为漂移，肉眼难以察觉；（3）**错误级联放大**——一次任务涉及几十步工具调用，前序步骤的小偏差沿链路逐级放大。对比传统软件：固定输入→固定输出 vs 相同输入→不同输出。 |
| **实操** | 回看 Ch1 的 3 次盲测结果，标注：哪次差异是非确定性造成的？哪次是黑盒行为？有没有错误级联？ |
| **开场白** | "传统软件测试有个美丽的前提：同样的输入，永远得到同样的输出。你刚跑完的盲测已经证明——Agent 的世界里没有这个前提。" |
| **时长** | 5 分钟 |
| **参考资料** | [[AgentEvaluation]] § Agent 评估 vs 传统软件测试；[[AgentEffect]]；[[GraderTypes]] § LLM 非确定性根源 |

## 第 3 章：评估核心概念 —— 把你刚做的事映射成专业语言

| 项目 | 内容 |
|------|------|
| **核心问题** | 你在 Ch1 做了盲测，在 Ch2 分析了差异。现在把这两个动作映射到评估的专业语言——你会发现自己已经完成了第一次 Agent 评估。 |
| **案例** | 用 Ch1 客服 Agent 的例子逐一映射：用户问题"我的订单怎么还没到？"= **Task**；第一次跑 = **Trial 1**；Agent 查物流→返回"已发货"的完整过程 = **Trace**；最终回复"您的快递预计明天到"= **Outcome**；你判断"这个回复对不对"的逻辑 = **Grader**；你跑 Agent 的环境 = **Agent Harness**；3 次 Trial 的组合 = **Suite**。 |
| **核心内容** | 8 个概念不是凭空定义的——它们就是你把 Ch1 的动作拆开后的结果。一张关系图：Task（题目）→ Agent Harness（考场）→ Trial × N（每次独立作答）→ 每次产生 Trace（过程）+ Outcome（答案）→ Grader（评分规则）→ 分数。核心公式：**Eval = Agent 输入 → 执行 → 捕获 Trace + 产物 → 检查规则 → 可对比的分数**。 |
| **实操** | 把你的 Ch1 盲测映射到 8 个概念：你的 Task 是什么？一次 Trial 是什么？Trace 能捕获到哪些字段？你的 Grader 目前是什么样的（哪怕只是你脑子里的判断）？ |
| **开场白** | "你去看 Anthropic 的评估文章，会被 8 个概念砸晕。但其实你刚在 Ch1 已经把它们全部实践了一遍——只是还没给它们起名字。" |
| **时长** | 5 分钟 |
| **参考资料** | [[AgentEvaluation]] § 评估的核心结构 |

## 第 4 章：不只评"对不对" —— 五大评估维度

| 项目       | 内容                                                                                                                                                                                                                                                             |
| -------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **核心问题** | 你在 Ch1 判断 Agent 好不好时，大概率只看了"最终答案对不对"。但如果 Agent 过程精彩但结果错了，或者结果对了但差点把数据库删了——你怎么评？                                                                                                                                                                                |
| **案例**   | 接 Ch1 客服 Agent 的三次盲测结果，从五个维度重新打分：（P0）功能正确性——第一次查物流正确→通过；第二次直接退款未确认→不通过；（P1）过程质量——第三次 Agent 顺便推荐新品，虽然结果正确，但这个"推荐"行为是否合理？（P0）鲁棒性——第二次物流 API 超时后 Agent 没有重试直接退款，容错能力差；（P1）效率——第三次多调了一次推荐 API，增加延迟；（P2）体验——第一次回复语气自然，第二次退款回复生硬。**关键洞察：只看最终结果你会漏掉过程质量和鲁棒性两个维度的问题。** |
| **核心内容** | 两套互补框架对照：（腾讯 TEG 实践）功能正确性(P0) / 过程质量(P1) / 效率成本(P1) / 鲁棒性安全(P0) / 体验对齐(P2)。（Agent 通用维度）任务完成度 / 工具使用质量 / 规划与推理 / 安全与效率。两套框架的区别：腾讯偏"结果导向"，通用五维偏"过程导向"。实际建议：两套对照使用。                                                                                 |
| **实操**   | 回看 Ch1 的 3 次盲测结果，分别从五个维度（功能、过程、效率、鲁棒、体验）各打一个主观分。找出你最薄弱的维度。                                                                                                                                                                                                     |
| **开场白**  | "评估和考试不一样。考试只看最终答案，Agent 评估还得看你是光明正大做出来的——还是差点把数据库删了但侥幸答对了。"                                                                                                                                                                                                   |
| **时长**   | 5 分钟                                                                                                                                                                                                                                                           |
| **参考资料** | [[AgentEvaluation]] § 五大评估维度；[[AgentEffect]]                                                                                                                                                                                                                   |

## 第 5 章：怎么算"好" —— 指标设计三原则

| 项目 | 内容 |
|------|------|
| **核心问题** | 你在 Ch4 给五个维度打了主观分——"功能正确性感觉是 7 分"。但"感觉 7 分"没法放进 CI/CD。怎么把主观判断变成可自动计算的指标？ |
| **案例** | 客服 Agent"功能正确性"的量化：主观判断"它查的物流对不对？"→ 可量化指标"物流单号是否与订单匹配？（是/否）"→ 可自动评测"从 Trace 中提取查到的物流单号，与订单数据库比对"→ 可归因"如果这个指标下降，定位到物流查询工具或 Prompt 中的物流提取逻辑"。 |
| **核心内容** | 客观指标（准确率/字段匹配/工具调用次数）和主观指标（回复自然度/推荐合理性）各有用处，最佳实践是组合。指标设计三原则递进：（1）**可量化**——把抽象概念转为可计算的值；（2）**可自动评测**——支持自动化才能融入 CI/CD；（3）**可归因**——指标下降时能定位到哪个模块（接口层→工具层→模型层→Prompt 层）。三原则关系：**可量化 → 可自动评测 → 可归因**。 |
| **实操** | 从 Ch4 五个维度中选最薄弱的那个维度，按三原则设计一个指标：可量化（怎么算？）→ 可自动（脚本怎么写？）→ 可归因（降了查哪？）。 |
| **开场白** | "有一个团队定义'回复长度 > 50 字'为高质量标准——结果 Agent 学会废话连篇，分数满分但用户投诉翻倍。你量什么，Agent 就优化什么。所以小心你量的是什么。" |
| **时长** | 5 分钟 |
| **参考资料** | [[MetricsDesign]] § 客观指标 vs 主观指标；§ 指标设计三原则 |

# 第二篇：方法篇 —— 从手工到自动

> **核心理念**：能写规则的不写 prompt，能用 prompt 的不用人。从你的人工判断开始，先手工标 Ground Truth，再写成代码规则，再请 LLM 当裁判，最后串成三层流水线。10 条校准好的规则胜过 100 条未校准的。
> **贯穿案例**：用 Ch1 盲测积累的 3 条 Agent 输出，逐步搭建三层评分流水线。

## 第 6 章：先手工评 —— 建立你的 Ground Truth

| 项目 | 内容 |
|------|------|
| **核心问题** | 在写任何自动化评分之前，你首先得知道"正确答案"长什么样。这个"正确答案"谁来定？ |
| **案例** | 对 Ch1 客服 Agent 的 3 条输出，自己手工标注：第一次"物流查询正确，回复准确"→ 通过；第二次"物流超时后直接退款未确认"→ 不通过，标注"应重试物流查询而非直接退款"；第三次"物流正确但多余推荐新品"→ 部分通过，标注"功能正确但不应触发推荐"。这就是你的 Ground Truth——后面所有自动化评分都以它为校准基准。 |
| **核心内容** | 人工评分的三种用途：（1）**建立 Ground Truth**——定义"什么是正确答案"；（2）**仲裁分歧**——当代码和 AI 评分器意见不一致时；（3）**发现盲点**——发现评估体系本身的缺陷。两种组织方式：专家评测（质量高但贵，适合专业场景）vs 众包评测（可扩展但需明确标准）。核心原则：**人类只用于建立标准和处理模糊案例，不做常规重复劳动。** |
| **实操** | 对 Ch1 的 3 条 Agent 输出做人工标注：每条标注"通过/不通过"并写下判断理由（1-2句话）。这就是你的第一版 Ground Truth。 |
| **开场白** | "人工评估是黄金标准。但黄金很贵——你的评估预算里，只有 5-10% 的样本配得上它。剩下的 90%，我们交给规则和 AI。" |
| **时长** | 5 分钟 |
| **参考资料** | [[GraderTypes]] § 人工评分器；§ 人工评测的组织方式 |

## 第 7 章：让规则替你打分 —— 代码评分器

| 项目 | 内容 |
|------|------|
| **核心问题** | 你在 Ch6 手工标了 3 条。如果现在有 300 条——你能把"手工判断的逻辑"写成代码规则，让它自动打分吗？ |
| **案例** | 把 Ch6 的标注逻辑转为代码规则：规则1——"物流单号是否与订单匹配？"→ `assert tracking_order_id == input_order_id`；规则2——"是否包含退款操作且未经用户确认？"→ `assert not (has_refund and not user_confirmed)`；规则3——"工具调用次数是否在合理范围？"→ `assert tool_call_count <= baseline * 1.5`。第2条的标注"部分通过"转规则时发现无法精确表达——这就是代码评分器的边界。 |
| **核心内容** | 三种确定性方法：（1）Rule-based（nDCG/Accuracy/F1/精确匹配）；（2）语义相似度（BERTScore/Cosine）；（3）二进制测试（编译通过？文件存在？Token 不超标？）。优点：毫秒级、零人力、100% 可复现。缺点：对"合法但形式不同"的变体脆弱——Agent 说"纽约"被判错因为预期是"New York"。两个关键特性：**确定性**（同样行为每次判断相同）+ **可调试**（能看到从哪一步开始偏）。 |
| **实操** | 把 Ch6 的 3 条标注转成至少 2 条代码规则。对每条规则标注：它是"确定能判对的"还是"可能误判的"？ |
| **开场白** | "代码评分器就像一个严格的会计——你多写了一个空格它都觉得你错了。但有时候，这种严格正是你需要的。关键是知道什么时候该用它，什么时候该放手。" |
| **时长** | 5 分钟 |
| **参考资料** | [[GraderTypes]] § 确定性评分器；[[SkillScorerSystem]] § 确定性检查 |

## 第 8 章：让 AI 替你打分 —— LLM-as-Judge

| 项目 | 内容 |
|------|------|
| **核心问题** | Ch7 的规则无法处理"回复语气是否自然"、"推荐是否合适"这类软判断。这时候你需要第二个裁判——LLM 自己。 |
| **案例** | 客服 Agent 第 3 条输出："您的快递预计明天到！对了，我们最近上新了春季限定款，要不要看看？"规则能判断物流信息正确，但判断不了"推荐新品是否合适"。让 LLM 当裁判：Prompt 包含（1）任务背景"这是一个物流查询场景"；（2）评分维度"功能正确性/语气自然度/额外行为合理性"；（3）打分标准"1-5 分"；（4）输出格式"JSON：{score, reason, evidence}"。LLM 输出："功能正确性 5 分（物流信息准确），语气自然度 4 分（友好但略过度），额外行为合理性 2 分（物流查询场景不应主动推销）"。 |
| **核心内容** | 五种评估方法：Rubric-based scoring / Natural language assertions / Pairwise comparison / Reference-based evaluation / Multi-judge consensus。关键细节：**输出格式必须被限死**——提前定义 JSON 模板，规定 LLM 必须输出哪些字段（整体是否通过、总分、每个维度得分、为什么这么打分、引用了哪些证据）。你收集的是**评估数据**，不是审稿意见——评估数据是给系统用的，会持续积累。 |
| **实操** | 设计一个 Rubric（至少 3 个独立维度），让 LLM 对 Ch6 的 3 条 Agent 输出打分。对比 LLM 打分和你的人工标注——有分差吗？ |
| **开场白** | "让 LLM 当裁判去评判另一个 LLM——这听起来像让一个学生批改另一个学生的试卷。能行吗？能，但你不能让它'自由心证'。" |
| **时长** | 5 分钟 |
| **参考资料** | [[GraderTypes]] § 模型评分器；§ LLM-as-Judge 的三种评测模式；[[SkillScorerSystem]] § 评分细则检查 |

## 第 9 章：分数怎么组织 —— GSB / MOS / Likert

| 项目 | 内容 |
|------|------|
| **核心问题** | Ch8 你让 LLM 打出了分数。但这个分数该长什么样？是一个数字、一个标签、还是一组数字？三种组织方式对应三种完全不同的决策场景。 |
| **案例** | 对客服 Agent 的一个新版本 Prompt 做评估：（GSB 方式）新版 vs 旧版 → "新版在物流准确性上 Same，在语气上 Better，在额外推荐上 Worse"→ 快速知道"要不要发布"；（MOS 方式）单独给新版打整体分："4.2/5"→ 知道整体水平但不知道为什么；（Likert 方式）逐维度打分：功能 5/5、过程 3/5、效率 4/5、鲁棒 3/5、体验 4/5 → 一目了然"过程质量和鲁棒性是短板"。 |
| **核心内容** | 三种方式对比：（1）**GSB（Good/Same/Bad）**——两版对比，最快，适合日常迭代（"这版 Prompt 比上版好吗？"）；（2）**MOS（1-5 均分）**——整体评分，最简单，适合趋势追踪；（3）**Likert（逐维度打分）**——诊断性最强，适合精细分析（"到底哪里好哪里不好？"）。选择决策树：只需知道哪个版本更好→GSB；只需知道整体好坏→MOS；需要知道哪里好哪里不好→Likert。核心技巧：system prompt 强制模型输出"100 字客观描述 + 逐维度打分理由 + 引用具体元素"。 |
| **实操** | 用 Ch8 的 LLM 打分结果，分别按 GSB（对比 Ch6 人工标注）、MOS（整体均分）、Likert（逐维度）三种方式组织。哪种方式给你的决策信息量最大？ |
| **开场白** | "你让 LLM 打分，它给你一个 4.2。4.2 到底是什么意思——比上一版好？整体不错？还是功能 5 分、体验 3 分？三种意思，三种完全不同的后续行动。" |
| **时长** | 5 分钟 |
| **参考资料** | [[GraderTypes]] § LLM-as-Judge 的三种评测模式 |

## 第 10 章：三层流水线 —— 把代码、AI、人工串起来

| 项目 | 内容 |
|------|------|
| **核心问题** | 你有代码规则（Ch7）、有 LLM 裁判（Ch8）、有人工标注（Ch6）。它们不是三选一，而是要串成一条自动化流水线。 |
| **案例** | 客服 Agent 评估流水线设计：Layer 1（代码过滤，100% 样本）→ 物流单号匹配？是/否；是否有未经确认的退款？是/否；工具调用次数是否超标？是/否。不通过的直接标记 Bad Case；Layer 2（LLM 质量评分，100% 样本）→ 对 Layer 1 通过的样本，用 Rubric 做 5 维度 Likert 打分。低置信度（如评分与上次差异>1.5 分）的样本升级到 Layer 3；Layer 3（人工仲裁，~10% 样本）→ 边界案例+定期抽检+校准。 |
| **核心内容** | 三层流水线架构：代码过滤层（100%样本，零成本，快速淘汰明确失败）→ 模型质量层（100%样本，语义质量评估，低置信度触发升级）→ 人工仲裁层（5-10%样本，边界案例+定期校准）。执行口诀：**"能写规则的不写 prompt，能用 prompt 的不用人。"** **先加快速检查，再按风险叠加慢检查。** 环境隔离前提：流水线跑出来的分数只有在干净环境下才可信。**Claude 内部真实案例——某个 Agent 评估成绩好得离谱，调查发现它在偷看之前试验留下的 git 历史。** 三大陷阱：git 历史残留、资源未释放（数据库/文件/网络连接）、文件残留（临时文件未清理）。每次试验必须是"第一天上班的新人"。Checkpoint 中断恢复：每阶段上报心跳，中断后从上次 Checkpoint 续跑。并发限流（QPM+TPM）：避免 Agent 跑评估时把自己打挂。 |
| **实操** | 用你 Ch7 的 2 条代码规则 + Ch8 的 LLM Rubric，写一段简单脚本（Python/Bash 均可）把 Ch6 的 3 条样本跑一遍：代码层先过滤，通过的再调用 LLM 打分，不通过的直接标记 Bad Case。**加一步：脚本开头先清理临时文件/重置状态，确保每次试验从干净环境启动。** 20 行脚本就够——你手里有全部原料。 |
| **开场白** | "考试有 100 道题——你用机器批 90 道选择题，AI 阅卷批 10 道简答题，只有 AI 拿不准的 3 道才送给老师。你的 Agent 评估也该这样。" |
| **时长** | 5 分钟 |
| **参考资料** | [[GraderTypes]] § 三层评估流水线；§ 推荐组合策略；[[SkillScorerSystem]] |

## 第 11 章：校准与多模型对抗 —— AI 裁判不可全信

| 项目 | 内容 |
|------|------|
| **核心问题** | Ch8 的 LLM 裁判打了分，但它的分准吗？和你的 Ground Truth（Ch6）对得上吗？更可怕的是——同一个 LLM 同一条 prompt 跑三次，能给三个不同的分。 |
| **案例** | 用 Ch6 的 3 条人工标注作为 Ground Truth，对比 Ch8 的 LLM 打分：第 1 条人评"通过"、LLM 评"4/5"→ 一致；第 2 条人评"不通过"、LLM 评"3/5"→ LLM 给了及格分但实际不通过，**召回不足**；第 3 条人评"部分通过"、LLM 评"5/5"→ LLM 完全漏掉了"多余推荐"的问题，**位置偏见**（LLM 被正确的物流信息吸引，忽略了后面的推销行为）。一致率 = 1/3 = 33%，远低于 85% 的可用标准。校准方向：在 Rubric 中增加"检查是否有多余行为"的独立维度。 |
| **核心内容** | 五种系统性偏差：（1）评分漂移（同一答案不同时间得分不同→温度=0，固定 seed）；（2）标准不一致（对"好"的定义模糊→编写详细 rubric）；（3）位置偏见（偏好先出现的选项→随机调换顺序）；（4）长度偏见（偏好更长回答→明确字数要求）；（5）自我提升（偏爱自己风格的答案→用不同模型家族当裁判）。校准三步：收集人工标注 → 计算 Spearman 相关性 → 迭代优化直到 > 0.8。**校准比扩集重要十倍**——10 条校准好的规则胜过 100 条未校准的。多模型对抗：2-3 个不同 LLM 对同一条输出打分，取均值/多数投票，降低单一模型偏差。代价是 2-3 倍耗时。 |
| **实操** | 对比 Ch6 人工标注和 Ch8 LLM 打分，计算一致率。如果不一致，找出最大的偏差来源（位置偏见？标准模糊？长度偏好？）并修改 Rubric。 |
| **开场白** | "你写了 Prompt，信心满满地点运行。第一次：85 分。第二次：72 分。第三次：91 分。是 Prompt 写错了？还是 LLM 在逗你？答案：都不是——LLM 本身就是个概率系统，你需要教会自己怎么管住它。" |
| **时长** | 5 分钟 |
| **参考资料** | [[GraderTypes]] § LLM-as-Judge 校准；§ LLM 非确定性根源与多模型对抗 |

# 第三篇：策略篇 —— 因类施策

> **核心理念**：E2E 告诉你该不该修，分层告诉你该修哪。先搞清楚你评的是哪类 Agent——编码型、对话型、RAG 型、计算机操作型——它们的"好"长得完全不一样。然后选策略、叠防御、定门禁。自动化保速度 + 监控给真相 + 人工做校准 + 风险定门禁。
> **贯穿案例**：用你自己的 Agent 来判断类型、选择策略、制定准出标准。

## 第 12 章：你的 Agent 是哪一类 —— 四类 Agent 评估差异

| 项目       | 内容                                                                                                                                                                                                                                                          |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **核心问题** | 编码 Agent、对话 Agent、研究 Agent、计算机操作 Agent——它们的"好"长得完全不同。你评估的到底是哪一类？                                                                                                                                                                                            |
| **案例**   | 四个对比场景：（1）编码 Agent（SWE-bench）：任务是修 bug，客观标准明确（测试是否通过），评估重点是功能正确性+代码质量；（2）对话 Agent（τ-Bench）：任务是帮用户订机票，没有"标准答案"——"帮订最便宜的"vs"帮订时间最好的"都是合法回答，评估重点是交互质量+任务完成；（3）研究/知识 Agent（RAG 型）：依赖文档检索来回答问题，如"这个模块的性能瓶颈在哪？""竞品定价策略是什么？"——没有唯一正确答案，评估重点是**推理链可追溯+信息来源可靠+检索质量**；（4）计算机操作 Agent：任务是打开网页填表，评估重点是操作序列正确+界面变化后仍能工作。 |
| **核心内容** | 四类 Agent 评估重点对比表：编码（功能正确性+代码质量+效率，SWE-bench/Terminal-Bench）、对话（交互质量+工具准确性+多轮管理，τ-Bench）、研究/知识型（本质是 RAG——推理可追溯+来源可靠+检索质量。这类 Agent 有专项评估框架如 Ragas，核心指标：检索到的文档对不对（Context Precision/Recall）、答案是否基于文档而非瞎编（Faithfulness）、答不上来时能否坦然承认（拒答能力）、事实是否正确（Factual Correctness））、计算机操作（序列正确性+鲁棒性+效率）。核心观点：**先搞清楚你造的是什么类型的 Agent，再决定用什么策略评估它。** 如果你的 Agent 混杂多类，按主导类型定主策略，其他类型作补充。                                                 |
| **实操**   | 判断你的 Agent 属于哪一类（或哪几类的组合），列出针对该类型最重要的 3 个评估指标。对比 Ch4 的五大维度——有新增维度吗？如果你的 Agent 依赖文档检索，它属于"研究/知识型"（RAG 型），请额外列出 3 个 RAG 专项评估指标。                                                                                                                                                                                         |
| **开场白**  | "你不能用考数学的方法来考作文，也不能用考作文的方法来考驾驶。Agent 也一样——编码 Agent、对话 Agent、研究 Agent、计算机操作 Agent，它们的'好'长得完全不一样。"                                                                                                                                                            |
| **时长**   | 5 分钟                                                                                                                                                                                                                                                        |
| **参考资料** | [[AgentEvaluation]] § 五大评估维度；[[AgentEffect]]                                                                                                                                                                                                                |

## 第 13 章：端到端 vs 分层评测 —— 该不该修？该修哪？

| 项目 | 内容 |
|------|------|
| **核心问题** | 你测出 Agent 有问题了——但问题出在意图理解、工具调用、还是结果包装？端到端告诉你"有问题"，分层告诉你"问题在哪"。 |
| **案例** | 客服 Agent 端到端测试发现"订单物流查询"任务通过率只有 60%。分层拆解：意图理解层（是否识别出用户意图为"物流查询"？）→ 通过率 95%，没问题；工具调用层（物流 API 调用是否成功？参数是否正确？）→ 通过率 65%，发现大量超时；结果包装层（返回给用户的回复是否友好？）→ 通过率 90%，没问题。结论：问题出在物流 API 超时处理，不是意图识别，不是回复包装。针对性优化 API 重试机制，通过率从 60%→92%。 |
| **核心内容** | 端到端（E2E）= 黑盒视角，回答"能不能发布"。分层评测 = 白盒视角，按三种维度拆解：架构逻辑（意图→检索→规划→包装）、评估粒度（系统→会话→节点）、业务维度（场景→任务）。最佳实践闭环：E2E 找到问题 → 分层定位根因 → 专项优化 → E2E 验证。**"E2E 告诉你该不该修，分层告诉你该修哪。"** |
| **实操** | 取 Ch1 盲测中那条"失败"的样本（如第二次：物流 API 超时后直接退款），按意图理解→工具调用→结果包装三层逐层诊断：每层预期行为是什么？实际行为是什么？偏差在哪一层？最后用一句话定位根因——"问题出在 X 层的 Y 环节"。 |
| **开场白** | "全链路测试告诉你车跑不快。但到底是发动机的问题还是轮胎的问题？你把整辆车拆了也找不到答案——除非你让师傅单独测每个部件。" |
| **时长** | 5 分钟 |
| **参考资料** | [[EvaluationStrategy]] § 端到端评测 vs 分层评测 |

## 第 14 章：线下 vs 线上评测 —— 为什么模拟考满分上线就翻车？

| 项目 | 内容 |
|------|------|
| **核心问题** | 你搭好了测试集，Agent 通过率 100%。信心满满上线——用户投诉翻倍。发生了什么？ |
| **案例** | 客服 Agent 线下测试 100 条用例全通过。上线后真实用户输入"我的快递卡在杭州三天了，你们到底送不送？？"，Agent 理解为"查物流"→正常回复。但用户其实在表达愤怒，期望的是"道歉+加急处理"。线下用例只覆盖了"标准查询"场景，没有覆盖"带情绪的投诉"场景。这就是线下和线上的 gap——真实用户不会按你的测试用例说话。 |
| **核心内容** | 线下评测 = "模拟考试"：受控环境，黄金数据集，目的准入验证+回归保护。线上评测 = "实战监测"：生产日志，目的是监测预警+效果闭环。关键闭环：**线上发现的失败案例回流到线下扩充黄金数据集**——这是 Agent 效果持续提升的核心路径。 |
| **实操** | 列出你 Agent 的"线下能测"和"只有线上才知道"的事项清单。你最近一次用户投诉，在你的线下测试集中有对应的用例吗？ |
| **开场白** | "模拟考次次满分，高考却翻车。Agent 也一样——真实用户不会按你的测试用例说话，他们会骂人、会打错字、会说反话。" |
| **时长** | 5 分钟 |
| **参考资料** | [[EvaluationStrategy]] § 线下评测 vs 线上评测 |

## 第 15 章：能力评估 vs 回归评估 —— "毕业"机制

| 项目 | 内容 |
|------|------|
| **核心问题** | 同一个测试 Agent 跑了一百次终于过了——然后呢？怎么保证它不会退化？ |
| **案例** | SWE-bench Verified：一年内 Agent 通过率从 40%→>80%。早期是能力评估——通过率低，目标是探索 Agent 的上限，回答"Agent 能把什么做好？"。现在接近饱和→升级为回归评估——每次 CI 必跑，目标是守住底线，回答"原有能力还在不在？"。这个"升级"就是**毕业机制**：能力评估通过率稳定到 100% → 毕业为回归评估 → 进入 CI 流水线。就像驾考：反复练反复考直到通过是能力评估，每年年检确认你还是合格司机是回归评估。 |
| **核心内容** | 能力评估（Capability Eval）= 低通过率起步，探索上限，回答"能把什么做好"。回归评估（Regression Eval）= 接近 100% 通过率，防御漂移，回答"原有能力是否还在"。毕业条件：通过率稳定=100%，进入 CI 每次变更都跑。关键原则：回归套件只增不减——每掌握一个新能力，就增加一道防线。 |
| **实操** | 回看 Ch1 的同一个任务：3 次盲测中几次通过？如果设定"连续 10 次 100% 通过才算毕业"，这个任务现在处于能力评估阶段还是毕业了？如果毕业，你会在 CI 中怎么跑它？ |
| **开场白** | "驾照考试反复练、反复考，直到通过——这是能力评估。年检每年确认你还是合格的司机——这是回归评估。问题是：拿到驾照后，你是越开越好，还是越开越野？" |
| **时长** | 5 分钟 |
| **参考资料** | [[CapabilityVsRegressionEval]]；[[AgentEvaluation]] § 能力评估 vs 回归评估 |

## 第 16 章：打分评估 vs 对比评估 —— 看体检报告还是看排名？

| 项目 | 内容 |
|------|------|
| **核心问题** | 评估时看绝对分数就够了，还是必须横向对比？两种参照系对应两种完全不同的决策。 |
| **案例** | 两个 Prompt 版本：版本 A 功能正确性 85%、版本 B 功能正确性 82%。只看绝对分数 → A 更好。但对比评估发现：版本 B 的过程质量 90%（vs A 的 70%），且通过了所有异常容错用例，Token 消耗少 30%。如果只看绝对分数选 A，上线后可能因鲁棒性差导致更多事故。反过来——如果只对比不设绝对阈值，可能两个都烂但选了"不那么烂"的那个。**打分评估和对比评估回答的是不同问题：能不能上线？vs 选哪个更好？** |
| **核心内容** | 打分评估 = "体检报告"：参照系是绝对标准或预设阈值，输出通过/不通过，适合发布前质量把关。对比评估 = "运动员排名"：参照系是其他版本或竞品，输出 A>B 或 GSB，适合选型决策和迭代验证。关键区别：打分只需待评对象+阈值，对比必须多对象同条件数据。实践：研发早期选型用对比，发布前把关用打分。两者不是二选一——先对比选最优，再打分验门槛。 |
| **实操** | 对比你 Agent 的两个版本（如不同 Prompt 或不同模型），做一次 GSB 判断：Good/Same/Bad？再对选出的版本做一次绝对打分：达到你的发布阈值了吗？ |
| **开场白** | "体检报告说你血压 120——是超标了（标准 140），还是比上次高了（上次 110）？两种参照，两种完全不同的行动。Agent 评估也一样：你到底是想知道'它合格吗'，还是'它比上一版好吗'？" |
| **时长** | 5 分钟 |
| **参考资料** | [[EvaluationStrategy]] § 打分评估 vs 对比评估 |

## 第 17 章：闭环策略 + Swiss Cheese + 分级准出

| 项目 | 内容 |
|------|------|
| **核心问题** | 评估跑完了，BadCase 出来了——然后呢？评估的价值不在"发现问题"，在"推动改进"。而且，单靠一种评估方法是挡不住所有问题的。改进到什么样才算"可以上线"？ |
| **案例** | 客服 Agent 端到端评测发现整体通过率 75%。归因分析：逻辑规划错误 40%（API 超时后没有重试策略）、工具调用失败 30%（物流 API 参数拼接错误）、RAG 问题 20%、包装问题 10%。Target 最大的——优化 API 重试逻辑 → 专项分层测试 → 二次 E2E 通过率从 75%→88%。同时叠了 5 层防御：CI 自动化评估+生产监控+A/B 测试+用户反馈+月度人工抽检（Swiss Cheese）。上线前按分级准出：这次改的是重试逻辑（L1 低风险），过单工具回归+结果评判即可上线。 |
| **核心内容** | 闭环五步：E2E 评测 → 失败归因 → 专项优化 → 二次验证。Swiss Cheese：自动化评估+生产监控+A/B 测试+用户反馈+人工研究，五层叠加互相补漏。分级准出：L1 低风险（Prompt 微调）→ 回归+结果评判；L2 中风险（模型升级）→ 多工具链+关键场景全量；L3 高风险（框架重写）→ 对抗任务+人工一致性+灰度。核心公式：**自动化保速度 + 监控给真相 + 人工做校准 + 风险定门禁。** |
| **实操** | 基于 Ch13 你定位的问题（物流 API 超时后不重试），写一条"归因→优化→验证"的闭环计划。画出你的 Agent 的 Swiss Cheese 五层图（已覆盖的层 vs 缺失的层）。定义你的 L1/L2/L3 准出标准——这次优化属于 L1/L2/L3 哪个级别？上线前该跑哪些检查？ |
| **开场白** | "世界上没有完美的评估方法，就像瑞士奶酪没有没洞的那一片。但你把五片叠在一起，光就穿不过去了。最后再问一个问题：Agent 好到什么样，你才敢点'发布'按钮？" |
| **时长** | 5 分钟 |
| **参考资料** | [[EvaluationStrategy]] § 闭环策略；[[AgentEvaluation]] § Swiss Cheese Model；§ 分级准出标准 |

# 第四篇：工程篇 —— 零件、组装、路线图

> **核心理念**：垃圾进垃圾出，但别因此不敢开始。评测集是零件，用例是图纸，基线是把"不可测"变成"可测"的魔法，Trace 是你的眼睛。10 条真实场景就能启动，Anthropic 也是这么开始的。
> **贯穿案例**：为你的 Agent 搭建从数据→用例→基线→Trace→路线图的完整评估工程。

## 第 18 章：评测集工程 —— 10 个真实场景就能启动

| 项目 | 内容 |
|------|------|
| **核心问题** | 评测集质量不好，整个评估体系就废了。但怎么从 0 搭建一个好的评测集？需要多少数据？ |
| **案例** | 客服 Agent 评测集从 0 搭建：（1）从最近一周的客服对话日志中挖 10 条真实用户问题——"我的订单怎么还没到？""能改成顺丰吗？""这个商品质量有问题怎么办？"；（2）清洗：去敏感信息、标准化问题表述、标注正确答案；（3）分类：按三层架构组织——Layer 1 单工具任务（纯查物流/纯查订单，200 条）、Layer 2 多工具链任务（查物流+推荐替代品，100 条）、Layer 3 对抗任务（物流超时/用户骂人/多订单混淆，50 条）。不要等"准备好"——10 条就能启动第一版。 |
| **核心内容** | 四项质量要求：高质量问题+业务真实数据+多样性覆盖+无歧义表达。四种构建方法对比：人工构造（最贵但最好）→ 日志挖掘（真实但需清洗）→ 数据增强（提多样性）→ 合成数据（快但需审核）。实践：**混合策略 + 10 条真实高频场景启动**。三层架构：单工具(200-500)→多工具链(100-200)→对抗边界(50-100)。 |
| **实操** | 从你的 Agent 的用户日志或历史对话中挖 5 条真实问题，清洗后作为第一批评测样本。按三层架构标注每条属于哪一层。 |
| **开场白** | "垃圾进，垃圾出。但别因此不敢开始——从 10 条真实用户问题出发，你的评测集就比 90% 的团队更靠谱了。" |
| **时长** | 5 分钟 |
| **参考资料** | [[EvalDatasetEngineering]] § 高质量评测集的四项要求；§ 四种构建方法 |

## 第 19 章：用例设计四场景 —— 一条好用例的四种写法

| 项目 | 内容 |
|------|------|
| **核心问题** | 你在 Ch18 挖了真实问题，但每条问题该怎么写成规范的测试用例？一条好用例应该同时覆盖四个场景。 |
| **案例** | 客服 Agent 物流查询场景的四条用例：（1）触发条件——"今天天气真好"（不应触发物流查询）vs "我的快递到哪了"（应触发）；（2）核心逻辑——"帮我查订单 12345 的物流"（预期：调用物流 API，参数 order_id=12345，返回物流状态+预计到达时间）；（3）产物质量——输出是否包含物流单号、当前状态、预计到达时间三个必要字段？（4）异常容错——"查订单 ABCDE 的物流"（订单号不存在，期望 Agent 回复"未找到该订单"而非崩溃）。 |
| **核心内容** | 四场景设计法：触发条件（15%）+核心逻辑（50%，用例量最大）+产物质量（20%）+异常容错（15%）。**如果只写核心逻辑用例，你的 Agent 就像一个只练正手的运动员——正手漂亮，反手一打就穿。** |
| **实操** | 从 Ch18 的 5 条真实问题中选 1 条，按四场景法展开为 4 条用例。检查：你的 Agent 目前的测试用例中，哪类场景最薄弱？ |
| **开场白** | "你写了 100 条测试用例，99 条在测'Agent 能不能做对'，只有 1 条在测'Agent 会不会做错'。这不是测试，这是给自己找信心。" |
| **时长** | 5 分钟 |
| **参考资料** | [[EvaluationStrategy]] § 用例设计 |

## 第 20 章：基线评估实操 —— 让"不可测"变成"可测"

| 项目 | 内容 |
|------|------|
| **核心问题** | 你的 Agent 生成了一份代码重构方案。好还是不好？你没有标准答案。很多 Agent 任务的输出没有唯一的"正确答案"——但你不能因此不评。 |
| **案例** | 一个代码分析 Agent，任务"分析这个模块的性能瓶颈"。没有标准答案。基线法：跑一次，人工确认"这个分析方向正确，指出的 3 个瓶颈中有 2 个确实存在"，把这次执行快照（思维链+工具调用序列+最终报告）存为基线。后续每次评估：对比本次执行与基线的差异——过程对比（关键步骤是否缺失/顺序是否偏离）、结果对比（是否遗漏关键分析维度）、效率对比（Token 消耗是否超基线 1.5 倍）。 |
| **核心内容** | 基线评估方法（Baseline-Based Evaluation）：不手写预期输出，先执行一次经人工确认后作为基线。后续评估与基线对比三维度：过程/结果/效率。基线更新条件：Agent 逻辑变更、模型升级、用例修改，每次更新走人工确认。 |
| **实操** | 从 Ch19 你展开的 4 条用例中，选一条没有标准答案的（如"回复是否得体"），执行一次 Agent，人工确认后存为基线。再跑一次，对比过程/结果/效率三个维度是否偏离。 |
| **开场白** | "你没有标准答案。但你跑一次，确认不错，把结果存成文件——这就是你的标准答案。下次跑出不一样的，就是退化。" |
| **时长** | 5 分钟 |
| **参考资料** | [[BaselineBasedEvaluation]] |

## 第 21 章：可观测性基础设施 —— Trace 是怎么来的

| 项目 | 内容 |
|------|------|
| **核心问题** | 前几章建的基线、用例、评测集都是评估的"零件"，但评估真正运行时，Trace 是怎么被采集的？怎么保证你看到的 Trace 是完整的？ |
| **案例** | 用 Langfuse 演示 Trace 采集的实际代码：一次 Agent 任务的完整调用链（意图识别→工具调用→生成回复），每个 span 记录了耗时、token 数、输入输出。四类数据缺一不可：输入/输出数据、中间过程数据、元数据、评估数据。 |
| **核心内容** | Trace 采集的完整链路：Agent 执行 → span 记录 → 存储到可观测平台。四类数据（输入/输出/中间过程/元数据）各解决什么问题。框架选型指南（Langfuse / Arize / Phoenix / 自建）。 |
| **实操** | 在你的 Agent 中接入一个 Trace 采集工具，跑一次任务，检查 Trace 是否完整记录了调用链。验收标准：能看到每个 span 的耗时和 token 数——如果都是 0，说明初始化配置有问题。 |
| **开场白** | "之前我们用 Ch1 的盲测感受不确定性，用 Ch3 的 JSON 理解 Trace 结构。但 Trace 不是凭空产生的——它是被工具'抓'出来的。这一章告诉你抓 Trace 的完整链路。" |
| **时长** | 5 分钟 |

## 第 22 章：Trace 分析与过程评分 —— 不看过程就是瞎子

| 项目 | 内容 |
|------|------|
| **核心问题** | 评分器告诉你 Agent 拿了 85 分。但如果 Agent 在过程中差点删了数据库，这 85 分还有意义吗？ |
| **案例** | 客服 Agent 的一条 Trace 记录：Step 1 正确识别用户意图"物流查询"→ Step 2 调用物流 API，超时→ Step 3 没有重试，直接调用了退款 API→ Step 4 退款 API 需要确认，Agent 发送了确认→ Step 5 用户取消退款。只看最终结果：订单状态未变，用户取消，Agent 回复"如有需要请再联系"——评分器可能给 4/5，因为对话没有严重错误。看 Trace 才发现：Agent 在 Step 3 有一个危险行为（跳过重试直接退款），只是运气好被用户取消了。**这就是只看结果的评分器盲区。** |
| **核心内容** | Trace 分析三个层次：表面层（调用序列正确？参数准确？）、效率层（绕了多少弯路？死循环？）、风险层（有没有危险操作？）。四个关键洞察：错误≠失败、正确≠成功、路径依赖、评分器盲区。过程评估三种方式：里程碑检查、错误恢复评分、与已知良好路径对比。 |
| **实操** | 回看 Ch20 你跑的基线评估——如果有偏离，打开那条偏离的 Trace，逐行标注：表面层（调用序列对了吗？）、效率层（绕路了吗？）、风险层（有危险操作吗？）。找出偏离的根因。 |
| **开场白** | "评分器告诉你 Agent 拿了 85 分。你看一眼 Trace——第三步差点删库，第五步绕了 10 个圈子才找到答案。这 85 分的水分有多大？" |
| **时长** | 5 分钟 |
| **参考资料** | [[Trace]]；[[AgentEvaluation]] § 四个关键洞察；[[EvalHarness]] § Trace 输出能力 |

# 第五篇：进阶篇 —— 上下文、统计与自进化

> **核心理念**：窗口大 ≠ 脑子好用。一次通过 ≠ 每次通过。基础工程搭好后，应对三个 hard 问题——LLM 裁判在长上下文里"犯糊涂"、深度思考模式什么时候值得开、非确定性让你没法下结论。
> **贯穿案例**：优化你的评估系统——从"能用"到"可信"。

## 第 24 章：上下文工程 —— LLM 裁判为什么在长文档里犯糊涂？

| 项目 | 内容 |
|------|------|
| **核心问题** | 给 LLM 裁判的 Prompt 总共几万 Token——你以为信息越多越准确？结果是 LLM 把所有 ID 都搞混了。 |
| **案例** | 一个评估 Prompt：包含 50 条客服对话记录（每条有 session_id）、20 条评分规则、10 个 Few-Shot 范例。LLM 裁判把 session_abc123 的物流错误归到了 session_abc124 上——因为两个 ID 太相似，在长上下文中发生了**上下文混淆**。瘦身方案：删掉 50 条中不需要的 metadata 字段（从 200 Token/条 压缩到 80 Token/条）、长 ID 映射为短编号（session_abc123 → S1）、拆分复合指令（先评功能正确性 → 另一次调用评体验）。评分准确率从 72% 提升到 89%。 |
| **核心内容** | 四种失效场景：上下文冲突（近因 vs 首因）、上下文混淆（张冠李戴）、上下文干扰（无关信息污染计算）、上下文丢失（开头信息被边缘化）。五种策略：逻辑拆分+内容精简（控制在 5w token 内）+强制约束（纯净 JSON）+Few-Shot+首因效应利用。核心原则：**降低 Token 量 ≠ 降低信息量——去冗余、做压缩、拆逻辑。** |
| **实操** | 对你 Ch8 的评估 Prompt 做瘦身：删冗余字段、长 ID 映射短编号、拆复合指令。对比瘦身前后的评分一致性。 |
| **开场白** | "LLM 号称支持百万 Token 窗口。你把 50 条记录、20 条规则、10 个范例全扔进 Prompt——LLM 把所有人的 ID 都搞混了。窗口大 ≠ 脑子好用。" |
| **时长** | 5 分钟 |
| **参考资料** | [[ContextEngineering]] § 四种上下文失效场景；§ 五种应对策略 |

## 第 25 章：深度思考模式 —— 你该不该让它多想一想？

| 项目 | 内容 |
|------|------|
| **核心问题** | 深度思考模式能提升推理准确性，但也让响应慢 N 倍、成本贵 N 倍。什么时候值得开？ |
| **案例** | 两个评估任务对比：（A）客服对话的"是否包含退款操作？"→ 代码评分器已能精确判断，开深度思考=浪费；（B）客服对话的"Agent 的回复是否得体？是否在正确的时间说了正确的话？"→ 需要多维度语义理解+语境判断，开深度思考后评分与人工评分的 Spearman 相关性从 0.65 提升到 0.82。关键判断依据：**任务需要"理解为什么"还是只需要"判断是什么"？** |
| **核心内容** | 深度思考 ≠ 普通 CoT：五步流程（问题解析→多路径推理→逐步验证→自我批判→最终整合）。适用：复杂推理+高可靠要求→开；事实查询+格式转换→关。代价：响应慢数倍，思考输出也消耗 TPM 配额。 |
| **实操** | 对同一条评估任务，分别开关深度思考各跑一次。对比：评分准确度、耗时、成本。你的评估场景值得开吗？ |
| **开场白** | "深度思考模式就像让裁判在打分前先自言自语十分钟。有时这十分钟值回票价，有时纯属浪费。" |
| **时长** | 5 分钟 |
| **参考资料** | [[DeepThinkingMode]] |

## 第 26 章：pass@k 与 pass^k —— 非确定性场景下的统计证据

| 项目 | 内容 |
|------|------|
| **核心问题** | 同一个任务 Agent 跑 10 次，成功 7 次。这算通过还是没通过？只跑一次得出结论是自欺欺人。 |
| **案例** | 客服 Agent 执行物流查询任务 10 次：pass@10 = 95%（至少一次成功，说明 Agent 有能力做好），pass^10 = 30%（10 次全部成功，说明稳定性很差）。这意味着 Agent 偶尔能做对，但大部分时候不可靠。**如果只看 pass@10 你会觉得 Agent 很棒；如果只看 pass^10 你会觉得 Agent 很烂。两个指标一起看，你才知道：Agent 能力够，但工程稳定性不足——需要优化的是重试机制和超时处理，而非 Prompt 或模型。** |
| **核心内容** | pass@k（k 次至少 1 次通过）→ 衡量峰值能力。pass^k（k 次全部通过）→ 衡量稳定性。实践：能力评估阶段关注 pass@k（探索上限），回归评估阶段关注 pass^k（守住下限）。稳定性容忍阈值：关键决策类 0% 失败容忍、辅助分析类 ≤10%、创意类 ≤40%。 |
| **实操** | 对同一个任务执行 10 次（或至少 5 次），计算 pass@5 和 pass^5。你的 Agent 是"能做好"还是"能做稳"？ |
| **开场白** | "同一道题做 10 次，做对 3 次——不是运气好叫能力强，是概率在帮你。真正的过关是 10 次都做对。" |
| **时长** | 5 分钟 |
| **参考资料** | [[passAtK]] |

## 第 27 章：Bad Case 系统化反推 + 阿里云"AI 评测 AI"（对话型 Agent）

| 项目 | 内容 |

| **核心问题** | （前半）Bad Case 列表越来越长，盯着单个修修不完。怎么从散乱的失败中找出系统性根因？（后半）找出来之后，真实团队是怎么落地"评估-诊断-优化"闭环的？ |
| **案例** | **前半部分（Bad Case 反推）**：客服 Agent 积累 50 条 Bad Case。按失败模式归类：意图识别错误 45%、工具调用失败 30%、上下文管理 15%、回复质量 10%。帕累托定位：前两类贡献 75% 问题。反推优化：在 Prompt 中加入"先检测情绪→再判断意图"推理顺序，意图识别类 Bad Case 减少 65%。**后半部分（阿里云）**：阿里云智能客服团队——用更大的 LLM + 深度思考模式评测较小的客服 Agent。架构：评估（四维度）→ 诊断（十几种根因归类）→ 优化（自动生成相似问+互联网检索补充）→ 二次验证。五大工程技巧：正负向混合评判、两级业务映射、多 LLM 对抗、上下文工程、Checkpoint 限流。效果：BadCase 发现准确率 85%+，根因归类准确率 80%+。 |
| **核心内容** | Bad Case 反推三步法（归类抽象→帕累托定位→反推优化）。对话型 Agent 评估核心挑战：没有标准答案，语义质量必须靠 LLM 裁判。架构模式：评估-诊断-优化自愈闭环。与 Ch17 闭环的关系：Bad Case 反推是闭环中"失败归因"步骤的详细操作手册。 |

## 第 28 章：实战案例 —— 腾讯 TPerf 智能分析（工具型 Agent）

| 项目 | 内容 |
|------|------|
| **核心问题** | 阿里云是对话型 Agent，如果换成工具型 Agent（如性能分析），评估体系有什么不同？ |
| **案例** | 腾讯 TEG 云架构平台部网关测试团队在 TPerf 性能平台智能分析 Agent 上的完整实践。**评分机制创新——100 分扣分制**：步骤正确性 10 分（工具调用序列+参数，LCS 最长公共子序列算法与基线比对）+ 效率 10 分（Token 消耗+工具调用次数是否超基线）+ 报告质量 80 分（数据准确性+分析深度+结构完整性+建议可执行性）。扣分制的好处：Agent 天然有得分冲动，扣分制暴露最薄弱环节。**稳定性"0 容忍"**：关键决策类 Agent，任一 Trial 不达标=整个用例不通过——不是取平均，不是多数投票，是"一次失败=全部失败"。**四种执行模式**：快速分析（单模型单用例，秒级，开发调试）→ 全量测评（所有模型×所有用例×多次 Trial，3-5h，发布前）→ 按模型筛选（指定模型跑全量，模型升级验证）→ 重新评分（Trace 已存，只重跑评分器，评分规则迭代）。 |
| **核心内容** | 工具型 Agent 评估特点：客观标准明确（步骤对不对、参数对不对），评估重点是过程正确性+结果准确性。核心方法：LCS 序列比对 + 扣分制 + 0 容忍稳定性。 |
| **实操** | 借鉴 TPerf 的扣分制，为自己的 Agent 设计一个 100 分制的评分表，标注各维度分值权重。你的 Agent 适合 0 容忍稳定性策略吗？ |
| **开场白** | "阿里云教你用 AI 评测 AI，腾讯 TPerf 教了你另一件事：对于关键决策类 Agent，'差不多对了'就是错了。一次失败，全部不通过。你敢用这个标准要求你的 Agent 吗？" |
| **时长** | 5 分钟 |
| **参考资料** | [[CapabilityVsRegressionEval]]；[[BaselineBasedEvaluation]]；[[Trace]] |

## 第 29 章：实战案例 —— AIGC 配图 Agent 评测从 0 到 1（内容生成型 Agent）

| 项目 | 内容 |
|------|------|
| **核心问题** | 对话型 Agent 评交互，工具型 Agent 评过程——如果是内容生成型 Agent（如 AI 配图），输出是"一张图好不好看"，怎么从 0 搭评估体系？ |
| **案例** | AIGC 配图 Agent 评测 6 步法完整拆解：**Step 1 拆维度**——主题相关性、风格一致性、视觉质量、安全合规（避免"图好不好看"这种空泛标准）；**Step 2 建黄金集**——10 个高频场景即可启动，每个 case 仅需输入+风格期望；**Step 3 人评+机评**——GSB/MOS/Likert 选择（对应 Ch9），system prompt 强制模型暴露 100 字客观描述+逐维度理由+证据引用；**Step 4 校准 AI 评分**——三个指标对齐人机：各维度分差（>1 不对齐）、排序倒置率（>20% 不对齐）、极端 case 一致率。**校准比扩集重要十倍。** 实践：修改 prompt 后风格一致性均分差从 0.8→0.3；**Step 5 扩集到生产**——黄金集 10 个校准评分器，生产集 100-500 个按 query 频率构建，5% 抽样人评校准，95% AI 跑分；**Step 6 Bad Case 反推**——"概念类文章配图普遍低分"→ 反推出生成 prompt 的抽象隐喻能力不足 → 针对性优化。**动态进化**：黄金集每月更新，新模型回跑历史问题，终极目标是自进化。 |
| **核心内容** | 内容生成型 Agent 评估特点：输出没有客观标准（什么是"好看的图"？），评估依赖校准驱动。核心方法：多维度 Likert + AI 评分 + 人工抽检校准。与其他两类 Agent 的关键差异：校准投入占比最高（校准比扩集重要十倍）。 |
| **实操** | 借鉴 6 步法，为你的 Agent 写一份"从 0 到 1 搭建评测系统"的执行计划，标注每步的具体产出和时间节点。 |
| **开场白** | "阿里云教你怎么评对话，腾讯教你怎么评工具。但如果你的 Agent 输出的是'一张图'——好不好看谁说了算？AIGC 配图团队用 6 步从 0 搭起了整套系统，每一步都有踩坑记录。" |
