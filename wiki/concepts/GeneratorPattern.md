## 定义

Generator 模式（固定输出结构）是 Google 总结的五种 Skill 设计模式之一，解决"为什么让 Agent 写文档，今天长这样、明天又长成另一种样子"的问题。核心做法：把输出结构固定在模板里（回答 what to produce）、把表达方式固定在风格指南里（回答 how to write）、SKILL.md 负责先补问缺失变量再把内容填进去。

## 解决的问题

**输出格式不稳定**：
- 让 Agent 写文档，今天长这样、明天长成另一样
- 每次输出格式、结构、风格都不一致
- 需要人工反复调整格式

**根本原因**：
- 模型每次都在现场重新决定输出该长什么样
- 没有固定的模板和风格约束
- 输出质量依赖模型的"即兴发挥"

## 核心做法

### 三层结构

```
SKILL.md (协调层)
├── 分析任务，判断需要哪些变量
├── 补问缺失的变量
└── 将变量填入模板

templates/ (结构层)
├── api-doc-template.md — API 文档模板
├── report-template.md — 报告模板
└── commit-template.md — Commit message 模板

style-guides/ (风格层)
├── technical-writing-style.md — 技术文档写作风格
├── concise-style.md — 简洁风格
└── formal-style.md — 正式风格
```

### 工作流程

1. **分析任务**：判断需要生成什么类型的文档
2. **选择模板**：根据文档类型选择对应的模板
3. **识别变量**：分析模板中需要填充的变量
4. **补问变量**：如果缺失变量，向用户询问
5. **填充内容**：将变量填入模板
6. **应用风格**：按照风格指南调整表达方式

## 两层约束

### 模板（回答 what to produce）

- **结构固定**：文档的章节、标题、顺序固定
- **必填字段**：明确哪些字段必须填写
- **可选字段**：明确哪些字段可选
- **示例值**：提供每个字段的示例值

### 风格指南（回答 how to write）

- **语言风格**：正式/简洁/技术性
- **句式规范**：使用什么样的句式
- **术语约定**：使用哪些术语、避免哪些术语
- **格式要求**：代码块、表格、列表的使用规范

## 适用场景

- **API 文档**：接口文档、SDK 文档、集成文档
- **标准化报告**：周报、月报、项目报告
- **Commit message**：Git 提交信息
- **脚手架生成**：项目初始化、代码模板生成
- **配置文件**：环境配置、部署配置

## 优势

### 输出一致性

- 相同类型的文档结构完全一致
- 风格统一，无需人工调整

### 质量保证

- 模板保证必需字段不遗漏
- 风格指南保证表达质量

### 可维护性

- 修改模板即可更新所有文档格式
- 修改风格指南即可更新所有文档风格

## 实现要点

### 模板设计

**变量占位符**：
```markdown
# {{title}}

## 概述
{{overview}}

## API 端点
{{endpoint}}

## 参数
| 参数名 | 类型 | 说明 |
|--------|------|------|
{{#parameters}}
| {{name}} | {{type}} | {{description}} |
{{/parameters}}
```

**条件块**：
```markdown
{{#has_examples}}
## 示例
{{examples}}
{{/has_examples}}
```

### 风格指南设计

**语言风格示例**：
```
- 使用主动语态："系统执行此操作" 而非 "此操作被系统执行"
- 使用简洁句式："返回用户列表" 而非 "该接口的主要功能是返回用户的完整列表"
- 术语统一：使用 "接口" 而非 "API"、"端点"、"接口地址"
```

## 与其他模式组合

- **Generator + Inversion**：先问清需求，再用模板生成
- **Generator + ToRapper**：加载领域知识后，按领域模板生成
- **Generator + Reviewer**：生成后自动审查是否符合模板
