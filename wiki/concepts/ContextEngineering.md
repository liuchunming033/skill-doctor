## 定义
上下文工程（Context Engineering）是 Prompt Engineering 的超集——后者关注"如何与 LLM 沟通"，前者在此基础上更关注"如何为 LLM 提供合适、有效的信息"。核心挑战：**给 LLM 的上下文内容，太长或太短都会出问题**。

## 为什么长上下文会失效

LLM 的底层 Transformer 注意力机制在长上下文中存在三个局限性：

| 局限 | 机制 | 影响 |
|------|------|------|
| **注意力权重稀释** | 所有权重之和为1，无关信息瓜分关键信息的注意力 | 重要信息被淹没 |
| **缓存局限** | KV 缓存大小=上下文窗口，FIFO 淘汰 | 早期信息直接被丢弃 |
| **位置编码模糊** | 超长上下文中 RoPE 等编码丧失区分度 | 相近位置的 Token 难以区分 |

## 四种上下文失效场景

| 场景 | 表现 | 根因 |
|------|------|------|
| **上下文冲突** | 包含矛盾信息，LLM 输出混乱或偏向某方 | 近因效应（关注尾部）vs 首因效应（关注头部）的注意力竞争 |
| **上下文混淆** | 张冠李戴，将实体A的属性安在实体B上 | 注意力泛化+位置编码模糊+缺乏实体追踪 |
| **上下文干扰** | 无关信息影响当前表现 | 注意力稀释+激活值传播+指令遵循脆弱性 |
| **上下文丢失** | 忘记窗口开头内容 | 缓存FIFO淘汰+窗口内注意力权重随位置衰减 |

## 工程应对策略

| 策略 | 具体做法 |
|------|---------|
| **逻辑拆分** | 将复杂任务拆成多次 LLM 调用，降低单次任务复杂度（如将十几种根因归类从一次调用拆为每种根因单独调用） |
| **内容精简** | Prompt 控制在 5w token 以内；重点规则放 Prompt 头部（利用首因效应）；去除冗余字段（ID/时间戳/状态等）；长 ID 映射为短编号 |
| **强制性约束** | 明确输出格式要求（纯净 JSON、禁止代码块标识符、禁止非标准格式） |
| **Few-Shot** | 添加示例引导 LLM 激活推理模式，作为"触发器"而非教学 |
| **系统指令前置** | 最重要规则放 Prompt 最前面——首因效应是"规则和背景"的守护者 |

> **核心原则**：在合适的场景，给 LLM 提供合适的内容。降低 Token 量 ≠ 降低信息量——去冗余、做压缩、拆逻辑。
