## 定义

Skill 生态运营是指 Claude Code 核心开发者总结的管理几百个 Skills 的三个关键机制：分发策略（按团队规模选择代码库或市场模式）、市场管理（试验区筛选而非审核委员会）、依赖关系（轻量协作而非重依赖管理）、度量与数据驱动优化（pre-use hook 记录使用情况）。核心思想：先把反馈回路搭起来，先让 Skill 进入真实使用，再在反馈里慢慢长出来。

## 四个维度

| 维度 | 做法 |
|------|------|
| **分发** | 按团队规模选方式（小→代码库，大→市场） |
| **筛选** | 不靠权威拍脑袋，先看有没有人持续在用 |
| **优化** | 不靠感觉，靠数据 |
| **迭代** | 先进入真实使用，再在反馈里慢慢长出来 |

## 一、分发策略

### 两种方式

#### 方式一：放进代码库

**适用场景**：小团队

**做法**：
- Skills 放进代码库，放在 `.cod/skills` 目录
- 团队成员拉下代码就能一起用

**优势**：简单省事，协作范围集中。

#### 方式二：做成插件进市场

**适用场景**：大规模团队

**做法**：
- 做成插件放进内部市场
- 谁需要谁装

**优势**：
- 系统轻，避免全包负担
- 团队大时，塞进仓库的每个 Skill 都给系统加负担
- 大家明明只需一小部分却要背整包
- 市场模式让团队自己决定装哪些

### 选择原则

- **小团队** → 放进代码库（最省事）
- **大规模** → 做成市场（避免负担）

## 二、市场管理机制

### 不靠审核委员会

**Claude 的办法**：先放到试验区跑一跑再看结果。

### 筛选流程

1. **上传**：做了 Skill → 传到 GitHub 沙盒目录
2. **宣传**：去 Slack 或论坛告诉大家可以试
3. **观察**：看有没有人真的用起来，有没有持续反馈
4. **正式**：Skill 有人持续在用且确实有价值 → 作者提 PR 挪进正式市场

### 关键判断

**Skill 做出来容易，做烂也容易**。

没有筛选机制的问题：
- 市场会堆满重复、模糊、没人维护的垃圾 Skill
- 真正好用的东西反而会被埋掉

**筛选原则**：不靠权威拍脑袋，先看有没有人持续在用。

## 三、Skill 之间的依赖关系

### 问题示例

- 一个 Skill 负责上传文件
- 另一个 Skill 生成 CSV 并上传
- 后者依赖前者

### 轻量协作而非重依赖管理

**不做的**：
- 不给一套很重的依赖管理系统
- 不声明版本
- 不处理装配关系
- 不处理一堆配置

**做的**：
- 在 Skill 里写清楚"如果要上传文件，就调用哪个 Skill"
- 只要那个 Skill 已装好，Claude 自己去调用

### 设计哲学

**简单**：Skills 体系还在快速变化，先让东西能接起来、能跑起来，比上复杂机制更重要。

## 四、度量与数据驱动优化

### 问题

Skill 做出来后，不知道：
- 有没有人用？
- 什么时候用？
- 哪些该被调用的场景却没有被调用？

### 解决方案：pre-use hook

**机制**：调用 Skill 前触发，记录：
- 时间
- 用户
- Skill 名称
- 参数

### 能回答的关键问题

- 哪个 Skill 最受欢迎？
- 哪个 Skill 明明应该常用，但实际很少被触发？
- 还有哪些 Skill 基本没人碰，可能该下线了？

### 价值

**真正被记录的不只是使用数据，更重要的是团队有了判断依据**。

- 以前：凭感觉说这个 Skill 应该很有用
- 现在：能看到真实情况，决定是继续优化 description、补文档，还是删除

## 核心思想

**先把反馈回路搭起来，不是先追求把系统设计到完美。**

Claude 内部很多 Skill 一开始也就几行——说明加一个踩坑点，然后边用边补。

**重点不是先规划出完美版本再发布，重点是先让它进入真实使用，再在反馈里慢慢长出来。**

**真正重要的从来都不是起步时有多完整，关键是你有没有办法让好 Skill 被看见、被用起来、再被持续改进好。**
