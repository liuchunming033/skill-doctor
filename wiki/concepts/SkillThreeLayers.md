## 定义

Skill 三层是指 Claude Code 核心开发者总结的 Skill 构建的三个递进层次：内容层（被看见和被理解）、结构层（可复用和不僵化）、高级技术层（有状态和可组合）。三个层次解决三个递进问题，对应 9 个最佳实践。

## 三层架构

| 层次 | 问题 | 解决方案 | 实践条数 |
|------|------|---------|---------|
| **内容层** | 被看见和被理解 | description/不重复显而易见/Gotchas | 3条 |
| **结构层** | 可复用和不僵化 | 文件系统/设置流程/避免过度约束 | 3条 |
| **高级技术层** | 有状态和可组合 | 内存与数据存储/脚本代码/按需 Hooks | 3条 |

## 第一层：内容层（被看见和被理解）

### 核心问题

**Claude 能不能找到你的 Skill？找到之后能不能理解该做什么？**

### 三个实践

#### 实践 1：description 字段写触发条件

**为什么 Skill 不被调用？**
- 原因：把 description 写成了摘要，而不是触发时机
- Claude 启动会话时扫描所有 Skill 的 description 来决定这个请求有没有对应 Skill
- description 需要的是**触发条件**，不是功能说明

**Description 决定的是什么时候该用它，这是 Skill 能不能被调用的第一道门槛。**

#### 实践 2：不要陈述显而易见的事情

**Claude 已经知道大量通用知识**，写 Skill 时最大的浪费就是重复它已经知道的东西。

**Skill 的价值在于补充 Claude 默认不知道的上下文、偏好和坑。**

**Anthropic 案例**：
- frontend-design Skill：不教 Claude 怎么写 React，只告诉它——别再用 Inter 和紫色渐变，客户不喜欢

#### 实践 3：构建 Gotchas 部分

**任何 Skill 里信号最强的内容往往不是教程，而是 Gotchas（坑点）**。
- 教程 Claude 本来就知道很多
- 但坑只有你们团队真的踩过

**真正值钱的是告诉它哪里别踩。**

### 内容层诊断

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 不被调用 | description 写成摘要 | 写触发条件（实践1） |
| 调用后不理解 | 重复已知知识 | 补充未知信息（实践2-3） |

## 第二层：结构层（可复用和不僵化）

### 核心问题

**Skill 能不能在不同场景下复用？会不会因为写得太死而失去灵活性？**

### 三个实践

#### 实践 4：使用文件系统和渐进式披露

**Skill 是文件夹，不是文件**。
- 可以放脚本、模板、数据文件
- 主文件里只写核心逻辑
- 告诉 Claude 详细文档在 `references.md`，需要的时候自己去读
- 按需加载才能最大化利用上下文窗口

#### 实践 5：仔细考虑设置流程

**让 Skill 从无状态工具变成有状态助手**。
- 用 `config.json` 存储配置
- 首次运行时询问，后续直接使用
- 用户只需要配置一次，之后每次调用都能直接用

#### 实践 6：避免过度约束

**约束目标，而不是约束路径**。
- 不要写固定步骤（先 test →再 lint →最后 build）
- 要写目标和灵活性（确保代码通过测试、规范构建，并根据情况调整顺序）
- 给 Claude 它需要的信息，但也要给它适应不同情况的灵活性

### 结构层诊断

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 无法复用 | Skill 写成单一文件 | 使用文件系统（实践4） |
| 僵化不灵活 | 步骤写太死 | 避免过度约束（实践6） |
| 缺少配置 | 每次都要配置 | 仔细考虑设置流程（实践5） |

## 第三层：高级技术层（有状态和可组合）

### 核心问题

**Skill 能不能记住上次做了什么？能不能直接给 Claude 可调用的代码？能不能在需要的时候开启护栏？**

### 三个实践

#### 实践 7：内存与数据存储

**Skill 不再是无状态的脚本，而是有记忆的助手**。
- 让 Skill 写日志——每次运行后追加一条记录
- 下次运行时读取日志，只报告新的变化
- Skill 记得上次做了什么

#### 实践 8：存储脚本并生成代码

**给 Claude 最好的工具不是更多文档，而是可调用的代码**。
- 把稳定能力封成脚本和辅助函数
- 让 Claude 负责组合，而不是重造轮子
- 别让 Claude 每次都从头写样板代码

#### 实践 9：按需 Hooks

**规则不一直开启但特定场景下需要**。
- Anthropic 案例：`careful` Skill 阻止危险命令（`rm -rf`、`drop table`）
- 调用 careful 时注册一个 hook，只在当前会话有效，会话结束后自动失效

### 高级技术层诊断

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 不记得上次 | 无状态运行 | 内存与数据存储（实践7） |
| 重造轮子 | 每次从头写代码 | 存储脚本（实践8） |
| 规则太死 | 规则一直开启 | 按需 Hooks（实践9） |

## 三层递进关系

### 层次递进

```
第一层（基础）：被看见和被理解
  ↓ Claude 能找到并理解 Skill
第二层（进阶）：可复用和不僵化
  ↓ Skill 在不同场景复用，保持灵活性
第三层（高级）：有状态和可组合
  ↓ Skill 有记忆、有代码、有护栏
```

### 问题诊断流程

**Skill 不被调用 → 检查第一层**：
- description 是否写触发条件
- 是否写了 Claude 已知知识
- 是否构建 Gotchas 部分

**Skill 效果差 → 检查第二层**：
- Skill 是否使用文件系统
- 步骤是否写得太死
- 是否考虑设置流程

**Skill 缺少高级能力 → 检查第三层**：
- Skill 是否有记忆
- 是否提供可调用代码
- 是否有按需 Hooks

## 核心思想

**根本原因**：把 Skill 当成 Prompt 来写。

**正确做法**：好的 Skill 不是告诉 Claude 每一步做什么，而是给它组合的能力，让它在执行时自己决定怎么做。

三层架构确保：
- 第一层：Skill 能被找到和理解
- 第二层：Skill 可复用且不僵化
- 第三层：Skill 有状态且可组合
