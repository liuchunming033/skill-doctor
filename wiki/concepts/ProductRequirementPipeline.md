## 定义

AI 驱动的产品需求流水线是一套将产品需求从 Issue 提出到评审就绪的全自动化工作流：基于统一 RID 追踪，依靠三大标准化 Agent 技能（Spec 生成 → 技术方案 → 测试用例）串行处理，以飞书项目为流程枢纽、以 Obsidian Wiki 为知识底座，人类仅在决策评审节点介入。

## 五节点流水线

```
Issue 提出 → Spec 生成 → 技术方案生成 → 测试用例生成 → 评审就绪
   (人类)      (SK-01)       (SK-02)          (SK-03)       (人类)
```

## 三大 Agent 技能

### SK-01-SPEC：需求规格文档自动生成

- 前置依赖：无（流水线入口）
- 核心能力：读取 wiki/business 全量知识，生成完整 PRD + API Spec
- 输出：含元数据（RID、模块、优先级）、业务流程、接口规范、数据模型的标准文档

### SK-02-TECH：代码适配型技术方案自动生成

- 前置依赖：SK-01（必须存在有效 Spec 产物）
- 核心能力：读取 wiki/repowiki 代码知识图谱，基于现有代码库生成可落地技术方案
- 强制约束：最小改动、复用优先，杜绝 AI 幻觉，明确区分新增/修改/复用/废弃

### SK-03-TEST：全维度测试用例自动生成

- 前置依赖：SK-01 + SK-02（两者均需存在有效产物）
- 核心能力：覆盖功能、边界、异常、性能、安全多维测试用例
- 输出：标准化测试用例集，可直接作为测试执行依据

## 自动依赖校验机制

当后置技能被调用时，自动校验前置产物：

- 前置不存在 → 自动先执行前置技能，再继续
- 前置存在但无效 → 提示补充 raw 素材，暂停生成

## RID 全链追踪

RID（飞书项目 Issue ID）= 全流程唯一索引。所有产物（Spec、技术方案、测试用例）均挂载于同一 RID，实现端到端可追溯。

## 质量的根本约束

"垃圾进，垃圾出"原则：

- 知识库含糊的需求描述 → 生成的 Spec 含糊
- 知识库缺少异常场景 → 测试用例覆盖率不足
- 知识库只有架构概述 → 技术方案只有壳

**产物质量上限 = ingest 质量上限。**
