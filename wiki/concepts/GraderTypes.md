## 定义
Agent 评估中用于打分的三类评分器（Grader），各有优劣，必须组合使用。选择优先级：**确定性评分器 > Rubric 评分器 > 人工评分器**。

## 三类评分器对比

### 1. 确定性评分器（Code-based Grader）
- **方法**：字符串匹配、二进制测试、静态分析、结果验证、工具调用验证、转录本分析
- **优势**：快、便宜、客观、可复现、易调试
- **劣势**：对有效但不符合预期模式的变体脆弱、缺乏细微判断力
- **适用场景**：所有"能用代码判断"的事——文件存在、编译通过、工具调用正确、Token 不超标

### 2. 模型评分器 / Rubric 评分器（Model-based Grader / LLM-as-Judge）
- **方法**：基于评分量表的打分、自然语言断言、成对比较、参考答案评估、多评委共识
- **优势**：灵活、可扩展、捕捉细微差别、处理开放式输出
- **劣势**：非确定性、比代码评分贵、需与人工评分校准
- **适用场景**："代码搞不定但能结构化描述"的事——回答质量、语气、推理合理性

### 3. 人工评分器（Human Grader）
- **方法**：领域专家审查、众包判断、抽样检查、A/B 测试
- **优势**：黄金标准质量、匹配专家用户判断
- **劣势**：贵、慢、通常需要规模化的专家资源
- **适用场景**：校准 LLM 评委（最核心用途）、诊断 0%/100% 异常、建立 Ground Truth、高风险兜底

## 评测方法细分

### 确定性评测的方法细节

**Rule-based 评测**（基于规则/数学公式，适用于有标准答案的任务）：
- **排序质量**：nDCG（Normalized Discounted Cumulative Gain）、MRR（Mean Reciprocal Rank）
- **分类/匹配**：Accuracy、Precision、Recall、F1-score
- 优点：稳定性高、可重复性强、便于横向比较
- 局限：难以评估开放式生成任务、对复杂语义理解能力有限

**语义相似度评测**（通过 Embedding 计算与参考答案的相似度）：
- BERTScore、Cosine Similarity、Semantic Textual Similarity
- 适用于问答、摘要生成、内容改写等任务

### LLM-as-Judge 的三种评测模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **单模型评分（Single Judge）** | 单个大模型按评分标准打分 | 常规质量评估 |
| **多模型投票（Multi-Judge Ensemble）** | 多个不同模型独立打分后投票/平均 | 降低单一模型偏差 |
| **成对比较（Pairwise Comparison）** | 对 A/B 两个输出做优劣判断 | 选型决策、迭代验证 |

### 人工评测的组织方式

- **专家评测**：领域专家评估，质量高但成本高、规模有限——适用于法律/医疗等专业场景
- **众包评测**：通过众包平台或标注团队大规模打分——成本较低、可扩展，需设计明确评分标准

## LLM 非确定性根源与多模型对抗

### 为什么同一 Prompt 同一 LLM 多次调用结果不同？

LLM 生成结果是自回归过程（逐 Token 生成），随机性来自两个关键参数：

- **Temperature（温度）**：在 Softmax 之前缩放原始分数（logits），低温度（0.1）放大高分词与低分词差距→更确定保守；高温度（1.5）缩小差距→概率分布更平缓，输出更多样但更不稳定
- **Top-P**：Softmax 之后按概率从高到低排序，只保留累计概率超过 P（如 0.9）的最小候选词集合，剔除极低概率词后重新归一化再采样

每一步的微小随机差异通过链式法则（P(全文) = P(x₁)·P(x₂|x₁)·...·P(xₙ|x₁,...,xₙ₋₁)）被指数级放大，导致最终结果显著不同。

### 为什么同一 Prompt 不同 LLM 结果不同？

- **分词器（Tokenizer）差异**：不同模型对同一文本的 Token 拆分不同（如 "unstable"→"un"+"stable" vs "unstable"），导致初始输入序列不同
- **训练参数差异**：千亿/万亿级参数从不同数据、不同训练过程学到不同的 logits 计算模式
- **指令微调（RLHF）差异**：不同模型的偏好对齐千差万别

### 多模型对抗方案

采用 2-3 个不同 LLM 执行相同 Prompt，对结果进行统计聚合（求均值/多数投票），显著降低单一模型偏差。代价是执行耗时增加，适用于对时间不敏感的长时评测任务。实践中双模型打分分值不同（如 30 vs 60）但远低于 BadCase 阈值，取均值 45 准确识别。

## 关键实践

- **校准**：LLM-as-Judge 需与人类专家紧密校准，一致率 ≥ 85% 才算可用
- **防作弊**：评分器应能抵抗绕过/黑客行为
- **部分得分**：对多组件任务应给部分得分，而非全或无
- **过程 vs 结果**：优先评估"Agent 产生了什么"，而非"Agent 走的具体路径"

## LLM-as-Judge 校准：五种偏差与对策

模型评分器存在五种系统性偏差，必须通过校准消除：

| 偏差类型 | 表现 | 解决方法 |
|---------|------|---------|
| **评分漂移** | 同一答案不同时间得分不同 | 温度设为 0，固定 seed |
| **标准不一致** | 对"好"的定义模糊 | 编写详细评分标准（rubric） |
| **位置偏见** | 偏好先出现的选项 | 随机调换答案顺序，多次采样 |
| **长度偏见** | 偏好更长的回答 | 明确字数要求，或标准化处理 |
| **自我提升** | 偏爱与自己风格相似的回答 | 使用不同模型家族作为评判者 |

**校准流程**：收集 100-500 条人工标注建立"黄金测试集" → 计算模型评分与人工评分的 Spearman 相关性 → 迭代优化评分标准直到相关性 > 0.8。

## 三层评估流水线（Evaluation Pipeline）

实际工程中，三类评分器按成本-质量梯度分层组合：

```
Layer 1: 代码评分器（过滤层）
  • 格式检查 → 失败直接淘汰
  • 工具调用验证 → 失败直接淘汰
  • 精确匹配项 → 自动计分
  处理量：100% 样本，成本几乎为零

Layer 2: 模型评分器（质量层）
  • 对通过 Layer 1 的样本进行语义质量评估
  • 多维度评分（准确性、完整性、安全性）
  • 置信度低 → 触发人工复核
  处理量：100% 样本，成本可控

Layer 3: 人类评分器（仲裁层）
  • 模型评分置信度低的边界案例
  • 定期抽检校准模型评分器（如每周 5%）
  • 新场景/新标准建立时的初始标注
  处理量：5-10% 样本，成本最高但必要
```

**核心设计原则**：
- **防御性分层**：先用最严格、最便宜的过滤器，逐步放宽
- **校准即信任**：模型评分器未经校准不可单独用于关键决策
- **人类是稀缺资源**：只用于建立标准和处理真正模糊的案例
- **持续监控**：三种评分器的分歧率是系统健康指标

执行口诀：**"能写规则的不写 prompt，能用 prompt 的不用人"**。
