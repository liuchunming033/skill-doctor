## 定义

Skill 评分器系统是 OpenAI 提出的两种评分器组合：确定性检查（Deterministic Check）抓底线，纯代码规则判断基础动作有无发生；评分细则检查（Rubric Check）抓质量，用 LLM 做裁判读仓库打分，但必须锁死输出格式。两种评分器分工明确，从一开始就设计成可以接进持续集成，变成持续运行的回归守门员。核心理念：收集的是评估数据而非审稿意见，评估数据是给系统用的，会持续积累。

## 两种评分器分工

| 评分器 | 职责 | 手段 | 回答的问题 |
|--------|------|------|-----------|
| **确定性检查** | 抓底线 | 纯代码规则 | 基础动作做了没有？ |
| **评分细则检查** | 抓质量 | LLM + 锁死输出格式 | 做出来的东西像不像你要的？ |

## 确定性检查（Deterministic Check）

### 定义

**纯代码规则，不用大模型**，写死判断逻辑，程序直接对着运行轨迹核对。

### 回答的问题

**基础动作做了没有？**

### 示例问题

- 有没有跑 `npm install`？
- 有没有创建 `package.json`？
- 命令顺序是否符合预期？

### 两个关键特性

| 特性 | 说明 |
|------|------|
| **确定性** | 同样行为每次判断结果一样，没有模糊地带 |
| **可调试** | 失败后打开记录文件能看到从哪一步开始偏——以前只知道任务翻车了，现在能看到从哪一步开始翻的 |

### 价值

给 Skill 装了**飞行记录仪**。

### 上限

能告诉动作发生了没有，**告诉不了做出来的东西是不是你想要的样子**。

## 评分细则检查（Rubric Check）

### 定义

**用 LLM 做裁判**，让它来读仓库打分。

### 适用场景

验收标准本来就不是硬规则：
- 代码结构够不够干净？
- 组件组织是否符团队约定？
- 样式配置是否按既定风格落地？

### 评分细则（Rubric）是什么

**一张拆解表**：把模糊质量标准拆成若干具体可独立判断的维度。

**例子**："代码质量好"拆成：
- 组件是否按功能分目录
- 是否有未使用的引用
- 样式类名是否遵循命名约定
- 配置项是否集中管理

**机制**：每个维度单独打分，最后汇总。

### 关键：输出格式必须被限定

**做法**：
- 提前写好模板，规定大模型必须输出哪些字段
- 字段：整体是否通过、总分、每个维度得分、为什么这么打分
- 大模型必须按模板填空，不能自由发挥

**为什么要严格限定格式？**

| 不限定格式的问题 | 限定格式的好处 |
|----------------|--------------|
| 只问"这段代码好不好"，给你散文式评论，每次措辞不同 | 可以把模型判断变成可被程序读取的数据 |
| 无法比较，无法进自动化流程 | 评估结果可以跨版本比较、长期追踪趋势 |
|  | 直接进持续集成流水线做自动判断 |

### 核心理念

**你收集的是评估数据，不是审稿意见。评估数据是给系统用的，会持续积累。**

## 组合设计

**这套组合从一开始就设计成可以接进持续集成的**。

一旦进了流水线：
- 不再依赖某个工程师记得手动去试
- 变成**持续运行的回归守门员**

### 顺序原则

**规则不够时，再让模型补上**：
- 先写确定性检查（纯代码规则）
- 确定性检查覆盖不了的，再用评分细则检查（LLM）
- 顺序不能反

## 扩展检查：Skill 成熟后的六类指标

前两步解决"对不对"的问题，Skill 真正进入生产后还需回答：**稳不稳？贵不贵？脏不脏？**

| 指标类型 | 检查内容 | 价值 | 频率 |
|---------|---------|------|------|
| **命令数和循环检测** | 执行多少条命令？以前6步现在18步？ | 行为退化最直接信号 | 每次 |
| **消耗量追踪** | 每次消耗多少 token？ | 防止 Skill 越写越胖，越贵越慢 | 每次 |
| **构建检查** | 做完后跑一次构建命令 | 最强端到端信号，成本低 | 默认开启 |
| **运行时冒烟检查** | 拉起开发服务器，发请求看响应 | 更慢更准 | 按风险加 |
| **仓库清洁度检查** | 任务完成后有无不该出现的文件？ | 任务做完不代表系统没被弄脏 | 每次 |
| **权限回归检查** | Skill 是否还能在最小权限下工作？ | 权限漂移在自动化后变成结构性风险 | 每次 |

### 扩展原则

**先加快速检查，再按风险叠加慢检查，不是一次全上，是按需扩展。**

## 实施步骤

### 第一步：记录运行轨迹

用 `codex --trace-json` 参数：
- Skill 每执行一步输出一条结构化记录
- 记录跑了什么命令、创建了什么文件
- 全部按时间顺序记下来

**价值**：
- 不是日志摘要，是完整的可被程序解析的运行轨迹
- 有了运行轨迹，评分器才有东西可以解析
- 没有运行轨迹，只能看最终产物，看不到过程

### 第二步：确定性检查

围绕运行轨迹写纯代码规则判断基础动作。

### 第三步：评分细则检查

确定性检查覆盖不了的，用 LLM + 锁死输出格式处理质量判断。

### 第四步：扩展检查

Skill 成熟后按需叠加六类指标。

## 五条原则

1. **量真正重要的东西**：不要为了有指标而有指标，先想清楚什么变差了你会真的在意
2. **先写清可检查的定义**：验收标准模糊，评估就没意义。先把"什么叫做对"写成可执行判断
3. **把评估扎在真实行为上**：记录 Skill 完整运行轨迹，围绕实际执行动作写确定性检查
4. **规则不够时，再让模型补上**：用评分细则加锁死格式处理规则覆盖不了的质量判断，顺序不能反
5. **让真实失败去驱动样本增长**：每次手动修复都是一个信号，把它变成测试用例，让 Skill 持续把这件事做对

## 核心思想

**真正成熟的 Skill 不只是能调用模型做事，它还得有一套评估系统，持续证明自己没有偏移方向。**
