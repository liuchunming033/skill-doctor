## 定义

Skill 四环节是指 Claude Code 核心团队总结的 Skill 能力地图的四个阶段：认知→生产→验证→交付。这四个环节沿着完整的研发流程排开，帮团队看清哪些能力已沉淀成系统，哪些关键经验还停留在人脑里。

## 四个环节详解

### 第一环节：认知

**核心任务**：让 Claude 先理解系统、接上数据、掌握排障入口。

**解决的问题**：Claude 到底知不知道自己面对的是什么系统？从哪里获得理解？遇到异常时从哪里开始查？

**Skill 类别**：
- 内部库与接口文档（补充组织内部专属知识）
- 数据获取与分析（解决数据入口问题）
- 故障排查手册（承接低频但认知负担极高的问题）

**价值**：把资深工程师的理解系统和排障思路变成组织可复用的基础设施。

### 第二环节：生产

**核心任务**：让 Claude 稳定地产出代码、流程结果和标准化产物。

**解决的问题**：Claude 怎么稳定地产出代码、流程结果和标准化产物，而不是每次从零起步。

**Skill 类别**：
- 代码脚手架与模板（给 Claude 一套稳定的起手式）
- 业务流程与团队自动化（处理高频重复、规则清楚的流程工作）

**价值**：稳定产出，避免每次从零起步，持续吃掉团队低价值、重复性劳动。

### 第三环节：验证

**核心任务**：让 Claude 不只是产出，还能做质量审查、完成产品验证。

**解决的问题**：Claude 能不能证明自己写出来的东西真的能工作。

**Skill 类别**：
- 代码质量与审查（检查代码、文档、方案质量）
- 产品验证（测试代码是否真的正确工作）

**关键技巧**：
- **Fresh eye review**：制造新鲜视角专门挑毛病，对抗模型的自我一致性偏差
- **录制视频**：让 Claude 录制输出视频，看到完整执行过程
- **编程式断言**：把断言放进 Skill，每一步强制检查状态

**价值**：验证类 Skill 值得工程师花一周时间打磨，真正决定质量的不只是生成能力，还有反馈闭环。让 Claude 有能力证明自己做对了，而不只是交差。

### 第四环节：交付

**核心任务**：把结果推进到代码合并、部署和后续运维。

**解决的问题**：Claude 怎么把结果真正推进到系统里，并在后续阶段持续维护。

**Skill 类别**：
- 持续集成与部署（负责代码合并、流水线部署、灰度回滚）
- 基础设施运维（处理基础设施和日常运维动作）

**问题诊断**：很多团队"一到部署就必须完全切回人工"，说明 Agent 能力链条只走到一半。

**特别强调**：基础设施运维涉及高风险动作（删资源、改环境、切流量），先把风险边界写清楚，再谈自动化。

## 四环节流程示意

```
认知阶段 → 生产阶段 → 验证阶段 → 交付阶段
   ↓          ↓          ↓          ↓
理解系统    稳定产出    质量验证    推进系统
接上数据    代码脚手架   代码审查    CI/CD
掌握排障    流程自动化   产品验证    基础设施运维
```

## 环节缺失诊断

**如何使用四环节诊断团队能力缺口**：

把自己团队现有的 Skill 摆在四环节地图上，很快就能看出来：

### 认知环节缺失

- **现象**：Claude 总是不理解系统结构，不知道数据在哪里
- **缺失 Skill**：内部库文档、数据分析、故障排查
- **后果**：每次都要人工解释系统，AI 无法自主理解

### 生产环节缺失

- **现象**：Claude 每次产出格式不一致，质量波动
- **缺失 Skill**：代码脚手架、流程自动化
- **后果**：产出不稳定，需要大量人工调整

### 验证环节缺失

- **现象**：Claude 产出后无法证明质量，需要人工验证
- **缺失 Skill**：代码审查、产品验证
- **后果**：缺少反馈闭环，质量不稳定，需要人工兜底

### 交付环节缺失

- **现象**："一到部署就必须完全切回人工"
- **缺失 Skill**：CI/CD、基础设施运维
- **后果**：Agent 能力链条只走到一半，无法闭环

## 核心思想

**团队落地 AI 研发时，真正的短板往往在于关键经验有没有被做成 Claude 能真正调用、执行、验证和复用的 Skill。**

四环节帮团队诊断短板：
- 哪些环节已经沉淀下来
- 哪些关键环节还停留在人脑里
- 哪些地方明明最需要 Skill，却还没被系统化
