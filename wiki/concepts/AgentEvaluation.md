## 定义
Agent 评估（Agent Evaluation / Eval）是对 AI Agent 输出质量的系统化评测。其核心公式为：**Eval = Agent 输入 → 执行 → 捕获执行过程（Trace + 产物） → 一组检查规则 → 可对比的分数**。

## 为什么 Agent 评估比传统 LLM 评估更难？

Agent 的自主性带来三个传统软件没有的独特挑战：

1. **非确定性**：同一 prompt 多次执行结果不同，"跑通一次"不代表"稳定能跑"
2. **黑盒化**：模型升级、Prompt 微调、工具链变化都可能导致行为漂移，肉眼难以察觉
3. **错误级联放大**：一次任务涉及几十步工具调用，前序步骤的小偏差沿链路逐级放大

## 评估的核心结构

根据 Anthropic 的定义：
- **任务（Task）**：单次测试，含定义好的输入和成功标准
- **试验（Trial）**：对同一任务的一次独立执行，因模型输出有方差，通常需多次试验
- **评分器（Grader）**：对 Agent 表现某方面打分的逻辑
- **转录本/轨迹（Transcript/Trace）**：一次试验的完整记录（输出、工具调用、推理、中间结果）
- **结果（Outcome）**：试验结束时的环境最终状态
- **评估框架（Eval Harness）**：端到端运行评估的基础设施
- **评估套件（Eval Suite）**：为衡量特定能力而设计的一组任务集合

## 核心难点：多轮评估中的错误传播累积

这是 Agent 评估区别于传统软件测试的**本质挑战**。

### 错误传播机制

```
第1轮：Agent 错误理解用户意图
    ↓
第2轮：基于错误理解，调用错误工具
    ↓
第3轮：工具返回异常/无关数据，Agent 尝试"修复"但方向错误
    ↓
第4轮：状态进一步混乱，Agent 进入循环或产生幻觉
    ↓
最终结果：完全偏离目标
```

### 四个关键洞察

| 问题 | 说明 |
|------|------|
| **错误 ≠ 失败** | 中间步骤出错后 Agent 可能自我纠正，最终成功——这是 Agent 能力的体现，不应简单判负 |
| **正确 ≠ 成功** | 中间步骤都对，但累积效率极低（如 20 轮完成本应 3 轮的任务），实际不可接受 |
| **路径依赖** | 早期微小差异（如工具返回顺序）可能导致完全不同的后续行为 |
| **评分器盲区** | 仅看最终结果的评分器会漏掉过程中的风险行为（如差点误删数据） |

### 应对策略

1. **过程评分（Process-based Grading）**：评分器分析完整转录本，而非仅最终输出
2. **里程碑检查（Milestone Checking）**：在关键中间步骤设置检查点
3. **错误恢复评分**：专门评估 Agent 从错误中恢复的能力
4. **对比基线**：与已知良好路径对比，检测偏离程度
5. **扰动测试（Perturbation）**：故意改变工具返回顺序/内容，观察鲁棒性

## 评估与其他方法的协同：Swiss Cheese Model

没有单一方法能捕获所有问题，多层组合才能互相补漏：

| 方法 | 阶段 | 作用 |
|------|------|------|
| **自动化评估** | 上线前 + CI/CD | 第一道防线，每次变更即运行 |
| **生产监控** | 上线后 | 检测分布漂移、未预料的真实世界故障 |
| **A/B 测试** | 流量充足后 | 验证重大变更的实际效果 |
| **用户反馈 + 对话抽样** | 持续进行 | 填补 gaps：分类反馈、每周抽样阅读 |
| **系统性人工研究** | 特定场景 | 校准 LLM 评分器、评估主观输出 |

## 五大评估维度（腾讯实践）

| 大类 | 核心问题 | 优先级 |
|------|---------|--------|
| 功能正确性 | 做对了吗？ | P0 |
| 过程质量 | 过程合理吗？ | P1 |
| 效率与成本 | 划算吗？ | P1 |
| 鲁棒性与安全 | 靠谱吗？ | P0 |
| 体验与对齐 | 好用吗？ | P2 |

## 能力评估 vs 回归评估

- **能力评估（Capability Eval）**："Agent 能把什么做好？"——起步通过率 20-50%，是"目标山峰"
- **回归评估（Regression Eval）**："原有能力是否还在？"——应接近 100% 通过率，是"防御漂移"
- 能力评估通过率稳定 = 100% 后"毕业"为回归评估，进入 CI 每次运行

## 从零到一的 8 步路线图（Anthropic）

0. 尽早开始（20-50 个简单任务即可起步）
1. 从手工测试转化
2. 写无歧义任务，附带参考解法
3. 构建平衡用例集（正例 + 负例）
4. 构建稳健的评估框架与环境隔离
5. 精心设计评分器
6. 读转录本！验证评估是否在测量真正重要的东西
7. 监控能力评估饱和
8. 通过开放贡献和维护保持评估套件健康
