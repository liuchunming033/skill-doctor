## 定义

Skill 评估框架是 OpenAI 提出的系统化评估 Skill 的方法论，核心思想：从靠体感维护换成靠最小证据闭环维护。框架包含：定义成功的四类目标（结果/过程/风格/效率）、识别两个维护面（执行体和触发边界）、找出三类隐藏假设失控点（触发/环境/执行假设）、构建回归样本集（四类样本覆盖 10~20 条 prompt）。让 Skill 从 prompt 技巧变成可被持续验证的工作流资产。

## 核心转变

**从靠体感维护换成靠最小证据闭环维护**。

### 问题背景

**每次改完都不知道这是优化还是回归**：
- 改了一版，跑了一下，看起来没出事，继续改
- 靠体感，不是证据
- Skill 接入生产会触发工具、生成文件、留下副作用
- 改任何一个地方，行为边界都可能飘

### 转变价值

这个转变比任何工具细节都重要。

Skill 一旦接入生产，不再只是 prompt 技巧，开始变成要被持续验证的工作流资产。

**维护资产至少回答三件事**：
1. **你想让它做成什么？**（四类目标）
2. **它会在哪些地方失控？**（三类假设）
3. **你准备用哪些样本持续盯住这些失控点？**（四类样本）

## 框架组成

### 一、四类成功目标

| 目标类型 | 问题 | 为什么关键 |
|---------|------|-----------|
| **结果目标（Outcome）** | 任务完成？应用能跑？ | 最直觉，易被单独当全部 |
| **过程目标（Process）** | 调用 Skill？按预想步骤？ | 结果对不代表路径对，可能绕路下次绕不过去 |
| **风格目标（Style）** | 符合规范？风格一致？ | 不影响跑，但下游无法接受 |
| **效率目标（Efficiency）** | 有无浪费？烧 token？ | 单次不觉，生产大量触发放大浪费 |

### 二、两个独立维护面

| 维护面 | 内容 | 失控表现 |
|-------|------|---------|
| **执行体** | instructions、工具链、步骤 | 改 instructions，行为变 |
| **触发边界** | name 和 description | 改 description，触发范围飘 |

**问题**：只盯执行体，看不见触发边界腐烂。

### 三、三类隐藏假设失控点

| 假设类型 | 盲区 | 特点 |
|---------|------|------|
| **触发假设** | 哪些应触发没触发？哪些误触发？ | 触发边界问题 |
| **环境假设** | 默认运行前提？ | 最隐蔽，不在代码在上下文 |
| **执行假设** | 执行顺序是否正确？ | 每步对但顺序错，卡死或垃圾输出 |

### 四、四类回归样本

| 样本类型 | 说明 | 作用 |
|---------|------|------|
| **显式调用** | 直接点名 Skill | 保证改动后显式调用不坏 |
| **隐式调用** | 只描述场景 | 测试能否自己选中 |
| **带上下文调用** | 混入业务上下文 | 测试噪声环境识别执行 |
| **负控样本** | 不该触发的请求 | 验证知道何时不该做 |

**样本数量**：10~20 条 prompt 已足够高效，重点在能否卡住真实失败模式。

## 评估流程

```
第一步：定义成功标准
  ↓ 四类目标（结果/过程/风格/效率）
第二步：识别失控点
  ↓ 三类假设（触发/环境/执行）
第三步：构建样本集
  ↓ 四类样本（显式/隐式/上下文/负控）
第四步：持续验证
  ↓ 改动后用样本集验证
结果：证据闭环维护
```

## 核心思想

**从"会写 Skill"走向"会维护 Skill"**。

**完成的不是完整评估系统，而是更基础的事**：
- 把靠体感维护换成靠最小证据闭环维护
- 让 Skill 从 prompt 技巧变成工作流资产
- 建立持续验证的机制
