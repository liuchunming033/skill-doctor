## 核心摘要

Claude Code 核心开发者 SACC 强调：写 Skill 最关键步骤不是写，而是想清楚要写的到底是什么类型。四大 Skill 类型：触发式（由事件/条件自动触发）、工具式（由 Agent 在需要时自主调用）、流程式（封装完整工作流程）、偏好式（固化团队偏好规范）。三个问题判断类型：由谁触发？有没有固定流程？核心价值是什么？类型决定维护策略：触发式关注误触发率、工具式关注调用成功率、流程式关注完成率、偏好式关注规范一致性。关键：理解主要属于哪一类，按那一类的设计原则主导设计。

## 关键洞察

### 核心观点

**写 Skill 最关键的步骤不是写，而是想清楚你要写的到底是什么类型的 Skill。**

不同类型的 Skill，设计思路、评估方式、维护策略都不一样。如果一开始类型没想清楚，后面越写越别扭。

### 四大 Skill 类型

| 类型 | 定义 | 典型场景 | 设计要点 |
|------|------|---------|---------|
| **触发式 Skill**（Trigger） | 由事件/条件自动触发 | 代码审查（PR 创建时）、部署前检查（部署前）、日志监控（异常模式） | 触发条件精确避免误触发、明确反馈机制、考虑并发重入 |
| **工具式 Skill**（Tool） | Agent 在需要时自主调用 | 数据分析、PDF 处理、API 调用 | name/description 语义清晰、输入输出标准化、清晰错误处理 |
| **流程式 Skill**（Workflow） | 封装完整工作流程，有明确开始结束 | 站会汇报、事故响应、发布流程 | 流程步骤清晰不过度约束、明确输入输出、状态管理 |
| **偏好式 Skill**（Preference） | 固化团队偏好、规范、判断标准 | 代码风格、审查标准、报告格式 | 用 gotchas 写坑点、区分必须和建议、定期更新 |

### 触发式 Skill（Trigger Skill）

#### 定义

**当系统检测到特定事件或条件时自动触发**。

#### 典型场景

- 代码审查 Skill：每次 PR 创建时自动触发
- 部署前检查 Skill：每次部署前自动运行安全检查
- 日志监控 Skill：检测到异常模式时触发排查流程

#### 设计要点

- 触发条件要精确，避免误触发
- 执行结果要有明确的反馈机制
- 要考虑并发和重入问题

### 工具式 Skill（Tool Skill）

#### 定义

**提供特定能力，由 Agent 在需要时自主决定是否调用**。

#### 典型场景

- 数据分析 Skill：Agent 需要分析数据时调用
- PDF 处理 Skill：Agent 需要处理 PDF 时调用
- API 调用 Skill：Agent 需要与外部系统交互时调用

#### 设计要点

- name 和 description 要语义清晰，让 Agent 知道什么时候该用
- 输入输出要标准化
- 要有清晰的错误处理

### 流程式 Skill（Workflow Skill）

#### 定义

**封装一个完整的工作流程，有明确的开始和结束**。

#### 典型场景

- 站会汇报 Skill：汇总任务、代码、聊天记录，生成站会内容
- 事故响应 Skill：创建报告、同步状态、通知相关人员
- 发布流程 Skill：代码合并、构建、测试、部署、通知

#### 设计要点

- 流程步骤要清晰，但不要过度约束顺序
- 每个步骤要有明确的输入输出
- 要有状态管理（记住做到哪一步了）

### 偏好式 Skill（Preference Skill）

#### 定义

**固化团队的偏好、规范、判断标准**。

#### 典型场景

- 代码风格 Skill：规定命名规范、文件结构、组件组织方式
- 审查标准 Skill：规定审查优先级、安全边界、性能要求
- 报告格式 Skill：规定输出格式、内容结构、语气风格

#### 设计要点

- 用 gotchas 写坑点，比写教程更有价值
- 要区分"必须遵守"和"建议遵守"
- 要定期更新（团队偏好会变化）

### 如何判断 Skill 属于哪一类

**三个问题**：

#### 1. 这个 Skill 由谁触发？

- 由事件/条件自动触发 → **触发式**
- 由 Agent 自主决定 → **工具式**
- 由用户明确指令触发 → **流程式或工具式**

#### 2. 这个 Skill 有没有固定流程？

- 有固定流程，每次步骤基本一致 → **流程式**
- 没有固定流程，每次调用方式不同 → **工具式**
- 不是流程，是约束和规范 → **偏好式**

#### 3. 这个 Skill 的核心价值是什么？

- 自动化重复劳动 → **流程式**
- 补充 Agent 能力短板 → **工具式**
- 固化团队经验 → **偏好式**
- 自动化响应事件 → **触发式**

### 混合类型

很多 Skill 不是纯粹的单一类型，可以同时具备多种属性：

- **工具 + 偏好**：API 调用 Skill（工具式）+ 调用规范（偏好式）
- **流程 + 触发**：部署 Skill（流程式）+ 合并事件触发（触发式）
- **工具 + 流程**：数据分析 Skill（工具式）+ 固定分析流程（流程式）

**关键**：不是找到纯类型，而是**理解你的 Skill 主要属于哪一类，然后按那一类的设计原则去主导设计**。

### 类型决定维护策略

| 类型 | 维护重点 | 评估重点 |
|------|---------|---------|
| **触发式** | 触发条件精确性 | 误触发率、漏触发率 |
| **工具式** | 接口稳定性 | 调用成功率、响应质量 |
| **流程式** | 流程完整性 | 完成率、步骤偏差 |
| **偏好式** | 规范一致性 | 输出是否符合规范 |
