## 核心摘要

Software3.0 上下文知识库建设专项：面向 12 条产品线的企业级知识库落地计划，目标是实现"需求-设计-开发-测试"全流程资产结构化沉淀、可检索、可复用、可自动化流转。采用统一赋能（工作坊）+ 实战落地（建设大赛）+ 竞赛评优模式推进。

## 核心目标

1. 12条产品线落地产品、设计、技术、测试四维完整知识库
2. 标准化产出 SPEC、技术方案、测试用例三类核心研发产物
3. 识别建设痛点，为企业级知识中心积累需求和经验

## 知识筛选原则

一句话原则：**建设的内容是产品的决策和洞察，跳过操作和状态。**

两个核心筛选维度：

- **半衰期**：这个知识多久变一次？变了是否让旧知识变成误导？
- **合成价值**：LLM 跨多个来源整合这类知识，能产生对你有用的洞察吗？

## 四维知识体系

| 维度     | 核心价值                               | 半衰期             |
| -------- | -------------------------------------- | ------------------ |
| 产品知识 | 业务边界、历史决策、用户洞察           | 最长，合成价值最高 |
| 设计知识 | 设计决策背后的思考（非设计稿本身）     | 较长               |
| 技术知识 | 架构决策层，跳过实现细节               | 中等               |
| 测试知识 | 线上问题复盘、高频风险点、业务异常场景 | 持续积累           |

## 三大研发产物的真实限制

- **SPEC**：知识库越完整，生成越少需人工补填；主要工作是校正而不是创作
- **技术方案**：能生成架构框架和约束边界，细节仍需结合代码库 AGENTS.md/CLAUDE.md
- **测试用例**：线上问题复盘和高频风险点是最有价值的测试来源，也是最难靠人脑枚举的
