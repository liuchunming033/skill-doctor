## 核心摘要

Google总结五种Skill标准设计模式,解决Skill开发核心痛点(格式免费但内容拼实力):Tool Wrapper(工具封装模式,按需注入知识)、Generator(内容生成模式,固定输出结构)、Reviewer(审查评分模式,分离审查与检查规则)、Inversion(反向访谈模式,先问清需求再开工)、Pipeline(流水线校验模式,分步执行流程)。所有模式可自由组合,覆盖绝大多数智能体工作场景,是标准化可复用的Skill内容设计范式。

## 当前Skill开发核心痛点

### 格式标准化,无技术壁垒
- 主流AI工具(Claude Code/Gemini CLI/Cursor)全部适配统一Skill规范
- 目录/YAML/文件结构不用再反复调试,新手也能快速搭出标准框架
- 参考https://agentskills.io/clients

### 内容无规范,全靠经验
- 官方只给"打包规则",不给"内容设计规则"
- 绝大多数人写的Skill只是空有框架,逻辑混乱/执行不稳定/复用性极差
- 同质化外壳,差异化内核:看似标准的Skill文件,内部工作流程/逻辑架构/执行精度完全取决于内容设计

## 五大核心Skill设计模式

### 模式一:Tool Wrapper(工具封装模式)

**核心定位**:赋予智能体特定技术栈的专属专业知识,是最简单最基础的Skill设计模式

**核心特点**:
- 不用把各类框架/类库的使用规范硬编码写进系统提示词,而是统一打包进Skill
- 智能体只有在用到对应技术时才会动态加载专属规范文档,做到按需加载精准匹配
- 非常适合沉淀团队内部编码规范/框架最佳实践

**工作逻辑**:
SKILL.md监听用户指令中的技术关键词,一旦匹配自动读取references目录下的规范文件,将对应规则作为绝对标准执行

**示例:FastAPI开发技能**
- 专门用于FastAPI项目开发/代码审查/问题调试
- 内置完整编码规范,智能体在写代码审代码时自动加载规范
- 严格遵循对应语法/依赖注入/类型注解标准,输出统一专业的代码

### 模式二:Generator(内容生成模式)

**核心定位**:解决智能体输出不稳定/格式不统一的问题,实现标准化模板化内容生成

**核心特点**:
- 通过"模板+样式规范"彻底解决随机性极强问题
- 全程采用填空式标准化生成,零随机偏差
- 依赖两个专属目录:assets存放输出模板,references存放样式规范
- Skill本身不写具体内容,只负责统筹流程:加载模板→读取样式→补齐缺失信息→标准化输出

**适用场景**:API文档生成/提交信息标准化/项目架构模板搭建/技术报告输出

**示例:标准化技术报告生成技能**
智能体先加载样式规范,再读取报告模板,主动询问用户缺失的核心信息(主题/核心结论/受众/数据),补齐所有内容后严格按照模板生成完整规范的Markdown技术报告

### 模式三:Reviewer(审查评分模式)

**核心定位**:模块化自动化审查/分级评分,实现代码/文档的标准化质检

**核心特点**:
- 核心逻辑是"审查流程和审查标准分离"
- 不用在提示词里堆砌冗长的审查规则,而是把所有质检标准/检查条目统一放在references审查清单文件中
- Skill只负责固定的审查流程,替换不同的清单文件就能快速切换审查场景,适配性极强
- 支持按严重程度分级输出结果(error/warning/info),审查结果更清晰更专业

**适用场景**:代码PR审核/漏洞检测/代码风格质检/文档合规校验

**示例:Python代码审查技能**
智能体自动加载Python代码审查清单,逐行核对用户代码,标记问题行数/区分三级严重程度/不仅标注问题还解释原因并给出修复代码/最终输出包含总结/问题清单/评分/优化建议的标准化审查报告

### 模式四:Inversion(反向访谈模式)

**核心定位**:反转人机交互逻辑,先摸清需求再落地执行,杜绝智能体盲目输出

**核心特点**:
- 普通智能体通病:用户刚说半句话就急于猜测需求直接生成结果
- Inversion模式彻底改掉这个问题,让智能体从"执行者"变成"访谈者"
- 通过强制门禁指令,要求智能体必须按固定顺序逐轮提问,收集完整需求/约束条件后才能开始生成最终方案
- 中途绝不提前输出/绝不跳过提问环节

**适用场景**:项目规划/系统设计/需求梳理/定制化方案输出

**示例:软件项目规划技能**
智能体分三阶段结构化访谈:
- Phase 1问题发现:业务问题/用户群体/项目规模
- Phase 2技术约束:部署环境/技术栈/硬性约束
- Phase 3综合:收集完所有信息后才加载模板生成完整项目方案,最后主动征求用户反馈迭代优化

### 模式五:Pipeline(流水线校验模式)

**核心定位**:强约束多步骤可校验的复杂工作流,杜绝步骤遗漏流程跳过

**核心特点**:
- 针对复杂任务专门设计带"硬性检查门禁"的串行工作流
- 每一步都必须校验通过/用户确认后才能进入下一步
- 绝对不允许跳过步骤/简化流程
- 按需调用各类模板/规范清单,不同步骤加载不同资源
- 精准控制上下文内容,保证流程严谨的同时不浪费Token

**适用场景**:复杂文档生成/代码标准化改造/多步骤工程化任务

**示例:Python API文档流水线技能**
严格执行四步流水线:
- Step 1盘点:盘点项目所有公开接口并确认
- Step 2生成:按规范生成文档注释并逐一对齐
- Step 3组装:组装完整API文档
- Step 4检查:按质检清单自查修复
每一步都有校验门禁,必须合规通过才能进入下一环节

## 技能模式选型

核心判断从一个问题开始:「这个技能是否会产生输出?」,根据回答(是/否)分成两大分支:

### 输出型技能
技能会主动生成内容/输出:
1. **From a template? → Yes → Generator模式**:基于固定模板批量生成文档/内容
2. **From a template? → No → Tool Wrapper模式**:封装一个能直接产生输出的工具/API

### 非输出型技能
技能不直接生成输出:
1. **Does it evaluate existing input? → Yes → Reviewer模式**:评估/检查已有的输入内容
2. **Needs user input first? → Yes → Inversion模式**:需要先向用户提问获取信息再继续执行
3. **Has ordered steps? → Yes → Pipeline模式**:有明确先后顺序的多步骤工作流
4. **No to all → Tool Wrapper模式**:纯工具调用,不涉及复杂逻辑

### 快速选型指南
1. 需要适配特定技术栈/沉淀团队规范 → 选Tool Wrapper
2. 需要统一输出格式/标准化生成内容 → 选Generator
3. 需要自动化质检/审查/分级评分 → 选Reviewer
4. 需要梳理模糊需求/精准确认条件 → 选Inversion
5. 需要复杂多步骤/强校验/不可跳过的工作流 → 选Pipeline

## 所有模式支持自由组合

五大设计模式互不排斥可自由组合,这是高阶专业Skill的核心设计思路:
- 流水线(Pipeline)结尾可嵌套审查模式(Reviewer):实现"流程执行+最终自检"双重保障
- 生成模式(Generator)可搭配反向访谈模式(Inversion):先收集完整需求再标准化生成内容
- 所有模式都依托Skill渐进式披露机制,运行时只会加载当前步骤所需的内容,不浪费Token
