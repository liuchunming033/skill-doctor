## 核心摘要

Anthropic官方Skill编写最佳实践指南完整版,系统讲解核心原则(简洁至关重要/设置适当自由度/多模型测试)、Skill结构(YAML Frontmatter规范/命名约定/有效描述编写)、渐进式披露模式(三层加载机制/三种组织模式/避免深层嵌套)、工作流与反馈循环设计、内容指南、评估和迭代方法、应避免的反模式,以及包含可执行代码的Skill高级编写技巧。

## 核心原则

### 简洁至关重要
- 上下文窗口是共享资源,与系统提示/对话历史/其他Skill元数据/用户请求共享
- 元数据预加载,SKILL.md仅在相关时读取,每个token与对话历史竞争
- 默认假设Claude已非常智能,只添加Claude不具备的上下文
- 对每条信息审视:"Claude真的需要这个解释吗?""我能否假设Claude已经知道这一点?"

### 设置适当的自由度
根据任务的脆弱性和可变性匹配具体程度:
- **高自由度**:适用于多种方法都有效,决策取决于上下文,启发式方法指导处理方式
- **中等自由度**:适用于存在首选模式,允许一定程度的变化,配置会影响行为
- **低自由度**:适用于操作脆弱且容易出错,一致性至关重要,必须遵循特定顺序

类比:将Claude想象成探索路径的机器人
- 两侧都是悬崖的狭窄桥梁:只有一条安全道路,提供具体防护措施和精确指令
- 没有危险的开阔地带:许多路径都能通向成功,给出大致方向相信Claude能找到最佳路线

### 多模型测试
Skill作为模型的补充,有效性取决于底层模型:
- **Claude Haiku**:快速、经济,测试Skill是否提供了足够的指导
- **Claude Sonnet**:平衡,测试Skill是否清晰高效
- **Claude Opus**:强大推理能力,测试Skill是否避免了过度解释

对Opus完美的可能需要为Haiku提供更多细节,如计划在多个模型中使用,力求编写对所有模型都适用的指令

## Skill结构

### YAML Frontmatter要求
- **name字段**:最多64字符,只能包含小写字母/数字/连字符,不能包含XML标签,不能包含保留词(anthropic/claude)
- **description字段**:必须非空,最多1024字符,不能包含XML标签,应描述Skill的功能以及何时使用它

### 命名约定
使用一致的命名模式,考虑动名词形式(动词+-ing):
- **良好示例**:processing-pdfs,analyzing-spreadsheets,managing-databases,testing-code,writing-documentation
- **可接受**:名词短语(pdf-processing),动作导向(process-pdfs)
- **避免**:模糊名称(helper/utils/tools),过于通用(documents/data/files),保留词,不一致模式

### 编写有效的描述
- 必须使用第三人称编写,描述会被注入到系统提示中
- 具体明确并包含关键术语,同时包含Skill的功能和何时使用的具体触发条件/上下文
- 每个Skill只有一个描述字段,对于Skill选择至关重要:Claude使用它从可能超过100个可用Skill中选择正确的Skill
- 有效示例:包含功能描述和触发场景(如PDF文件/表单/文档提取关键词)

## 渐进式披露模式

### 可视化概览:从简单到复杂
- 基本Skill:仅包含元数据和指令的SKILL.md文件
- 增长Skill:打包额外的内容,Claude仅在需要时加载(参考文件/示例/脚本)

### 三层加载机制详解
1. **Level 1:YAML元数据层**
   - 文件位置:SKILL.md顶部YAML frontmatter
   - 内容:技能的name和description
   - 加载方式:始终加载
   - Token消耗:约100 token
   - 作用:告诉模型有哪些技能和它们做什么

2. **Level 2:正文内容层**
   - 文件位置:SKILL.md的Markdown正文部分
   - 内容:技能的核心指令与SOP逻辑
   - 加载方式:在技能被触发时加载
   - Token消耗:平均小于5000 token
   - 作用:当Claude决定使用这个技能时才真正打开这部分内容

3. **Level 3及以上:附属文件层**
   - 文件位置:技能文件夹中除SKILL.md外的其他文件
   - 内容:markdown文档(forms.md/reference.md)/配置文件(.json)/脚本(.py)
   - 加载方式:按需加载
   - Token上限:理论上无限
   - 作用:Claude只有在SOP指令中被要求访问这些文件时才会读取或执行

### 三种组织模式
1. **模式1:带引用的高级指南**
   - SKILL.md作为概述,根据需要将Claude指向详细材料
   - Claude仅在需要时加载FORMS.md/REFERENCE.md/EXAMPLES.md

2. **模式2:按领域组织**
   - 对于涉及多个领域的Skill,按领域组织内容避免加载不相关的上下文
   - 当用户询问销售指标时,Claude只需读取销售相关架构而不是财务或营销数据

3. **模式3:条件性详情**
   - 显示基本内容,链接到高级内容
   - Claude仅在用户需要这些功能时才读取高级文件

### 避免深层嵌套引用
- 当文件从其他被引用的文件中被引用时,Claude可能会部分读取文件
- 遇到嵌套引用时,Claude可能使用`head -100`等命令预览内容而不是读取整个文件
- **保持引用距SKILL.md仅一层深度**,所有参考文件都应直接从SKILL.md链接
- 为超过100行的参考文件添加目录结构,确保即使在部分读取预览时Claude也能看到可用信息的完整范围

## 工作流和反馈循环

### 对复杂任务使用工作流
- 将复杂操作分解为清晰的顺序步骤
- 对于特别复杂的工作流,提供检查清单,Claude可以将其复制到响应中并在进展过程中逐项勾选
- 清晰的步骤防止Claude跳过关键验证
- 检查清单帮助Claude和您跟踪多步骤工作流的进度

### 实现反馈循环
- **常见模式**:运行验证器→修复错误→重复,可大大提高输出质量
- **示例1:风格指南合规性**(不含代码的Skill)
  - 验证器是STYLE_GUIDE.md,Claude通过读取和比较来执行检查
  - 草稿→审查清单→发现问题→修订内容→再次审查→继续
- **示例2:文档编辑流程**(含代码的Skill)
  - 编辑→立即验证→失败则查看错误→修复→再次验证→继续→重建→测试输出
  - 验证循环可及早发现错误

## 内容指南

### 避免时效性信息
- 不要包含会过时的信息
- 使用"旧模式"部分提供历史背景而不使主要内容变得杂乱

### 使用一致的术语
- 选择一个术语并在整个Skill中使用它
- 一致性有助于Claude理解和遵循指令
- 良好:始终使用"API endpoint"/"field"/"extract"
- 不良:混用"API endpoint"/"URL"/"API route"/"path"

## 常见模式

### 模板模式
为输出格式提供模板,根据需求匹配严格程度:
- **严格要求**:API响应或数据格式,使用"ALWAYS use this exact template structure"
- **灵活指导**:当适应性有用时,使用"sensible default format, but use your best judgment"

### 示例模式
对于输出质量依赖于查看示例的Skill,提供输入/输出对:
- 像常规提示一样提供示例
- 示例能帮助Claude更清楚地理解所需的风格和详细程度

### 条件工作流模式
引导Claude通过决策点:
- 确定修改类型→创建新内容/编辑现有内容
- 每种类型对应不同工作流
- 如果工作流变得庞大或复杂,考虑将它们放入单独文件并告诉Claude根据当前任务读取相应文件

## 评估和迭代

### 首先构建评估
- 在编写大量文档之前创建评估,确保Skill解决的是真实问题
- **评估驱动的开发**:
  1. 识别差距:在没有Skill的情况下让Claude执行代表性任务,记录具体的失败或缺失的上下文
  2. 创建评估:构建三个测试这些差距的场景
  3. 建立基线:测量Claude在没有Skill的情况下的表现
  4. 编写最少的指令:创建刚好足够的内容来解决差距并通过评估
  5. 迭代:执行评估,与基线比较,并进行优化

### 与Claude一起迭代开发Skill
最有效的Skill开发过程涉及Claude本身:
- 与Claude A合作创建Skill,Claude B在实际任务中测试
- Claude A帮助设计和完善指令,Claude B通过实际使用揭示差距

**创建新Skill**:
1. 在没有Skill的情况下完成任务:注意反复提供的信息
2. 识别可重用模式:哪些上下文对未来类似任务有用
3. 请Claude A创建Skill:它原生理解Skill格式和结构
4. 审查简洁性:检查是否有不必要的解释
5. 改进信息架构:更有效地组织内容
6. 在类似任务上测试:Claude B测试该Skill
7. 基于观察进行迭代:带着具体信息返回Claude A改进

**迭代现有Skill**:
- 与Claude A合作→使用Claude B测试→观察Claude B的行为→将洞察带回Claude A

### 观察 Claude如何浏览Skill
在迭代Skill时,注意Claude在实践中如何实际使用它们:
- **意外的探索路径**:以未预料的顺序读取文件,可能表明结构不如想象那样直观
- **遗漏的连接**:未能跟随对重要文件的引用,链接可能需要更明确或更突出
- **过度依赖某些部分**:反复读取同一个文件,考虑该内容是否应放在主SKILL.md中
- **被忽略的内容**:从未访问某个打包文件,可能不必要或在主指令中标示不清

## 应避免的反模式

### 避免 Windows 风格的路径
- 始终在文件路径中使用正斜杠,即使在 Windows 上也是如此
- Unix 风格的路径在所有平台上都有效,而 Windows 风格的路径在 Unix 系统上会导致错误

### 避免提供过多选项
- 除非必要,否则不要提供多种方法
- 提供默认方案并给出逃生出口

## 高级:包含可执行代码的 Skill

### 解决问题而非推卸
- 处理错误条件而不是推卸给 Claude
- 配置参数应有充分理由并加以记录,避免"巫术常量"

### 提供实用脚本
- 比生成代码更可靠,节省 token 和时间,确保跨使用的一致性
- **重要区别**:
  - 执行脚本(最常见):"运行 analyze_form.py 以提取字段"
  - 作为参考阅读(用于复杂逻辑):"参见 analyze_form.py 了解字段提取算法"

### 创建可验证的中间输出
- "计划-验证-执行"模式通过让 Claude 首先以结构化格式创建计划,然后在执行之前用脚本验证该计划,从而及早发现错误
- 工作流变为:分析→创建计划文件→验证计划→执行→验证

### 包依赖
- **claude.ai**:可以从 npm 和 PyPI 安装包,并从 GitHub 仓库拉取
- **Claude API**:没有网络访问权限,也无法在运行时安装包
- 在 SKILL.md 中列出所需的包,并在代码执行工具文档中验证它们是否可用

### 运行时环境
Skill 在具有文件系统访问、bash 命令和代码执行能力的代码执行环境中运行:
1. **元数据预加载**:在启动时,所有 Skill 的 YAML frontmatter 中的名称和描述会被加载到系统提示中
2. **按需读取文件**:Claude 在需要时使用 bash Read 工具从文件系统访问 SKILL.md 和其他文件
3. **高效执行脚本**:实用脚本可以通过 bash 执行,而无需将其完整内容加载到上下文中,只有脚本的输出会消耗 token
4. **大文件无上下文开销**:参考文件、数据或文档在实际读取之前不会消耗上下文 token

文件路径很重要:
- 使用正斜杠(reference/guide.md),而不是反斜杠
- 描述性地命名文件:使用指示内容的名称
- 为发现而组织:按领域或功能构建目录
- 打包全面的资源:包含完整的 API 文档、大量示例、大型数据集
- 对确定性操作优先使用脚本:编写 validate_form.py而不是要求 Claude 生成验证代码

### MCP 工具引用
如果 Skill 使用 MCP 工具,请始终使用完全限定的工具名称以避免"未找到工具"错误:
- 格式:`ServerName:tool_name`
- 示例:BigQuery:bigquery_schema,GitHub:create_issue

### 避免假设工具已安装
不要假设包可用:
- 不良示例:"Use the pdf library to process the file."
- 良好示例:"Install required package: pip install pypdf. Then use it..."

## 高效 Skill 检查清单

### 核心质量
- 描述具体且包含关键术语
- 描述同时包含 Skill 的功能和何时使用它
- SKILL.md 正文在 500 行以内
- 额外详情在单独的文件中(如果需要)
- 无时效性信息(或在"旧模式"部分中)
- 全文术语一致
- 示例具体而非抽象
- 文件引用仅一层深度
- 适当使用渐进式披露
- 工作流有清晰的步骤

### 代码和脚本
- 脚本解决问题而不是推卸给 Claude
- 错误处理明确且有帮助
- 没有"巫术常量"(所有值都有理由)
- 所需包在指令中列出并验证为可用
- 脚本有清晰的文档
- 没有 Windows 风格的路径(全部使用正斜杠)
- 关键操作有验证/核实步骤
- 质量关键任务包含反馈循环

### 测试
- 至少创建了三个评估
- 使用 Haiku、Sonnet 和 Opus 进行了测试
- 使用真实使用场景进行了测试
- 整合了团队反馈(如适用)
