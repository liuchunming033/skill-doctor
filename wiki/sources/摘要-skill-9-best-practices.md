## 核心摘要

Claude Code 核心开发者总结构建 Skill 的 9 个最佳实践，解决三个递进问题：内容层（被看见和被理解）、结构层（可复用和不僵化）、高级技术层（有状态和可组合）。内容层：description 写触发条件、不陈述显而易见的事、构建 Gotchas 部分。结构层：使用文件系统和渐进式披露、仔细考虑设置流程、避免过度约束。高级技术层：内存与数据存储、存储脚本并生成代码、按需 Hooks。核心思想：好的 Skill 不是告诉 Claude 每一步做什么，而是给它组合的能力，让它在执行时自己决定怎么做。

## 关键洞察

### 问题诊断

**为什么 Skill 不被调用？**
- 把 description 写成了摘要，而不是触发时机

**为什么调用后效果差？**
- 写了一堆 Claude 已经知道的东西
- 步骤写得太死，失去灵活性

**根本原因**：把 Skill 当成 Prompt 来写。

### 三个递进问题

| 层次 | 问题 | 解决方案 |
|------|------|---------|
| 内容层 | 被看见和被理解 | description/不重复显而易见/Gotchas |
| 结构层 | 可复用和不僵化 | 文件系统/设置流程/避免过度约束 |
| 高级技术层 | 有状态和可组合 | 内存与数据存储/脚本代码/按需 Hooks |

## 内容层：被看见和被理解

### 第一条：description 字段是给模型看的

**错误写法**：
```
description: "这是一个监控 PR 的工具"
```

**正确写法**：
```
description: "当用户说监控 PR 时触发"
```

**原理**：
- Claude 启动会话时会扫描所有 Skill 的 description 来决定这个请求有没有对应 Skill
- description 需要的不是功能说明，而是**触发条件**
- **Description 决定的是什么时候该用它，这是 Skill 能不能被调用的第一道门槛**

### 第二条：不要陈述显而易见的事情

**Claude 已经知道大量通用知识**，写 Skill 时最大的浪费就是重复它已经知道的东西。

**Anthropic 内部案例**：
- 问题：Claude 生成的前端界面总是用 Inter 字体和紫色渐变，看起来很"AI味"，客户反馈不喜欢
- Skill：不教 Claude 怎么写 React（Claude 本来就会），只告诉一件事——**别再用 Inter 和紫色渐变了，客户不喜欢**
- 价值：补充 Claude 默认不知道的上下文、偏好和坑

### 第三条：构建 Gotchas 部分

**核心发现**：任何 Skill 里信号最强的内容往往不是教程，而是 **Gotchas**（坑点）。
- 教程 Claude 本来就知道很多
- 但坑只有你们团队真的踩过

**真正值钱的是告诉它哪里别踩**。理想情况下，应该随着时间不断更新 Skill，把新踩的坑也加进去。

## 结构层：可复用和不僵化

### 第四条：使用文件系统和渐进式披露

**错误做法**：把 Skill 写成一个巨大的 Markdown 文件，所有内容都塞在里面。

**正确做法**：
- **Skill 是文件夹，不是文件**
- 可以放脚本、模板、数据文件
- 主文件里只写核心逻辑
- 告诉 Claude 详细文档在 `references.md`，需要的时候自己去读
- 按需加载才能最大化利用上下文窗口

### 第五条：仔细考虑设置流程

**问题**：有些 Skill 需要用户配置（如站会要发布到哪个 Slack 频道），这个信息每个团队都不一样。

**解决方案**：
- 用 `config.json` 存储配置
- 首次运行时询问，后续直接使用

**价值**：让 Skill 从**无状态工具变成有状态助手**——用户只需要配置一次，之后每次调用都能直接用。

### 第六条：避免过度约束

**错误写法**：
```
"先运行 test，再运行 lint，最后运行 build"
```

**正确写法**：
```
"在部署前，确保代码通过测试、规范构建，并根据情况调整顺序。"
```

**核心原则**：**约束目标，而不是约束路径。** 给 Claude 它需要的信息，但也要给它适应不同情况的灵活性。

## 高级技术层：有状态和可组合

### 第七条：内存与数据存储

**问题**：Skill 每次运行都是全新的，不记得上次做了什么。

**解决方案**：
- 让 Skill 写日志——每次运行后追加一条记录
- 下次运行时读取日志，只报告新的变化

**价值**：Skill 不再是**无状态的脚本，而是有记忆的助手**。

### 第八条：存储脚本并生成代码

**问题**：让 Claude 每次都从头写样板代码。

**解决方案**：
- 把稳定能力封成脚本和辅助函数
- 让 Claude 负责组合，而不是重造轮子

**核心思想**：**给 Claude 最好的工具不是更多文档，而是可调用的代码。**

### 第九条：按需 Hooks

**场景**：有些规则太严格，不想一直开启，但特定场景下又需要。

**Anthropic 案例**：
- `careful` Skill 会阻止危险命令（如 `rm -rf`、`drop table`）
- 当调用 `careful` 时，它会注册一个 hook
- 只在当前会话有效，会话结束后自动失效

## 核心思想总结

**好的 Skill 不是告诉 Claude 每一步做什么，而是给它组合的能力，让它在执行时自己决定怎么做。**

### 问题与原因对照

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 不被调用 | description 写成摘要而非触发时机 | description 写触发条件 |
| 效果差 | 写了 Claude 已知的东西或步骤太死 | 补充未知信息+避免过度约束 |
| 根本原因 | 把 Skill 当成 Prompt 来写 | 给组合能力而非固定步骤 |
