## 核心摘要

OpenAI 文章《Testing Agent Skills Systematically with Evals》提出系统评估 Skill 的方法：先定义成功的四类目标（结果/过程/风格/效率），识别 Skill 的两个维护面（执行体和触发边界），找出三类隐藏假设失控点（触发假设/环境假设/执行假设），构建回归样本集（10~20条 prompt 覆盖显式/隐式/带上下文/负控样本）。核心转变：从靠体感维护换成靠最小证据闭环维护。从"会写 Skill"走向"会维护 Skill"。

## 关键洞察

### 核心问题

**每次改完都不知道这是优化还是回归**：
- 改了一版 Skill，跑了一下，看起来没出事，就继续改
- 靠的是体感，不是证据
- Skill 一旦接入真实工作流，会触发工具、生成文件、留下副作用
- 改任何一个地方，整个行为边界都可能一起飘

### 一、定义 Success 的四类目标

| 目标类型 | 问题 | 为什么关键 |
|---------|------|-----------|
| **结果目标（Outcome）** | 任务有没有完成？应用能不能跑？ | 最直觉的一层，容易被单独当成全部 |
| **过程目标（Process）** | Codex 有没调用 Skill？是否按预想步骤执行？ | 最终结果对了不代表路径对了，可能靠绕路完成，下次换输入可能绕不过去 |
| **风格目标（Style）** | 输出是否符合规范？文件结构/组件组织/代码风格？ | 不影响能不能跑，但在工作流里风格不一致让下游无法接受 |
| **效率目标（Efficiency）** | 有无 thrashing/乱跑命令/烧 token？ | 单次测试感觉不到，接入生产大量触发后浪费被放大 |

### 二、Skill 的两个独立维护面

Skill 有两个独立的维护面，可以独立失控：

| 维护面 | 内容 | 失控表现 |
|-------|------|---------|
| **执行体** | instructions、工具链、步骤 | 改了 instructions，触发边界没变，但行为变了 |
| **触发边界** | name 和 description | 改了 description，执行体没变，但触发范围飘了 |

**问题**：如果只盯着执行体，触发边界的腐烂根本看不见。

### 三、三类隐藏假设（失控点）

#### 1. 触发假设（Trigger Assumptions）

**关于触发边界的盲区**：
- 哪些 prompt 明明应该触发这个 Skill，却没触发？
- 哪些 prompt 只是相邻需求，却误触发出来？

**例子**：
- "set up a quick react demo" → 应该触发 setup-demo-app
- "add tailwind styling" → 只给现有项目加样式，如果也触发 setup-demo-app，说明触发边界已糊

#### 2. 环境假设（Environment Assumptions）

**关于运行前提的盲区**：
- Skill 会不会默认永远在空目录运行？
- 会不会默认系统里有 npm？
- 会不会默认用户偏好的包管理器和想的一样？

**特点**：
- 本地测试几乎不暴露（本地环境刚好满足前提）
- 换机器、换用户，前提可能全部失效
- **环境假设最隐蔽**：不在 Skill 代码里，在 Skill 运行上下文里

#### 3. 执行假设（Execution Assumptions）

**关于执行顺序的盲区**：
- Agent 会不会跳过 `npm install`（误以为依赖已装好）？
- 会不会在项目未建前就配 Tailwind？

**特点**：每步单独看是对的，但顺序错了，整个执行会在中间状态卡死或产生垃圾输出。

### 四、构建回归样本集

找到失控点后，固定成回归样本集。对单个 Skill，**10~20 条 prompt 已足够高效**——重点不在数量，在这些样本能否卡住真实的失败模式。

#### 四类样本

| 类型 | 说明 | 作用 |
|------|------|------|
| **显式调用** | prompt 直接点名 Skill | 保证 name/description/instructions 改动后，显式调用不会坏 |
| **隐式调用** | 用户不提 Skill 名，只描述场景 | 测试 name 和 description 是否足够强，让 Codex 自己选中 |
| **带上下文调用** | 任务中混入真实业务上下文 | 测试 Skill 在噪声环境下能否被正确识别和执行 |
| **负控样本** | 不该触发这个 Skill 的请求 | 测试 Skill 该不该触发——最危险的失控，验证它知道何时不该做 |

**Negative Control**：给 Skill 划边界——不只验证它能做什么，还要验证它知道什么时候不该做。

### 五、核心转变

**完成的不是完整评估系统，而是更基础的事**：把靠体感维护换成了靠最小证据闭环维护。

这个转变比任何工具细节都重要。Skill 一旦接入生产，不再只是 prompt 技巧，开始变成要被持续验证的工作流资产。

**维护这种资产至少先回答三件事**：
1. **你想让它做成什么？**（四类目标）
2. **它会在哪些地方失控？**（三类假设）
3. **你准备用哪些样本持续盯住这些失控点？**（四类样本）

做到这里，才算真正从"会写 Skill"开始走向**"会维护 Skill"**。
