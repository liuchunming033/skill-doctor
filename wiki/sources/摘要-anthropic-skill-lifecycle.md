## 核心摘要

Anthropic 重写 Skill 生命周期：写 Skill 变成反复测试、度量、比较、修改的闭环，评估不再是最后验收而变成持续提升前提。Skill 分两类：Capability Uplift（补能力）和 Encoded Preference（固化偏好）。生命周期七步流程：决定做什么→写草稿→创建测试提示词→看结果→展示结果→根据反馈改Skill→扩大测试集。三个评测场景：基准测试（证明有用）、回归测试（保证没破坏旧能力）、版本对比（决定下一轮改哪）。核心转变：从"我相信"变成"我可以证明"。成熟的 Skill 是可验证、可比较、可复盘、可淘汰、可迭代的工作流资产。

## 关键洞察

### 核心问题

**凭感觉改 Skill 的空洞**：
- 你怎么知道它真的变好了？
- 你怎么知道它没有只在刚才那个例子里变好？
- 你怎么知道它没有悄悄把另一个场景搞坏？
- 如果下一个模型版本发布了，模型自己能完成这件事了，你怎么判断这个 Skill 还该不该存在？

### 核心转变

**Anthropic 重写 Skill 生命周期**：写 Skill 不再只是写说明文档，变成反复测试、度量、比较、修改的闭环。评估不再是最后验收，变成 Skill 能持续提升的前提。

### Skill 的两大分类

| 类型 | 定义 | 价值前提 | Eval 要回答 |
|------|------|---------|-----------|
| **Capability Uplift Skills** | 补能力：教 Claude 做原本不稳定、不会做或容易做错的事 | 模型现在还不能可靠地自己完成 | 这个 Skill 现在还带不带来真实增益？模型升级后裸模型能否做得同样好？ |
| **Encoded Preference Skills** | 固化偏好：固化你的偏好、流程、判断标准 | 它有没有忠实执行你的工作流 | 它还在按你的方式工作吗？ |

**共同点**：没有 Eval，你都只能猜。

### Skill 生命周期七步流程

#### 第一步：决定 Skill 要做什么

决定后面所有评估的方向。如果连这个 Skill 要解决什么任务都说不清，后面就无法判断它有没有成功。

#### 第二步：写一个草稿

SKILL.md 还没有到成品阶段，更像候选方案。先把流程、约束、需要 Claude 注意的细节写出来。

#### 第三步：创建少量测试提示词，马上跑起来

**和普通写 prompt 习惯分开**：普通做法是写完凭感觉试一下，Anthropic 做法是**草稿一出现就立刻让它面对测试用例**。

#### 第四步：看结果

不止看通过还是失败，还要看：
- 输出哪里对、哪里偏、哪里不稳定
- 这个测试用例本身写得好不好——有些时候错误不在 Skill，是你根本没有把什么叫好结果写清楚

#### 第五步：展示结果

工具里有 Eval Viewer，作用不是炫技，而是**把输出指标、失败原因放到人面前**。因为 Skill 质量不能只交给自动评分人，还要看它是否符合真实工作流。

#### 第六步：根据反馈改 Skill

进入迭代——不再停留于"感觉这句话可以写得更顺"，而是追问：
- 哪个测试失败了？
- 失败原因是什么？
- 是流程漏了一步，还是约束写得太宽，还是示例不够？

改完后再跑同一批测试，确认旧问题被修掉。

#### 第七步：扩大测试集

少量测试只能证明你没有在几个例子上失败，不能证明这个 Skill 稳定。后面要把测试集扩大。

**Anthropic 案例**：
- PDF Skill 在处理非填充式表单时会出问题
- 如果没有 Eval，大概率只能靠偶然发现
- 有了 Eval，失败模式就被隔离出来——测试用例告诉你在这个场景里失败了
- 然后修复定位逻辑，修完后再跑 Eval，确认这个失败模式被解决

**Eval 的第一层价值**：目标并非给 Skill 打漂亮分数，而是**把失败变成可复现的对象**。只要失败可以复现，就可以定位；只要可以定位，就可以修复；只要修复后还能再测，Skill 才进入迭代闭环。

### 三个评测场景

| 场景 | 回答的问题 | 作用 |
|------|-----------|------|
| **基准测试（Benchmark）** | 这个 Skill 有没必要存在？能不能完成预期任务？相比不用 Skill 有无增量？ | 比较 with skill 和 without skill，有 Skill 明显更稳更快更省 token 就证明增量。Capability Uplift Skill 会遇到模型进步问题 |
| **回归测试（Regression Test）** | 我改了 Skill 以后，有没有把原来能做的事弄坏？ | 守住旧能力。每次改 SKILL.md 都改变 Claude 注意力分配，可能修好一个场景但让另一个变差 |
| **版本对比（Version Comparison）** | 这一次改动到底有没有把 Skill 改好？ | A/B Testing 把两个版本并排放在同一组任务上比较，逼你回答哪个版本赢在哪里输在哪里 |

### 三类场景顺序

**先证明 Skill 有用 → 再保证修改没有破坏旧能力 → 最后比较不同版本决定下一轮往哪里改。**

### 核心转变

**从"我相信"变成"我可以证明"**：

| 没有评估时 | 有了评估以后 |
|-----------|------------|
| 写 Skill 像写信条——你相信这段流程会让 Agent 更好，你相信这些规则会被遵守，但这些都没有被验证 | 我可以证明它在这些场景下有效 |
|  | 我可以证明它比不用 Skill 更好 |
|  | 我可以证明修改没有破坏旧能力 |

**成熟的 Skill 不能只靠表面上的专业感，而应该是可以被验证、被比较、被复盘、被淘汰，也可以被持续迭代的工作流资产。**

没有 Eval 的 Skill，只是一份还没被检验过的假设——它可能有用，也可能只是看起来有用。只有当你能测试它、衡量它、比较它、改进它，它才进入工程世界。
