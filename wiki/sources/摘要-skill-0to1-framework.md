## 核心摘要

从 0 到 1 搭建 Skill 系统的四阶段框架：第一阶段认知与定位（团队能力盘点、识别 Skill 化优先级、确定 Skill 类型），第二阶段最小可行 Skill（选最简场景、写草稿、建最小测试集、跑起来、迭代），第三阶段扩展与结构化（建立目录结构、命名规范、配置管理、依赖管理），第四阶段运营与持续优化（使用度量、反馈回路、维护节奏）。核心原则：先搭反馈回路再迭代，先进入真实使用再在反馈中长出来，先证明有用再证明可维护，先跑起来再优化。

## 关键洞察

### 四阶段框架总览

```
第一阶段：认知与定位
  ↓ 搞清楚要解决什么问题
第二阶段：最小可行 Skill
  ↓ 用最低成本跑通第一个 Skill
第三阶段：扩展与结构化
  ↓ 从单个 Skill 扩展到 Skill 系统
第四阶段：运营与持续优化
  ↓ 让 Skill 系统持续运转
```

### 第一阶段：认知与定位

**目标**：搞清楚 Skill 系统要解决什么问题，以及现在团队处于什么阶段。

#### 第一步：做团队能力盘点

把团队现有的工作流画出来，标记出：

| 类型 | 说明 | 示例 |
|------|------|------|
| **高频重复** | 每天/每周都在做的事 | 站会汇报、周报、代码审查 |
| **认知负担高** | 需要资深经验才能做的事 | 故障排查、架构评审 |
| **交接成本高** | 换人做容易出问题的事 | 部署流程、权限管理 |

#### 第二步：识别 Skill 化的优先级

**不是所有工作都适合 Skill 化**。优先选：

1. **规则明确**：输入输出清楚，判断标准客观
2. **频率高**：值得投入自动化
3. **影响大**：做对了能显著提升效率

#### 第三步：确定 Skill 类型

根据识别出来的场景，确定要写什么类型的 Skill：
- 触发式
- 工具式
- 流程式
- 偏好式

### 第二阶段：最小可行 Skill

**目标**：用最低成本跑通第一个 Skill，验证流程。

#### 第一步：选一个最简单的场景

**不要选最复杂的**。选一个频率高、规则明确、影响可控的场景作为第一个 Skill。

#### 第二步：写草稿

- 确定 name 和 description（触发条件）
- 写核心流程（约束目标，不是约束路径）
- 加 gotchas（踩过的坑）
- 写清楚输入输出

#### 第三步：建最小测试集

**3~5 条 prompt**，覆盖：

| 样本类型 | 数量 | 作用 |
|---------|------|------|
| 显式调用 | 1条 | 直接点名 Skill |
| 隐式调用 | 1条 | 只描述场景 |
| 带上下文调用 | 1条 | 混入业务上下文 |
| 负控样本 | 1条 | 不该触发的请求 |

#### 第四步：跑起来

让 Agent 用这个 Skill 完成真实任务。观察：
- 触发是否正确？
- 执行是否稳定？
- 输出是否符合预期？

#### 第五步：迭代

根据真实反馈改 Skill。改完再跑测试集，确认旧问题被修掉。

### 第三阶段：扩展与结构化

**目标**：从单个 Skill 扩展到 Skill 系统，建立结构。

#### 第一步：建立 Skill 目录结构

```
skills/
├── README.md          # 总览
├── code-review/       # 代码审查 Skill
│   ├── SKILL.md       # 主文件
│   ├── references.md  # 参考文档
│   └── templates/     # 模板
├── deploy/            # 部署 Skill
│   ├── SKILL.md
│   ├── config.json    # 配置
│   └── scripts/       # 脚本
└── ...
```

#### 第二步：建立命名规范

- Skill 名称要语义清晰
- description 要写触发条件
- 避免名称冲突

#### 第三步：建立配置管理

- 用 config.json 管理团队特定的配置
- 首次运行时询问，后续直接使用

#### 第四步：建立依赖管理

- Skill 之间可以互相调用
- 在 SKILL.md 里声明依赖
- 保持轻量，不要过度工程化

### 第四阶段：运营与持续优化

**目标**：让 Skill 系统持续运转，而不是建完就荒废。

#### 第一步：建立使用度量

- 记录 Skill 的调用次数、成功率、失败原因
- 定期 Review 使用数据

#### 第二步：建立反馈回路

- 用户反馈渠道（好用的、不好用的、缺什么的）
- 定期收集和整理

#### 第三步：建立维护节奏

| Skill 类型 | 更新频率 |
|-----------|---------|
| **偏好式 Skill** | 每季度更新一次 |
| **工具式 Skill** | 接口变化时更新 |
| **触发式 Skill** | 新事件类型时更新 |
| **流程式 Skill** | 流程变更时更新 |

#### 第四步：建立淘汰机制

- 定期 Review：哪些 Skill 基本没人用？
- 评估：是 Skill 本身没用，还是 description 写得不好？
- 决策：继续优化、重构还是删除？

### 核心原则

| 原则 | 说明 |
|------|------|
| **先搭反馈回路再迭代** | 重点不是先规划出完美版本，而是先让它进入真实使用 |
| **先进入真实使用再成长** | 在反馈里慢慢长出来 |
| **先证明有用再证明可维护** | 先证明 Skill 有增量，再保证修改不破坏旧能力 |
| **先跑起来再优化** | 用最低成本跑通第一个 Skill，验证流程 |
