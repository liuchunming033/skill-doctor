## 核心摘要

本文精读 Anthropic 官方文档第三部分，系统阐述 Skill 的开发、评估与安全使用方法。核心思想：用问题驱动建设（失败样本反推缺陷）、分层设计文件结构、从 Claude 角度设计（name/description 是决策信号）、人机共创迭代（让 Claude 反思沉淀 SOP）、安全优先（只安装可信来源）。展望未来：Skill 生态共享、与 MCP 互补、智能体自主创建技能。

## 关键洞察

### 一、评估驱动建设：问题驱动

**核心理念**：用失败样本反推缺陷，不要空想功能。

- **暴露问题**：让智能体执行真实任务，观察出错、卡顿、需要额外提示的地方
- **失败样本**：这些错误和瓶颈就是 Skill 设计的起点
- **小步迭代**：先写简单版本验证可行性，通过多轮使用逐步扩展
- **持续验证**：Skill 的成长来自持续验证，而不是一次写完

### 二、分层文件结构：为规模设计

当 Skill 变复杂时，一个文件不够用，需拆分为三层：

| 层级 | 内容 | Token 消耗 | 加载方式 |
|------|------|-----------|---------|
| 最上层（元数据） | SKILL.md 的 name+description | ~100 tokens | 始终加载 |
| 中间层（正文） | 主要步骤和说明 | < 5000 tokens | 触发时加载 |
| 底层（附属文件） | 脚本、参考文档 | 理论无限 | 按需读取 |

**关键设计**：代码既可被执行也可被阅读，需明确区分哪些脚本要运行、哪些是参考说明。

### 三、从 Claude 角度设计

**核心原则**：Skill 的作者是人类，但使用者是 Claude。

- **决策依据**：Claude 主要依赖 name 和 description 判断是否调用技能
- **命名重要性**：Skill 的命名和简介不是装饰，而是 Claude 决策的信号
- **准确描述**：一句准确的描述决定技能能否被正确使用
- **观察行为**：观察 Claude 是否误触发、过度依赖特定上下文，用作改进依据

### 四、人机共创迭代

**核心理念**：Skill 的成长是人机共同完成的过程。

- **成功时**：让 Claude 总结哪些做法有效、哪些提示最关键，写回 Skill 成为标准步骤
- **失败时**：让 Claude 反思哪里出错、为什么误解、缺少什么信息
- **沉淀 SOP**：通过反思式迭代，经验真正沉淀为 SOP

### 五、安全使用

Skill 的强大也意味着风险（运行脚本、访问文件系统、连接网络）。三条安全建议：

1. **只安装可信来源的技能**
2. **使用前仔细阅读所有文件**：特别是脚本依赖和外部资源链接
3. **理解执行逻辑**：明确哪些代码需要运行，哪些只是文档说明

**安全哲学**：安全不是限制能力，而是保护信任。

### 六、未来展望

#### Skill 共享与发现生态

- **完整生命周期**：创建、编辑、发现、共享、使用
- **社会技能库**：Claude 访问全社会技能库，成为互联知识网络
- **知识互联**：个人上传专业技能模块，企业共享内部 SOP/API/工作流

#### Skill 与 MCP 的关系

互补而非重复：

| 维度 | Skill | MCP |
|------|-------|-----|
| 提供内容 | 任务知识（How） | 外部连接（What） |
| 功能定位 | 思维的框架 | 行动的手臂 |
| 示例 | 教 Claude 如何审查合同 | 连接企业数据库和 API |
| 组合效果 | 混合式智能：既懂方法又能执行 |

**愿景**：合同审查 Skill 通过 MCP 连接数据库 → 提取合同 → 比对风险字段 → 输出审查报告。

#### 更远愿景

Agent 自主创建、编辑、评估自己的技能，把验证过的思维与操作模式转化为可复用能力模块。

**终极形态**：Skill 不仅人类写给智能体的说明书，更是智能体写给自己的成长记录。

## 总结

Anthropic 用这套体系为未来智能体世界建立了**第一个可编程的行为层标准**。
